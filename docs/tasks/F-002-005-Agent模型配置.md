## 0. 快速理解

```txt
一句话：普通 Agent 继承模型服务里的默认文本模型；高级用户才给某个 Agent 单独覆盖模型和参数。
为什么改：模型服务已经有默认文本模型，不能再让普通用户重复给每个 Agent 选模型。
做完后有什么用：普通用户不用理解 Agent 模型分配；高级用户仍能精细调剧本、生产、分镜等 Agent。
这一步不碰什么：不做 Agent 对话页面，不做 API Key 配置，不做供应商管理，不做图片/视频/语音模型真实调用。
```

## 1. 废弃旧方案

旧文档里的“Agent 模型配置”方案不再作为执行依据。

废弃原因：

```txt
1. 它要求普通用户在 Agent 页面再次选择模型。
2. 它和 F-002-017 的“默认文本模型”重复。
3. 它会制造 API配置、默认模型、Agent模型配置三处跳转。
4. 用户会不知道到底哪个模型真正生效。
```

新结论：

```txt
普通模式不配置 Agent 模型。
所有 Agent 默认继承“模型服务 -> 默认文本模型”。
只有高级模式允许覆盖模型。
```

## 4. 数据规则

### 1. 数据表复用

```txt
复用现有 agent_model_configs：
  key：Agent 唯一键。
  model_id：高级覆盖模型，格式 connectionId:modelName。
  vendor_id：兼容旧字段，建议和 model_id 的 connectionId 保持一致。
  model_label：展示用模型名称。
  temperature：Agent 参数，可为 0。
  max_output_tokens：Agent 参数，0 表示自动。
  disabled：是否禁用。

不新增 AgentOverride 表。
不把覆盖数据另存一套 JSON。
不把默认文本模型复制到每个 Agent 行。
```

### 2. 默认继承

```txt
如果 agent_model_configs.model_id 为空：
  使用模型服务里的 DefaultUsageBinding.text。

如果 DefaultUsageBinding.text 为空：
  Agent 配置页显示“默认文本模型未配置”。
  Agent 执行时返回 MODEL_NOT_CONFIGURED。

如果 DefaultUsageBinding.text 指向的连接或模型失效：
  Agent 配置页显示“默认文本模型失效”。
  Agent 执行时返回 MODEL_NOT_CONFIGURED。
```

### 3. 高级覆盖

```txt
如果 agent_model_configs.model_id 有值：
  视为该 Agent 开启高级覆盖。
  model_id 必须是已启用 text 模型。
  解析顺序：覆盖模型 -> 默认文本模型 -> 未配置错误。

覆盖模型失效：
  只显示该 Agent 覆盖失效。
  不静默换成默认模型，避免用户不知道实际用哪个模型。

用户关闭覆盖：
  清空 model_id、vendor_id、model_label。
  该 Agent 恢复继承默认文本模型。
```

### 4. 参数继承

```txt
普通模式参数作为全局 Agent 参数。
高级模式可覆盖单个 Agent 参数。
单个 Agent 未配置参数时，继承全局 Agent 参数。

temperature：
  允许 0-2。
  0 是合法值，不能用 truthy 判断，否则会被忽略。

maxOutputTokens：
  0 表示自动，不传给模型。
  大于 0 时才传给模型。
```

### 5. 失效处理

```txt
默认文本模型失效：
  所有继承默认的 Agent 显示默认模型失效。

覆盖模型失效：
  只显示该 Agent 覆盖失效。
  不静默换成默认模型，避免用户不知道实际用哪个模型。

用户可手动关闭覆盖，恢复继承默认模型。
```

## 5. 与模型服务的关系

```txt
模型服务：
  管 API、Key、URL、启用模型、默认文本/图片/视频/语音模型。

Agent 参数与高级覆盖：
  默认读取模型服务里的默认文本模型。
  只在高级模式覆盖某个 Agent 的文本模型。
```

关系图：

```txt
模型服务.defaultTextModel
  -> 剧本 Agent
  -> 生产 Agent
  -> 通用 AI
  -> 子 Agent

agent_model_configs.model_id
  -> 有值：只覆盖指定 Agent
  -> 空值：继承默认文本模型
```

## 6. 参考项目怎么处理

Toonflow 参考事实：

```txt
1. 有普通/高级模式。
2. 普通模式配置主 Agent。
3. 高级模式配置子 Agent。
4. 每个 Agent 都能保存模型、temperature、maxOutputTokens。
```

VT Studio 不照搬点：

```txt
1. 不让普通用户逐个配置 Agent 模型。
2. 不做 Toonflow 一键填 Key。
3. 不把 TTS 混进当前任务。
4. 不按供应商分组显示模型，统一从模型服务的已启用文本模型中选择。
```

原因：

```txt
VT Studio 已经有模型服务默认文本模型。
重复配置会降低可用性。
高级覆盖保留了专业能力，不牺牲扩展性。
```

## 0. 快速理解

```txt
一句话：把当前“模型供应商配置”重构成“模型与能力 + 连接与接入 + 开发者模式”。
为什么现在做：现有供应商列表页更像底座管理台，不适合普通用户快速接入模型服务。
做完后有什么用：普通用户先建连接，再给文本/图片/视频/语音选择当前模型；协议和 adapter 下沉到系统内部。
这一步不碰什么：不推翻 CORE-007 模型适配层，不删除现有供应商底座，不直接开始业务页面大面积改造。
```

## 1. 本次要解决的问题

```txt
现状问题：
  1. 普通用户先看到供应商列表，不知道先配什么。
  2. API、供应商、协议、模型几个概念混在一起，心智负担太重。
  3. OpenAI 官方、中转、兼容协议这些底层差异暴露到了主入口。
  4. 当前页面适合维护底座，不适合快速完成“现在我要用哪个模型”。

重构目标：
  1. 默认入口不再是供应商管理。
  2. 普通用户只处理“当前能力用哪个模型”和“怎么接一个新服务”。
  3. 协议、adapter、调试详情下沉到开发者模式。
  4. 底层仍然保留模型适配层，不做简单粗暴的删底座。
```

## 2. 目标结构

### 1. 模型与能力

```txt
这是默认主入口。
面向普通用户。
只关心四类能力当前用什么：
  文本生成
  图片生成
  视频生成
  语音生成
```

每个能力卡片只显示：

```txt
当前连接
当前模型
可用状态
测试按钮
更换按钮
```

### 2. 连接与接入

```txt
这是第二层入口。
面向“我要接一个新服务”。
这里管理连接，不管理供应商。
```

一条连接至少包含：

```txt
name
serviceType
protocolType
baseUrl
apiKey
capabilities
status
```

普通用户看到的是：

```txt
OpenAI 官方
OpenAI 中转
Claude
DeepSeek
Gemini
本地工作流
其他高级接入
```

系统内部再把它们映射到：

```txt
openai-official
openai-compatible
anthropic
deepseek
gemini
workflow
media-async
```

### 3. 开发者模式

```txt
不作为普通入口。
只放底层和高风险能力：
  协议
  adapter
  特殊 headers
  特殊 body 参数
  调试日志
  请求响应详情
  异步轮询配置
```

## 3. 核心数据关系

```txt
Connection
  一条可复用的模型服务连接

ModelRegistration
  某条连接下有哪些模型、它们属于什么能力类型

CapabilityBinding
  当前文本/图片/视频/语音分别绑定到哪个连接和哪个模型
```

关系：

```txt
模型与能力 -> 读 CapabilityBinding
CapabilityBinding -> 指向 Connection + ModelRegistration
Connection -> 决定 URL/Key/协议
ModelRegistration -> 决定模型名和能力类型
协议适配层 -> 决定系统怎么调用
```

## 4. 为什么不直接删掉底座

```txt
1. 同一个模型名不能唯一决定协议。
   例如 gpt-5.5 可能走官方 OpenAI，也可能走中转。

2. 图片、视频、语音供应商差异明显大于文本。
   有些是同步接口，有些要任务提交、轮询、下载结果。

3. 特殊厂商和本地工作流仍然需要 adapter。

4. 用户不看协议，不代表系统没有协议。
```

结论：

```txt
前台要简单化。
底层不能删协议适配层。
普通模式隐藏底层概念。
开发者模式保留底层能力。
```

## 14. 模型服务单入口收口方案

### 14.1 为什么要再收口

```txt
当前已完成的 API配置 方案虽然能用，但普通用户仍然会看到：
  1. API配置
  2. 默认模型
  3. Agent 模型配置
  4. 开发者模式

问题：
  用户会感觉“一下这里一下那里”，不知道真正该配哪里。
  默认模型和 Agent 模型配置容易重复。
  供应商、连接、模型、用途、Agent 覆盖几个概念仍然会泄露给普通用户。
```

专业结论：

```txt
普通用户只保留一个模型相关入口：模型服务。
默认模型、服务接入、模型启用、模型测试都合并到模型服务里。
Agent 默认继承文本默认模型，不再作为普通用户的模型选择入口。
开发者能力保留，但默认折叠到模型服务的高级区域或开发者模式。
```

```txt

  模型服务
```

### 14.4 数据关系

```txt
ModelService
  一条服务连接，保存服务类型、名称、Base URL、API Key、状态。

EnabledModel
  某条服务下启用的模型，模型自己声明能力：text/image/video/tts。

DefaultUsageBinding
  文本/图片/视频/语音当前默认用哪个 service + model。

AgentOverride
  高级配置。默认不存在；存在时表示某个 Agent 不继承默认文本模型。
```

规则：

```txt
1. 能力属于模型，不属于 API Key。
2. 默认用途绑定只从已启用模型里选择。
3. 同名模型显示为“服务名称 / 模型名”。
4. 内部唯一标识仍然是 serviceId + modelName。
5. Agent 默认读取 DefaultUsageBinding.text。
6. AgentOverride 只在高级模式显示和编辑。
```

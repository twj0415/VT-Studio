状态：等待用户确认

## 0. 快速理解

```txt
一句话：统一规定后续业务怎么写任务记录，不能每个页面各写一套状态。
为什么现在做：任务中心如果只会查，但业务不统一写，后面列表会缺数据。
做完后有什么用：图片、视频、Agent、导出都能用同一套任务记录展示进度和失败原因。
这一步不碰什么：不实现某个具体业务生成，不做 UI 重试，不做供应商调用细节。
```

## 1. 本次做什么

```txt
目标：
  对齐 Toonflow `taskRecord.ts` 和 `utils/ai.ts` 的任务写入语义，落成 VT Studio 统一任务写入规则。

只做：
  1. 任务创建规则
  2. 成功/失败/取消状态写入规则
  3. 关联对象 JSON 规则
  4. 模型调用和任务记录的边界
  5. 后续业务模块必须复用的 task service 能力

不做：
  1. 具体图片生成
  2. 具体视频生成
  3. 具体 Agent 对话
  4. 任务重试 UI
```

## 2. 参考项目怎么做

| 参考文件 | 关键逻辑 |
|---|---|
| `Toonflow-app/src/utils/taskRecord.ts` | `u.task(projectId, taskClass, modelName, { describe, content })` 插入任务，返回 done 函数 |
| `Toonflow-app/src/utils/ai.ts` | `u.Ai.Image/Video.run(taskRecord)` 内部创建任务，成功/失败自动更新 |
| `Toonflow-app/src/routes/assetsGenerate/generateAssets.ts` | 图片生成把 taskMeta 传给模型 run |
| `Toonflow-app/src/routes/production/storyboard/batchGenerateImage.ts` | 分镜图生成传入任务相关元信息 |

参考项目关键事实：

```txt
1. 创建任务时 state=进行中，startTime=Date.now()。
2. content 是 string 时直接写 relatedObjects。
3. content 是 object 时 JSON.stringify 后写 relatedObjects。
4. content 是 function 时抛不支持。
5. done(1) 更新已完成，reason=null。
6. done(-1, reason) 更新生成失败并写 reason。
7. 模型 run 成功失败会通过 u.Ai 内部任务记录处理。
```

## 4. 要做什么功能

### 1. 创建任务记录

怎么做：
- 输入：projectId、category、modelName、description、relatedObject。
- 输出：taskId 和任务记录。
- 写什么数据：写 tasks。
- 状态怎么变：pending 或 running，具体看是否立即执行。
- 异常怎么处理：projectId 不存在、category 为空、relatedObject 不可序列化时失败。
- 限制：relatedObject 必须是 string 或可 JSON 序列化对象。

### 2. 更新任务成功

怎么做：
- 输入：taskId、可选结果摘要。
- 输出：任务状态 succeeded。
- 写什么数据：更新 tasks.status、finished_at、fail_reason=null。
- 状态怎么变：running -> succeeded。
- 异常怎么处理：任务不存在返回错误；已 cancelled 的任务不能被覆盖为成功。
- 限制：不能把成功结果写进 failReason。

### 3. 更新任务失败

怎么做：
- 输入：taskId、failReason。
- 输出：任务状态 failed。
- 写什么数据：更新 tasks.status、fail_reason、finished_at。
- 状态怎么变：running -> failed。
- 异常怎么处理：失败原因为空时写兜底原因。
- 限制：失败原因必须脱敏。

### 4. 取消任务记录

怎么做：
- 输入：taskId。
- 输出：任务状态 cancelled。
- 写什么数据：更新 tasks.status、finished_at。
- 状态怎么变：pending/running -> cancelled。
- 异常怎么处理：任务不存在或任务已结束时返回明确错误。
- 限制：取消不能只改状态，必须有业务 cancel handler；没有 handler 的任务不能伪取消。

### 5. 模型适配层写任务

怎么做：
- 输入：模型调用参数和 taskMeta。
- 输出：任务记录和模型结果。
- 写什么数据：模型开始前创建任务；模型完成后更新任务。
- 状态怎么变：pending/running -> succeeded/failed/cancelled。
- 异常怎么处理：供应商失败写 failReason；SDK 原始 detail 只进日志。
- 限制：页面不能自己假造任务成功。

## 5. 数据和状态

```txt
字段：
  tasks.id
  tasks.project_id
  tasks.category
  tasks.related_object
  tasks.model_name
  tasks.description
  tasks.status
  tasks.fail_reason
  tasks.started_at
  tasks.finished_at

接口/能力：
  task.create
  task.markRunning
  task.markSucceeded
  task.markFailed
  task.cancel
  task.list

数据读写：
  写 tasks
  更新 tasks

任务状态：
  pending -> running -> succeeded
  pending -> running -> failed
  pending/running -> cancelled

轮询/Socket：
  本任务不做 UI 轮询，只提供状态可查

模型调用：
  模型适配层使用 task service 写记录

删除影响：
  删除项目或业务对象时要考虑是否保留历史任务，后续对应模块决定
```

## 7. 偏差

```txt
和 Toonflow 不同的地方：
  1. 内部状态增加 pending/cancelled。
  2. 取消必须有 handler，不能只改状态。
  3. relatedObject 用结构化 JSON，不只是一段字符串。

原因：
  VT Studio 要支持本地任务队列和后续取消能力，任务记录必须可定位来源对象。

是否写入 04：
  写入，因为这是对 Toonflow 任务记录的规范化增强。
```

## 8. 验收

```txt
1. task service 能创建任务。
2. 任务成功能写 succeeded。
3. 任务失败能写 failed 和 failReason。
4. 任务取消不会伪取消没有 handler 的任务。
5. relatedObject 能保存对象并在任务中心摘要展示。
6. 失败原因会脱敏。
7. 后续业务不能绕过 task service 自己写状态。
```

# F-002-015 关于与版本信息

状态：等待用户确认  
所属菜单：M-002 设置  
对应功能文档：`docs/features/M-002-设置.md`  
原则：先确认本文档，再改代码

## 0. 快速理解

```txt
一句话：设置里显示应用版本、项目信息、外链，并提供检查更新和下载安装入口。
为什么现在做：用户需要知道当前版本，也需要后续定位问题和升级。
做完后有什么用：能看到版本号、打开项目相关链接、检查是否有新版本。
这一步不碰什么：不做自动静默更新，不做强制联网启动，不做复杂安装器。
```

## 1. 本次做什么

```txt
目标：
  对齐 Toonflow about 页的版本信息和更新入口，落成 VT Studio 本地桌面版可控更新能力。

只做：
  1. 显示应用名称、Logo、当前版本
  2. 打开 GitHub/Gitee/License 等外链
  3. 检查更新
  4. 选择更新源
  5. 自定义更新地址
  6. 下载/重新安装入口

不做：
  1. 启动时强制检查更新
  2. 静默安装
  3. 自动替换正在运行的程序
  4. 用户无确认自动下载
```

## 3. 用户操作

```txt
入口：
  设置弹窗 -> 关于与版本信息。

按钮/操作：
  1. 点击检查更新。
  2. 选择 GitHub/Gitee/自定义 URL。
  3. 输入自定义更新地址。
  4. 点击下载安装。
  5. 点击打开 GitHub/Gitee/License 外链。

弹窗/表单：
  自定义更新地址输入。

成功反馈：
  当前已是最新版；发现新版本；下载开始；下载完成显示安装路径。

失败反馈：
  网络失败、更新地址无效、版本格式不合法、下载失败。
```

## 4. 要做什么功能

### 1. 读取当前版本

怎么做：
- 输入：无。
- 输出：应用名、版本号、构建信息、运行环境。
- 写什么数据：不写。
- 状态怎么变：页面显示当前版本。
- 异常怎么处理：读取失败显示未知版本。
- 限制：版本来源以 package/electron app 为准，不让页面硬编码。

### 2. 检查更新

怎么做：
- 输入：更新源 `github/gitee/custom` 和可选 URL。
- 输出：是否有新版本、新版本号、下载地址、更新说明。
- 写什么数据：不写。
- 状态怎么变：检查中 -> 有更新/无更新/失败。
- 异常怎么处理：网络失败、响应结构不对、版本号非法分别提示。
- 限制：用户主动点击才请求网络。

### 3. 下载更新

怎么做：
- 输入：下载 URL、版本号、是否重新安装。
- 输出：下载文件路径或下载任务状态。
- 写什么数据：下载到 `userData/downloads` 或系统下载目录。
- 状态怎么变：下载中 -> 完成/失败。
- 异常怎么处理：磁盘不足、网络中断、校验失败要明确提示。
- 限制：下载完成后让用户手动确认安装，不静默执行安装包。

### 4. 打开外链

怎么做：
- 输入：外链 key。
- 输出：系统浏览器打开。
- 写什么数据：不写。
- 状态怎么变：无。
- 异常怎么处理：打开失败提示。
- 限制：只能打开白名单 URL，不能让页面传任意链接。

## 7. 偏差

```txt
和 Toonflow 不同的地方：
  1. 不做静默安装。
  2. 外链使用白名单 key。
  3. 检查更新只在用户主动触发时联网。

原因：
  桌面应用更新属于高风险操作，必须用户确认。

是否写入 04：
  不需要，属于安全边界。
```

## 8. 验收

```txt
1. 能显示当前版本。
2. 检查更新有 loading、成功、失败状态。
3. 自定义 URL 非法时不能请求。
4. 有新版本时显示版本号和下载入口。
5. 下载失败有明确原因。
6. 外链只允许白名单。
7. 不自动安装。
8. 页面不直接访问 Node shell。
```

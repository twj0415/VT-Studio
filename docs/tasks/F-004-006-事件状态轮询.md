# F-004-006 事件状态轮询

状态：等待用户确认

## 0. 快速理解

```txt
一句话：原文列表自动轮询事件分析中的章节，把结果刷新到表格。
为什么现在做：事件分析是异步的，用户需要看到什么时候完成或失败。
做完后有什么用：页面不用手动刷新，也能看到生成中章节的最新状态。
这一步不碰什么：不重新生成事件，不做任务中心自动轮询，不做 Socket。
```

## 1. 本次做什么

```txt
目标：
  对齐 Toonflow 每 3 秒轮询 `/novel/getNovelEventState`。

只做：
  1. 找出事件生成中的章节
  2. 每 3 秒轮询状态
  3. 更新表格行事件摘要、状态、失败原因
  4. 没有生成中章节时停止轮询
  5. 离开页面停止轮询

不做：
  1. Socket 推送
  2. 任务中心自动刷新
  3. 重新生成事件
```

## 2. 参考项目怎么做

| 参考文件 | 关键逻辑 |
|---|---|
| `Toonflow-web/src/views/novel/index.vue` | 过滤 `!item.eventState` 的章节，每 3 秒轮询 |
| `Toonflow-app/src/routes/novel/getNovelEventState.ts` | 接收 ids，只返回 eventState != 0 的章节 |
| `Toonflow-app/src/lib/fixDB.ts` | 启动时把残留 eventState=0 改为失败 |

参考项目关键事实：

```txt
1. 轮询间隔是 3 秒。
2. 后端只返回不再生成中的章节。
3. 前端拿返回结果更新当前行 eventState、event、errorReason。
4. 没有生成中章节时停止轮询。
5. 软件异常退出时，fixDB 把残留生成中改失败。
```

## 4. 要做什么功能

### 1. 识别生成中章节

怎么做：
- 输入：当前表格数据。
- 输出：running 状态章节 id 列表。
- 写什么数据：不写。
- 状态怎么变：有 running 才启动轮询。
- 异常怎么处理：无 id 的行忽略。
- 限制：不能用 `!eventState` 这种含糊判断，必须用明确枚举。

### 2. 轮询事件状态

怎么做：
- 输入：running chapterIds。
- 输出：已完成或失败的章节状态。
- 写什么数据：不写，服务只查询。
- 状态怎么变：每 3 秒请求一次。
- 异常怎么处理：请求失败记录错误并保留下次轮询。
- 限制：页面卸载必须清理 timer。

### 3. 更新表格行

怎么做：
- 输入：轮询返回的章节状态。
- 输出：表格局部更新。
- 写什么数据：不写。
- 状态怎么变：running -> succeeded/failed。
- 异常怎么处理：返回不存在 id 时忽略。
- 限制：不整体丢失用户当前分页和选择状态。

### 4. 启动修复残留状态

怎么做：
- 输入：应用启动时残留 running 章节。
- 输出：失败状态和原因。
- 写什么数据：更新 event_status=failed、event_error=软件退出导致失败。
- 状态怎么变：running -> failed。
- 异常怎么处理：修复失败写日志。
- 限制：该能力属于启动修复服务，不写在页面里。

## 7. 偏差

```txt
和 Toonflow 不同的地方：
  使用明确 running 枚举，不用 `!eventState` 判断。

原因：
  避免 null、0、undefined 混淆。

是否写入 04：
  不需要，属于状态规范修正。
```

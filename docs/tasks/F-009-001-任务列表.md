状态：等待用户确认

## 0. 快速理解

```txt
一句话：做任务中心的主表格，只负责查看任务，不负责执行模型生成。
为什么现在做：后续图片、视频、Agent、导出都会写任务，必须先能统一看到状态和失败原因。
做完后有什么用：用户能一眼看到哪些任务正在跑、哪些成功、哪些失败，以及失败在哪里。
这一步不碰什么：不做重试、不做取消、不直接调用模型、不写业务生成逻辑。
```

## 1. 本次做什么

```txt
目标：
  对齐 Toonflow `/task/getTaskApi` 的任务列表能力，落到 VT Studio 的统一任务表。

只做：
  1. 任务中心页面主表格
  2. 查询任务列表
  3. 显示任务分类、关联对象、模型、描述、状态、失败原因、开始时间
  4. 空状态、加载状态、失败提示
  5. 明确项目字段别名，避免 join 后覆盖

不做：
  1. 项目筛选
  2. 分类筛选
  3. 状态筛选
  4. 任务重试
  5. 任务取消
  6. 任务详情弹窗
```

## 2. 参考项目怎么做

| 参考文件 | 关键逻辑 |
|---|---|
| `Toonflow-web/src/views/task/index.vue` | 页面进入后调用任务列表接口，表格展示任务字段，失败状态用 tooltip 展示原因 |
| `Toonflow-app/src/routes/task/getTaskApi.ts` | 从 `o_tasks` leftJoin `o_project`，按条件分页查询，按 id desc 排序 |
| `Toonflow-app/src/utils/taskRecord.ts` | 生成任务写入任务表，字段包含 taskClass、relatedObjects、model、describe、state、startTime、reason |
| `Toonflow-app/src/lib/initDB.ts` | 定义 `o_tasks` 参考字段 |

参考项目关键事实：

```txt
1. 任务中心只展示任务，不直接执行模型。
2. 表格列包括任务大类、关联对象、模型、描述、失败原因、状态、开始时间。
3. 查询按 o_tasks.id desc 排序。
4. 接口 leftJoin 项目表，但正式页面没有展示项目名列。
5. Toonflow select 里有 `o_project.*`，存在字段覆盖风险；VT Studio 必须用明确别名。
```


## 4. 要做什么功能

### 1. 打开任务中心并加载任务

怎么做：
- 输入：进入任务中心页面。
- 输出：任务列表和总数。
- 写什么数据：不写。
- 状态怎么变：loading=true -> loading=false。
- 异常怎么处理：查询失败显示错误提示，表格保留空数组。
- 限制：不默认自动轮询。

### 2. 展示任务表格

怎么做：
- 输入：任务数组。
- 输出：表格行。
- 写什么数据：不写。
- 状态怎么变：无。
- 异常怎么处理：字段为空时显示 `-` 或空态文案。
- 限制：关联对象如果是 JSON，只展示简短摘要，完整内容留给后续详情能力。

### 3. 展示状态和失败原因

怎么做：
- 输入：任务状态、失败原因。
- 输出：状态标签和失败原因列。
- 写什么数据：不写。
- 状态怎么变：无。
- 异常怎么处理：失败任务 reason 为空时显示“无失败原因”。
- 限制：不能暴露密钥、完整 stack、本地敏感路径。

### 4. 明确查询字段

怎么做：
- 输入：分页参数。
- 输出：带别名的任务数据。
- 写什么数据：不写。
- 状态怎么变：无。
- 异常怎么处理：字段冲突时按服务层 DTO 输出固定字段。
- 限制：不能照搬 `o_tasks.* + o_project.*` 的覆盖写法。

## 7. 偏差

```txt
和 Toonflow 不同的地方：
  1. 内部状态用英文枚举，UI 显示中文。
  2. 查询字段使用明确别名，不照搬 `o_project.*`。

原因：
  避免字段覆盖，避免后续状态字符串混乱。

是否写入 04：
  状态枚举偏差如 04 已有统一记录则不用重复；若新增字段名偏差，补到 04。
```

# 对齐验收与偏差记录

## 1. 职责

本文档只记录两件事：

```txt
1. VT Studio 和 Toonflow 哪里不一样
2. 每个功能最后是否对齐通过
```

不写功能过程，不写按钮清单，不写接口细节。那些内容写到 `features/` 和 `tasks/`。

## 2. 对齐原则

```txt
技术栈可以不同
代码结构可以不同
组件库可以不同
UI 可以更清晰
高风险实现可以换成更稳的实现
业务语义不能随意改
用户能做的事不能少
状态和失败原因不能少
数据关系不能凭空改
```

## 3. 必须对齐

每个功能验收时至少检查：

```txt
入口是否一致
页面区域是否完整
按钮/操作是否缺失
弹窗/表单字段是否缺失
默认值和必填规则是否一致
服务调用语义是否一致
参数和返回是否可对应
数据库读写是否可对应
本地文件读写是否可对应
任务记录是否完整
状态流转是否完整
轮询或 Socket 是否完整
模型调用参数是否完整
成功反馈是否完整
失败反馈是否完整
删除和级联影响是否清楚
```

## 4. 允许偏差

允许，但必须记录：

```txt
Electron/Vue/Tailwind/SCSS/TDesign 的实现差异
renderer -> preload -> main 的本地安全分层
不用 Toonflow 的 Express 路由形态，改为 Electron IPC 服务
运行数据不放项目目录，改放系统 userData/temp
登录弱化为本地优先
请求地址配置弱化为本地服务配置
剪映草稿导出属于 VT Studio 增强能力
命名拼写修正
更清楚的错误提示或交互布局
```

## 5. 不允许偏差

```txt
漏掉参考项目已有功能
漏按钮、漏表单字段、漏批量操作
漏失败原因、漏任务状态、漏轮询
把复杂流程做成静态占位
页面直接访问 SQLite、文件系统、模型 SDK
没看源码就按想象实现
为了省事改变业务语义
```

## 6. 已知基础偏差

| 编号 | 范围 | 偏差点 | 原因 | 结论 |
|---|---|---|---|---|
| D-BASE-001 | 技术栈 | 不复制 Toonflow 的前后端结构，VT Studio 使用 Electron main/preload/renderer | 本项目是本地桌面端，需要更清晰的本地能力边界 | 可接受 |
| D-BASE-002 | 运行目录 | Electron runtime/userData 不放项目目录 | 项目目录应保持干净，可提交内容和运行缓存分离 | 可接受 |
| D-BASE-003 | API 形态 | 不直接照搬 HTTP routes，业务语义映射到 `window.vtStudio` IPC 契约 | renderer 不能直接碰 Node，本地桌面更适合受控 IPC | 可接受 |
| D-BASE-004 | 登录 | Toonflow 登录能力已弱化为本地优先 | 用户目标是本地创作工具，账号不是主链路；已由 D-BASE-013 和 CORE-010 收口 | 可接受 |
| D-BASE-005 | 导出 | Windows 剪映专业版草稿导出是新增增强能力 | VT Studio 的核心目标之一 | 可接受 |
| D-BASE-006 | 任务队列 | Toonflow 只有 `进行中/已完成/生成失败`，VT Studio 额外增加 `cancelled` | 后续长任务、模型生成、导出需要统一取消标记 | 可接受 |
| D-BASE-007 | 任务队列 | Toonflow 任务表只有 `startTime`，VT Studio 增加 `finished_at/created_at/updated_at` | 需要完整生命周期、排序、耗时统计和排查能力 | 可接受 |
| D-BASE-008 | 任务队列 | 启动时把遗留 `running` 任务改成 `failed` | 对齐 Toonflow `fixDB.ts` 的异常退出修复思路，避免任务永久卡住 | 可接受 |
| D-BASE-009 | 模型适配层 | 动态供应商代码只允许在 main 侧 vm2 沙盒执行 | 对齐 Toonflow 动态供应商能力，同时避免 renderer 接触 SDK、密钥和本地文件 | 可接受 |
| D-BASE-010 | 模型适配层 | ComfyUI 不硬编码进核心层，统一作为供应商 adapter 扩展 | 避免核心层绑定某个协议，后续 OpenAI Compatible、ComfyUI、火山、可灵等都走同一协议 | 可接受 |
| D-BASE-011 | 模型适配层 | 依赖使用参考项目同代 `ai@6/@ai-sdk`，不使用最新 `ai@7` | 当前开发环境是 Node 20.19，`ai@7` 要求 Node >=22；需要优先保证项目可运行 | 可接受 |
| D-BASE-012 | 媒体显示 | OSS 抽象改为本地受控 HTTP 媒体服务（127.0.0.1 + token + 随应用启停 + 真缩略图） | 参考项目用本地 HTTP 服务显示图片但裸奔，缩略图被注释；VT Studio 要导剪映，视频需 range 分段加载，统一走 HTTP 服务最省事且安全可控 | 可接受，见 CORE-012、00-公共底层 8.1.1 |
| D-BASE-013 | 登录 | 登录降级为本地默认用户，完整云账号体系后置 | 本项目本地优先，账号不是主链路；收口 C-BASE-001 | 可接受 |
| D-BASE-014 | 嵌入与记忆 | 嵌入沿用参考项目本地 ONNX（all-MiniLM-L6-v2），不走供应商 embedding 接口 | 本地优先：离线、零成本、不泄露创作内容；模型层不加 embedding 类型，文本摘要复用 CORE-007 | 可接受，见 CORE-011 |
| D-BASE-015 | 日志诊断 | 新增统一日志层，终端只显示中文短状态，详细对象写 runtime logs | 参考项目没有统一日志层；VT Studio 后续模型、任务、导出日志很多，必须先收口 | 可接受，见 CORE-013 |


## 9. 结论规则

```txt
完全对齐：业务语义、流程、状态、异常、数据都一致，没有偏差
可接受偏差：有差异，已记录，不影响业务语义
不可接受偏差：漏功能、漏异常、漏状态、影响业务语义，或没有记录
未确认：源码还没看完，不能进入实现
```

只有“完全对齐”或“可接受偏差”的功能，才能在 `03-执行进度.md` 标为完成。

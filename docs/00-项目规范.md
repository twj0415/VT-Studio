# 项目规范

## 1. 项目目标

```txt
项目：VT Studio
形态：本地桌面端创作工具
参考：Toonflow 全功能
原则：技术栈可不同，业务语义必须先对齐
重点：本地优先、模型可配置、任务可追踪、最终支持 Windows 剪映专业版草稿导出
```

不能做成演示版。参考项目里已有的入口、按钮、字段、状态、异常、数据关系，分析清楚前不能删、不能改、不能脑补。

## 2. 技术栈

```txt
桌面端：Electron
构建：electron-vite / Vite
前端：Vue 3 + TypeScript
样式：Tailwind CSS + SCSS
组件库：TDesign Vue Next
状态：Pinia
路由：Vue Router
数据库：SQLite
本地能力：Electron main / preload
模型层：main 侧 Model Adapter
任务层：main 侧本地任务队列
导出：Jianying Draft Exporter
```

## 3. 推荐代码架构

```txt
src/
  main/
    app/                 # Electron 启动、窗口、运行目录
    ipc/                 # window.vtStudio 能力入口
    services/            # 业务、数据库、文件、任务、模型统一放这里，先简单
  preload/
  renderer/
    pages/
    components/
    stores/
    router/
    styles/
  shared/
    types/
    constants/
```

硬规则：

```txt
renderer 不能直接访问 SQLite
renderer 不能直接访问文件系统
renderer 不能直接调用模型 SDK
renderer 只能调用 window.vtStudio
main 侧能力先统一放 services，不提前拆太多目录
功能复杂到 services 里单文件难维护时，再按 db/files/tasks/models 拆子目录
不要一开始为了“看起来专业”搞很多空目录
不要把同一个功能拆到十几个地方
```

推荐调用链保持简单：

```txt
renderer -> window.vtStudio -> main/ipc -> main/services
```

后期如果某个 service 文件变得明显过大，再在对应任务文档里提出拆分方案，用户确认后再拆。不要预先设计一堆目录。

## 4. 运行目录规则

Electron 的运行缓存、用户数据、临时文件不能放在项目源码目录。

```txt
项目目录：只放源码、文档、配置、可复现资源
开发 userData：系统临时目录下的 VT Studio Dev
生产 userData：系统用户数据目录下的 VT Studio
项目业务文件：由用户选择或由 VT Studio 在用户数据目录下创建
临时生成文件：放系统临时目录或受控 cache 目录
```

推荐落位：

```txt
开发环境：%TEMP%\VT Studio Dev\user-data
生产环境：%LOCALAPPDATA%\VT Studio\user-data
```

禁止落位：

```txt
D:\project\vt-studio\.runtime
D:\project\vt-studio\data
D:\project\vt-studio\temp
```

例外：只有明确属于测试夹具、示例素材、可提交配置的文件，才允许放进项目目录。

## 5. 文档架构

```txt
docs/
  00-项目规范.md                 # 长期规则
  01-参考功能总表.md             # 文档入口和功能实现地图索引
  02-菜单功能清单.md             # 只列菜单和功能
  03-执行进度.md                 # 当前做到哪里
  04-对齐验收与偏差记录.md       # 只记录偏差和验收结论
  features/                      # 每个菜单怎么实现
  tasks/                         # 真正开发时，一个功能一个执行文档
```

`features/` 写完整分析，`tasks/` 写单个功能的开工方案。不要把所有细节塞进 `01`。

文档分工必须保持简单：

```txt
00：规则，不写进度
01：入口，不塞细节
02：菜单和功能清单，不写实现
03：进度，不写功能实现
04：偏差和验收，不写任务过程
features：参考项目怎么做，必须细
tasks：本次准备怎么做，一个功能一个文件
```

任务确认规则：

```txt
每个功能或基础任务开工前，必须先优化或创建对应 docs/tasks/{真实功能ID}-{功能名}.md。
task 文件名必须和 features 文件前缀对齐：
  docs/features/00-公共底层.md        -> CORE-xxx
  docs/features/M-001-全局导航与项目入口.md -> F-001-xxx
  docs/features/M-002-设置.md         -> F-002-xxx
  docs/features/M-003-项目管理.md     -> F-003-xxx
  docs/features/M-004-小说原文.md     -> F-004-xxx
  docs/features/M-005-剧本.md         -> F-005-xxx
  docs/features/M-006-资产中心.md     -> F-006-xxx
  docs/features/M-007-角景音频绑定.md -> F-007-xxx
  docs/features/M-008-生产工作台.md   -> F-008-xxx
  docs/features/M-009-任务中心.md     -> F-009-xxx
  docs/features/M-010-导出.md         -> F-010-xxx
  docs/features/M-011-剧本Agent.md    -> F-011-xxx
模板、扫描、搭框架等非功能文档用 TEMPLATE/DOC/FRAME，不占功能编号。
不允许用 013、014、015 这种流水号命名正式功能。
tasks 文档必须写清楚：快速理解、本次做什么、参考项目怎么做、准备改哪些文件、本次不做什么、验收标准、需要用户确认点、最后大白话。
tasks 文档开头必须有“快速理解”小节，用通俗语言说明：一句话、为什么现在做、做完后有什么用、这一步不碰什么。
tasks 文档必须有“要做什么功能”小节，逐项列出本任务提供哪些功能/操作，以及每个功能怎么做、输入、输出、写什么数据、状态怎么变、异常怎么处理、限制。
tasks 文档结尾必须有“最后大白话”小节，用通俗语言说明：我准备怎么做、我不会做什么、确认后才执行。
tasks 文档不能只写页面按钮，必须把“照着做就不会漏”的基础契约写进去。
能从参考源码、现有架构、已定规则推出来的内容，必须直接写进 task，不能留到实现时再猜。
参考项目确实没有、源码确实没找到、或必须做产品取舍的内容，必须明确写“源码未找到”或“需要你确认”，不能伪造成已定事实。
用户明确确认 tasks 文档后，才允许改代码。
用户未确认前，只能继续改文档，不能开始实现。
如果 task 文档或执行过程中出现“需要用户确认点”，必须先把确认点整理给用户确认；确认前不能开始或继续实现该任务。
新增确认点不能默认按 AI 判断执行，也不能用“先做再说”的方式绕过确认。
```

连续执行规则：

```txt
用户如果明确要求“按顺序一直做下去”，后续执行默认进入连续推进模式。
连续推进模式下，每完成一个 task，都必须继续做下一个，不停在“等待安排”状态。
每个下一个 task 开始前，必须先做 4 件事：
1. 对照 Toonflow 参考源码和当前 VT Studio 实现，检查有没有业务语义、交互或安全边界出入。
2. 如果有出入，先从专业性、可维护性、用户使用便利性 3 个角度给出明确优化建议。
3. 先把建议和结论写回对应 task 文档，再开始实现；不能只在聊天里说。
4. 实现完成后，必须做专项验证 + typecheck + 必要回归 + build，再更新 03。

连续推进模式不等于可以跳过确认规则。
如果下一个 task 文档里存在“需要用户确认点”，或执行中新增产品边界确认点，必须先停下来确认。
只有“无确认点”或“确认点已经被用户明确确认”的 task，才允许直接进入代码实现。
如果用户之前已经给出“按建议默认执行”“都按你说的做”这类全局授权，只能用于已列出的确认点；执行中新增确认点仍然必须重新确认。
```

`快速理解` 写法要求：

```txt
不要写官方套话。
不要堆技术名词。
不要超过 5 行。
必须让用户不用看完整文档，也能知道这一步要解决什么问题。
```

`最后大白话` 写法要求：

```txt
不要写成官方总结。
必须直接说明“我会做什么”和“我不会做什么”。
必须保留确认规则，尤其是有确认点时，未确认不能执行。
如果执行前或执行中发现新的确认点，必须先停下来问用户；用户确认前不能继续改代码。
```

`要做什么功能` 写法要求：

```txt
不能写“先看参考项目”“去分析源码”“后续确认字段”这种过程话。
需要参考源码时，必须先查完，再把查到的事实写进 task。
每个功能都必须写“怎么做”。
基础能力也要写能操作什么，例如任务队列能创建、更新、查询、取消。
业务功能要写用户操作 -> 服务调用 -> 数据读写 -> 状态变化。
```

`不能漏的收口点` 写法要求：

```txt
每个未实现 task 开工前，必须补齐这些点；能确定的直接写结论，不能确定的写清原因：
1. 依赖谁：前置任务、前置配置、前置表、前置服务。
2. 被谁依赖：后续哪些功能必须复用它，禁止绕过。
3. 服务契约：window.vtStudio 能力名、main service 方法、输入、输出、失败 msg。
4. 数据契约：表、字段、枚举、默认值、唯一约束、引用关系、删除影响。
5. 状态契约：loading/saving/running/succeeded/failed/cancelled 等状态怎么来，不能用中文状态直接查库。
6. 文件契约：文件放哪里、保存相对路径还是 URL、是否清理、是否可恢复。
7. 模型契约：用哪个模型配置、提示词从哪里取、失败是否写任务中心。
8. 任务契约：是否创建任务、任务分类、关联对象、失败原因、取消规则。
9. 异常契约：参数缺失、配置缺失、文件缺失、模型失败、数据库失败、权限失败分别怎么提示。
10. 回滚契约：写库和写文件同时发生时，失败怎么回滚或怎么标记残留。
11. 参考项目缺陷：参考项目有但不能照搬的坏逻辑，要明确写出。
12. VT Studio 差异：只要改变参考项目业务语义或新增能力，必须同步 04。
```

如果一个 task 已经写过，但复扫发现漏了上面这些点：

```txt
未实现：直接补进 task，状态仍为“等待用户确认”。
已实现：先把 task 状态改成“已完成，新增补充待补做”，补完实现和验证后再改回“已完成”。
不能只在聊天里说明，必须写进对应 task 或 features。
```

## 7. 命名和旧痕迹规则

Toonflow 的表名、字段名、接口名、拼写错误只作为“参考源码名”记录，不能直接变成 VT Studio 的正式命名。

```txt
功能文档必须写清楚 Toonflow 源码原名，方便回查源码
功能文档同时写清楚业务语义名，方便后续设计 VT Studio
VT Studio 正式命名不使用 o_ 前缀
VT Studio 正式命名不保留明显拼写错误
VT Studio 正式命名不直接照搬 Toonflow 的接口路径
```

命名建议：

```txt
数据库表：小写 snake_case 复数，例如 projects、scripts、assets
数据库字段：小写 snake_case，例如 source_type、image_model_id、created_at
TypeScript 类型：PascalCase，例如 Project、Script、VisualManual
前端变量和服务方法：camelCase，例如 createProject、getProjectList
IPC/API 能力名：领域.动作，例如 project.create、project.update、asset.generateImage
```

参考项目旧命名在 VT Studio 中修正：

```txt
agentConfog -> agentConfig
nodel -> novel 或 node，按语义判断
Scrpt -> Script
Direcor / Directorl -> Director
Genereate -> Generate
o_project / o_script / o_assets 等 o_ 表名 -> 只作为参考源码名，不作为 VT Studio 正式表名
```

字段写法必须使用双栏或三栏：

| 业务含义 | Toonflow 源码名 | VT Studio 建议名 |
|---|---|---|
| 项目来源类型 | `projectType` | `sourceType` / `source_type` |
| 项目简介 | `intro` | `description` |
| 视觉手册 | `artStyle` | `visualManualId` / `visual_manual_id` |

## 10. 不确定信息写法

```txt

```

```txt
正确写法：需要你确认是否删除项目时同步删除项目业务文件
```

## 11. VT Studio 总落地规则

这些规则只写在总规范里，单个功能文档不重复大段描述。功能文档只写本功能的特殊差异。

```txt
renderer 不能直接访问 SQLite
renderer 不能直接访问文件系统
renderer 不能直接调用模型 SDK
renderer 不能执行供应商代码
renderer 统一调用 window.vtStudio
preload 只暴露受控 API
main ipc 只做参数接收、权限边界和服务转发
业务逻辑统一放 main/services
services 可以按功能建文件，不提前拆复杂层级
运行缓存和 userData 不放项目源码目录
```

功能文档里涉及落地时，只需要写具体能力名和特殊差异，例如：

```txt
能力名：project.create
调用链：renderer -> window.vtStudio.project.create -> main/services/project
特殊差异：Toonflow 使用 HTTP route，VT Studio 使用 IPC；业务语义不变
```

## 6. 推荐开发顺序

后续正式开发按这个顺序走，不按页面随便开工。

| 阶段 | 先做什么 | 原因 |
|---|---|---|
| 1 | 公共底层 | 后面所有页面都依赖数据库、文件、任务、模型、IPC |
| 2 | 模型配置和模型测试 | 先确认供应商、API Key、文本/图片/视频模型能真实调用，否则后面业务无法验收 |
| 3 | Agent 参数、提示词、模型提示词映射 | 让 Agent 和生成链路能稳定拿到模型和 prompt |
| 4 | 本地媒体访问服务 | 图片、视频、音频预览和测试结果展示都依赖它 |
| 5 | 基础应用壳、项目、任务中心 | 先有统一入口、项目上下文和任务可观测能力 |
| 6 | 小说、剧本 Agent、剧本 | 内容输入和剧本产出是主链路 |
| 7 | 资产、角景、生产、导出 | 资产和生产依赖前面的剧本、模型、任务 |

## 8. 更新规则

开始整理一个菜单时：

```txt
1. 更新当前焦点
2. 打开 Toonflow-web 和 Toonflow-app 对应源码
3. 更新对应 features/M-xxx.md
4. 不写代码
```

开始实现一个功能时：

```txt
1. 先读 00/01/02/03/04
2. 读对应 features/M-xxx.md
3. 重新核对参考项目源码
4. 创建或优化 docs/tasks/{真实功能ID}-{功能名}.md
5. tasks 必须写清楚改动范围、不开工范围、验收标准、用户确认点
6. 用户确认后再改代码
7. 模板、扫描、搭框架等非真实功能文档不得占用正式功能编号
```

连续执行一个阶段任务时：

```txt
1. 按 07 的顺序取下一个 task，不自己跳号，不按页面喜好改顺序。
2. 开工前先检查参考项目和当前实现是否有出入。
3. 如果有出入，先给出专业优化建议，重点看用户是否更好用、边界是否更稳、后续是否更容易复用。
4. 先把优化建议和执行结论写进对应 task 文档，再决定实现方案。
5. 没有确认点，或确认点已被用户明确确认时，直接实现，不停下来等“下一步安排”。
6. 如果 task 存在未确认点，必须停在确认，不能因为“连续执行模式”跳过。
7. 实现完成后做专项验证、typecheck、必要回归、build。
8. 更新 task 执行后记录、03 当前焦点/已完成/下一步，然后立刻进入下一个 task。
```

完成一个功能时：

```txt
1. 更新 tasks 执行后记录
2. 更新 03 已完成和下一步
3. 如有偏差，更新 04
4. 必要时更新 01/02/features
```

## 1. 职责

本文档是入口，不写所有细节。

```txt
看长期规则：00-项目规范.md
看菜单列表：02-菜单功能清单.md
看当前进度：03-执行进度.md
看偏差结论：04-对齐验收与偏差记录.md
看功能实现：features/
看单功能执行：tasks/
```

后续 AI 开工前必须先读 `00`，再读本文档，再读对应 `features/M-xxx`，最后再写或读取对应 `tasks/xxx`。

## 4. features 文档必须写到什么程度

```txt
页面入口在哪里
页面有什么区域
有哪些按钮和操作
每个按钮会打开什么弹窗或触发什么动作
表单有哪些字段、字段来源、默认值、是否必填
字段的 Toonflow 源码名是什么
字段的业务语义是什么
前端调用哪个接口
接口参数是什么
接口返回后页面怎么变
后端读写哪些表
表名只作为参考源码名，不代表 VT Studio 正式命名
是否读写本地文件
是否创建任务记录
任务状态怎么流转
是否轮询或 Socket
是否调用模型
模型调用走 SDK、vendor 脚本、HTTP 还是 ai-sdk
成功怎么提示
失败怎么提示
删除有没有级联影响
哪些点不能随意改
每个功能缺少哪些实现逻辑和细节
哪些源码未找到
哪些需要用户确认
```

`features/M-xxx.md` 里不要重复大段 VT Studio 总落地规则。总规则在 `00-项目规范.md`。功能文档只写本功能特殊点。

## 5. 不确定项规则

```txt
已经查过但没找到：写“源码未找到”
需要产品取舍：写“需要你确认”
参考项目没有但 VT Studio 要新增：写“VT Studio 增强能力”
需要 VT Studio 改动：写入 04 偏差记录
```

不要写模糊占位词。没有源码事实就写“源码未找到”，需要产品取舍就写“需要你确认”。

## 6. 开工规则

```txt
1. 先读对应 features 文档
2. 重新打开参考项目源码核对
3. 创建 tasks/xxx.md
4. tasks 开头必须先写“快速理解”，用通俗语言说明这一步要做什么
5. tasks 必须写“要做什么功能”，逐项写清每个功能怎么做
6. tasks 结尾必须写“最后大白话”，说明我准备怎么做、不会做什么
7. 优化 tasks 到可执行粒度
8. 用户确认 tasks 后再写代码
9. 完成后更新 03 和 04
```

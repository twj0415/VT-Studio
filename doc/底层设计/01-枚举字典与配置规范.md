# 枚举字典与配置规范

> 这篇是全项目“固定值”的单一来源。任何业务枚举、状态、字典、配置、错误码、Provider 能力矩阵，都以这里为准。

## 一、三类值必须分开

### 1. 强协议枚举

强协议枚举会影响数据库、Rust 业务逻辑、Tauri 前后端通信、任务状态机、Provider 调用、文件安全策略。

特点：

```text
不能让用户新增
不能页面临时定义
不能存中文
不能存数字下标
必须前后端一致
```

### 2. 可配置字典

可配置字典用于 UI 下拉、标签、筛选、模型选项、模板分类等。

特点：

```text
有稳定 code
有 label_key
有 order
有 is_enabled
有 meta
可以内置
以后可以从 DB 或插件扩展
```

### 3. UI 设计 token

UI token 只影响视觉，不进入业务协议。

包括颜色、间距、圆角、阴影、字体、动效时长、布局密度、组件尺寸等。

---

## 二、强协议枚举来源

Rust 后端是强协议枚举源头。

建议目录：

```text
src-tauri/src/core/enums/
  mod.rs
  app.rs
  project.rs
  task.rs
  storyboard.rs
  provider.rs
  template.rs
  storage.rs
  error.rs
```

示例：

```rust
#[derive(Debug, Clone, Serialize, Deserialize, sqlx::Type, specta::Type)]
#[serde(rename_all = "snake_case")]
#[sqlx(type_name = "TEXT")]
pub enum TaskStatus {
    Pending,
    Running,
    Succeeded,
    Failed,
    Cancelled,
}
```

数据库存：

```text
pending / running / succeeded / failed / cancelled
```

不存：

```text
成功 / 失败 / 1 / 2 / SUCCESS
```

前端类型从 Rust 生成，或第一版统一镜像到：

```text
src/shared/enums/generated.ts
src/shared/types/generated.ts
```

页面禁止私写 `'running'`、`'succeeded'`、`'topic'` 这类魔法字符串。

## 三、应用基础枚举

### AppLocale

软件界面语言。

```text
zh-CN / en-US
```

### ContentLanguage

视频内容语言。

```text
zh-CN / en-US / ja-JP / ko-KR
```

第一版只需实现：

```text
zh-CN / en-US
```

`AppLocale` 和 `ContentLanguage` 必须分开：

```text
AppLocale       控制菜单、按钮、错误提示、设置页、状态文案
ContentLanguage 控制脚本生成语言、字幕语言、TTS voice 默认值、Prompt 输出语言
```

### ThemePreset（主题）

> 主题不是一个主色，而是一整套视觉皮肤；可深可浅。

```text
graphite / aurora / ember / porcelain / sandstone
```

切换主题必须同时切换背景、面板、卡片、边框、文字、阴影、辉光和强调色。配置示例：

```json
{
  "theme_preset": "graphite"
}
```

### LayoutDensity

```text
compact / comfortable
```

## 四、项目相关枚举

### InputType

```text
topic / paste / article / novel / material
```

### InputProcessMode

内容进入生产线后的处理模式：

```text
generate / fixed / asset_based
```

说明：

```text
generate     AI 重新组织内容，例如主题、文章、小说事件生成脚本
fixed        用户文案一字不改，只做机械切分
asset_based  用户已有图片/视频素材，先分析素材再组织成片
```

### ProjectLifecycle

Project 是容器，不承载生成流程状态。项目生命周期只表示项目自身是否可用：

```text
draft / active / archived / deleted
```

项目列表要显示“生成中 / 成功 / 失败”时，从 `latest_task.task_status` 读取，不从 Project 读取。

### VideoType

第一版作为内置字典，后续允许扩展：

```text
knowledge / story / emotion / sales / list / tutorial / commentary / news
```

### AspectRatio

协议 code 使用稳定名称：

```text
vertical_9_16 / horizontal_16_9 / square_1_1
```

不要用 `9:16 / 16:9 / 1:1` 作为协议 code。真实比例放字典 meta：

```json
{
  "code": "vertical_9_16",
  "label_key": "dict.aspectRatio.vertical_9_16",
  "meta": {
    "ratio": "9:16",
    "width": 1080,
    "height": 1920
  }
}
```

## 五、任务相关枚举

### WorkflowType

制作流程和内容来源必须分开。`WorkflowType` 表示最后走哪条生产流程。

```text
image_to_video
数字入口：digital_human
素材剪辑：material_edit
纯图文成片：image_slideshow
```

当前主线：

```text
image_to_video = 输入文字 → 分镜 → 生图 → 图生视频 → 合成
```

### TaskKind

```text
image_to_video_pipeline
script_only
storyboard_only
image_only
video_only
tts_only
subtitle_only
cover_only
render_only
compose_only
export_only
digital_human
material_edit
image_slideshow
canvas_edit
```

说明：旧的 `full_pipeline` 容易被误解成“脚本/TTS/字幕/封面全都必做”，后续应按 `workflowType` 生成步骤。

### TaskStatus

任务整体状态只保留大状态：

```text
pending / running / succeeded / failed / cancelled
```

不放 `retrying / skipped / waiting_user`，这些是步骤级状态。

### TaskStepKind

```text
project_init
script_generation
script_review
storyboard_generation
storyboard_review
image_prompt_generation
image_generation
image_review
video_prompt_generation
video_generation
video_review
tts_generation
subtitle_generation
asset_analysis
cover_generation
cover_review
template_rendering
segment_composition
final_composition
export
canvas_editing
cleanup
```

用户确认类步骤统一用 `review`，不用 `confirm`。

TaskStepKind 分组规则：

```text
image_to_video 主线步骤：project_init / storyboard_generation / storyboard_review / image_prompt_generation / image_generation / image_review / video_prompt_generation / video_generation / video_review / final_composition / export / cleanup
后续增强步骤：script_generation / script_review / tts_generation / subtitle_generation / cover_generation / cover_review / template_rendering / segment_composition
扩展 workflow 步骤：asset_analysis / canvas_editing 等只在对应 workflow 中使用。
```

### TaskStepStatus

```text
pending
running
retrying
succeeded
failed
skipped
cancelled
waiting_user
```

`waiting_user` 用于脚本检查、分镜检查、图片确认、封面确认等人工节点。

StoryboardItem 里的 `image_status / audio_status / video_status / subtitle_status`，以及 ImageCandidate / VideoSegment 的资源状态复用 `TaskStepStatus`，不另造状态体系。

## 十一、字典项结构

第一版字典来源：

```text
不建 dictionaries 数据库表。
后端在 Rust 文件 src-tauri/src/domain/dictionary/builtin.rs 集中维护内置字典。
前端必须通过 get_dictionary / list_dictionaries / useDictOptions 获取，不允许页面私写 options。
后续如果需要运营配置，再迁移到数据库字典表。
```

前端：

```ts
export interface DictItem<TCode extends string = string> {
  code: TCode
  labelKey: string
  descriptionKey?: string
  group?: string
  icon?: string
  colorToken?: string
  order: number
  isEnabled: boolean
  meta?: Record<string, unknown>
}
```

后端：

```rust
pub struct DictItem {
    pub code: String,
    pub label_key: String,
    pub description_key: Option<String>,
    pub group: Option<String>,
    pub icon: Option<String>,
    pub color_token: Option<String>,
    pub order: i32,
    pub is_enabled: bool,
    pub meta: serde_json::Value,
}
```

字典不存中文：

```json
{
  "code": "running",
  "label_key": "dict.taskStatus.running",
  "color_token": "status.running",
  "order": 20,
  "is_enabled": true
}
```


## 十三、配置分层

配置必须分层，不要全部塞进一个大 JSON。

```text
AppConfig       软件偏好：界面语言、主题、密度
SystemConfig    工作区、输出目录、sidecar 路径
ProviderConfig  供应商连接、密钥别名、base_url；模型能力放 ProviderModel
PipelineConfig  默认生成参数、并发数、重试次数
UiConfig        高级选项、布局偏好
ExportConfig    导出格式、码率、分辨率策略
```

### AppConfig

```json
{
  "app_locale": "zh-CN",
  "theme_preset": "graphite",
  "layout_density": "comfortable"
}
```

### PipelineConfig

```json
{
  "default_content_category": "knowledge",
  "default_aspect_ratio": "vertical_9_16",
  "default_content_language": "zh-CN",
  "default_target_duration_seconds": 60,
  "default_target_scene_count": 8,
  "max_concurrent_provider_calls": 2,
  "retry_max_attempts": 3,
  "retry_backoff_seconds": 2
}
```

### ProviderConfig

```json
{
  "provider_id": "provider_xxx",
  "provider_kind": "llm",
  "vendor": "openai_compatible",
  "display_name": "DeepSeek",
  "base_url": "https://api.deepseek.com/v1",
  "key_alias": "deepseek_main",
  "is_enabled": true,
  "config": {}
}
```

真实 API Key 只进系统钥匙串，配置里只存 `key_alias`。

## 十四、任务运行时快照

任务创建时必须冻结运行配置：

```json
{
  "provider_snapshot": {},
  "model_snapshot": {},
  "pipeline_config_snapshot": {},
  "template_snapshot": {},
  "retry_policy_snapshot": {}
}
```

原因：

```text
1. 用户在任务运行中改 Provider，不能影响正在运行的任务。
2. 断点续跑必须基于当时的配置。
3. 历史任务要可追溯。
```

## 十五、Provider 能力矩阵

每个模型必须声明能力，后端负责校验，不只靠前端禁用。

示例：

```json
{
  "provider_kind": "image",
  "vendor": "dashscope",
  "model_name": "wanx2.1-t2i",
  "ability_types": ["text_to_image"],
  "input_modalities": ["text"],
  "output_modalities": ["image"],
  "feature_flags": [
    "seed",
    "negative_prompt",
    "aspect_ratio"
  ],
  "limits": {
    "max_prompt_length": 1000,
    "supported_aspect_ratios": [
      "vertical_9_16",
      "horizontal_16_9",
      "square_1_1"
    ]
  }
}
```

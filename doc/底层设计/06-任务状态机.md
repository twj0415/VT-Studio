# 任务状态机

> 这篇定义 Task / TaskStep 的状态流转。枚举取值以 `01-枚举字典与配置规范.md` 为准，字段结构以 `数据结构.md` 为准。

## 一、Task 和 TaskStep 的边界

```text
Task       一次生成动作的整体记录
TaskStep   任务里的一个可执行/可等待/可重试步骤
```

Task 只关心整体状态：

```text
pending / running / succeeded / failed / cancelled
```

TaskStep 关心细状态：

```text
pending / running / retrying / succeeded / failed / skipped / cancelled / waiting_user
```

不要把 `retrying / skipped / waiting_user` 放到 TaskStatus。

---

## 二、TaskStepKind 顺序

当前 `workflowType=image_to_video` 主线顺序：

```text
project_init
storyboard_generation
storyboard_review
image_prompt_generation
image_generation
image_review
video_prompt_generation
video_generation
video_review
final_composition
export
cleanup
```

增强步骤按 workflow 配置插入，不是当前主线必经步骤：

```text
script_generation
script_review
tts_generation
subtitle_generation
cover_generation
cover_review
template_rendering
segment_composition
```

用户确认类步骤统一使用 `review`，不用 `confirm`。

## 三、状态流转

### 自动步骤

```text
pending → running → succeeded
pending → running → retrying → running → succeeded
pending → running → failed
pending → skipped
running → cancelled
```

### 人工确认步骤

```text
pending → waiting_user → succeeded
pending → waiting_user → cancelled
```

适用场景：

```text
script_review
storyboard_review
image_review
video_review
cover_review
```

---

## 四、断点续跑

断点续跑规则：

```text
1. 读取 TaskStep 状态。
2. 找到第一个 failed / pending / retrying 的步骤。
3. 如果遇到 waiting_user，必须等用户确认。
4. 校验该步骤需要的输入文件是否存在。
5. 已 succeeded 的步骤不重复跑，除非用户强制重跑。
6. 重跑某一步时，必须按影响范围重置后续步骤状态。
```

断点续跑必须基于：

```text
数据库状态
+ 文件存在性
+ 任务运行时快照
```

---

## 五、任务运行时快照

任务创建时必须冻结：

```text
provider_snapshot
model_snapshot
pipeline_config_snapshot
template_snapshot
retry_policy_snapshot
```

原因：

```text
用户在任务运行中改配置，不能影响当前任务。
历史任务要可追溯。
失败续跑要使用当时配置。
```

---

## 六、状态重置规则

用户修改内容后，只重置受影响的后续步骤，不全项目重跑。

示例：

| 修改内容 | 需要重置 |
|---|---|
| narration_text / source_text | storyboard_generation / image_prompt_generation / audio_status / subtitle_status / video_prompt_generation / final_composition |
| visual_description / character_ids_json / location_id | image_prompt_generation 后续、image_status、video_status、final_composition |
| image_prompt | image_status / video_status / final_composition |
| selected_image_id | video_status / final_composition |
| video_prompt | video_status / final_composition |
| selected_video_segment_id | final_composition |
| audio_path / audio_duration_seconds | subtitle_status / final_composition（启用音频时） |
| template_params | template_rendering / final_composition（启用模板增强时；字段必须先在数据结构中定义） |

---

## 七、最终状态判定

```text
所有必需 TaskStep succeeded → Task.succeeded
任一必需 TaskStep failed 且不可重试 → Task.failed
用户取消 → Task.cancelled
存在 waiting_user → Task.running，但当前步骤显示等待用户
```


---

## 八、TaskStep 和资源状态分层

`TaskStep.step_status` 是步骤级整体状态，`StoryboardItem.image_status / audio_status / video_status / subtitle_status`、`ImageCandidate.status`、`VideoSegment.status` 是单项资源状态。二者不互相替代。

规则：

```text
1. TaskStep 成功条件由该步骤覆盖范围内的 StoryboardItem / ImageCandidate / VideoSegment 状态汇总决定。
2. 单个 StoryboardItem / ImageCandidate / VideoSegment 失败必须写 last_error_json，并汇总到 TaskStep.error_json。
3. Task.last_error_json 只记录导致任务阻断的最后一个错误。
4. 资源状态复用 TaskStepStatus，但按字段限制取值：image_status / video_status 可使用 waiting_user；audio_status 不使用 waiting_user。
```

# 技术栈与架构

> 这篇定义“用什么技术、各部分怎么分工、进程怎么协作”。详细工程规范不要写在这里，分别看 `00-工程规范总纲.md`、`02-前端工程规范.md`、`03-后端工程规范.md`。
> 当前开发顺序以项目 `plan/阶段路线图.md` 为准。

## 一、技术栈（落地技术组合）

```text
前端：    Vue 3 + Vite + TypeScript + Pinia + Naive UI + vue-i18n
桌面壳：  Tauri 2
后端核心：Rust + Tokio + Reqwest + Serde（thiserror/anyhow 错误，tracing 日志）
数据库：  SQLite（sqlx 或 rusqlite，最终以实现选型为准）
视频处理：FFmpeg + FFprobe（sidecar 打包）
模板渲染：HTML 模板 + Playwright sidecar 控制随包 Chromium
类型协议：Rust DTO/enum 作为源头，建议 specta / tauri-specta 生成 TypeScript 类型
LLM：     OpenAI-compatible Chat Completions 起步，后续扩展多 Provider
TTS：     Edge TTS 起步，后续扩展云 TTS / workflow TTS
图片：    DashScope Image 或 OpenAI Image 起步，后续扩展 workflow / 多云 API
视频：    image_to_video Provider 起步，后续扩展 text_to_video / 首尾帧 / 数字人
密钥：    系统钥匙串（keyring crate）
```

### 选型理由

- **Tauri 而非 Electron**：更轻、包更小，Rust 后端天然适合本地文件、进程、任务和安全边界。
- **Rust 做调度 + 安全 + 文件层**：不自己实现 AI 和视频编码，AI 调用交给 Provider，视频编码交给 FFmpeg。
- **HTML 模板 + Playwright/Chromium 截图**：适合后续字幕、封面、画面包装和轻动效；不是当前 image_to_video 主线前置条件。
- **TTS**：配音、字幕、口播 workflow 需要，但不阻塞当前图生视频主线。
- **SQLite**：桌面端本地持久化足够稳定，适合任务、配置、历史、项目数据。

## 二、分层架构

```text
Vue 前端（只做交互和展示，不承载生产线逻辑）
  ↓ Tauri invoke 调用 / event 推进度
Tauri Rust 后端
  ↓
核心服务层 Core Services
  ├── Project Service       项目和设定集
  ├── Task Service          任务/步骤的增删查、状态机
  ├── Pipeline Engine       按 workflowType 串起每一步
  ├── Prompt Skill Engine   提示词技能（分镜/生图词/视频词…）
  ├── Provider Manager      统一管 LLM/TTS/图片/视频/VLM/workflow
  ├── Template Renderer     HTML 模板 + Chromium 截图（后续增强）
  ├── FFmpeg Service        视频拼接、转码、字幕、混音
  ├── Asset Service         图片/视频/音频/BGM/字体/参考图
  ├── Storage Service       文件读写，所有路径过 PathGuard
  ├── Config Service        配置分层管理
  ├── Dictionary Service    枚举字典和可配置选项出口
  └── Path Guard            路径安全，所有文件操作必经
  ↓
SQLite + 文件系统 + Sidecars(FFmpeg/FFprobe/Chromium)
```

核心原则：

```text
前端不碰生产线逻辑。
业务代码不直接拼 FFmpeg 命令。
业务代码不直接调具体 Provider。
业务代码不直接读写任意路径。
状态、枚举、字典、配置统一走底层规范。
```

## 五、核心生产线（端到端流程）

### 当前主线：image_to_video

```text
内容导入 / 项目初始化
→ 分镜生成
→ 分镜检查
→ 生图提示词 / 反推提示词
→ AI 生图
→ 图片选择
→ 视频提示词
→ 图生视频
→ 视频片段确认
→ 最终合成
→ 导出 MP4
→ 历史
```

对应 TaskStepKind 建议：

```text
project_init
storyboard_generation
storyboard_review
image_prompt_generation
image_generation
image_review
video_prompt_generation
video_generation
video_review
final_composition
export
cleanup
```

后续增强步骤：

```text
script_generation / script_review
tts_generation
subtitle_generation
cover_generation / cover_review
template_rendering
segment_composition
```

这些不是不做，而是不阻塞当前主线。

完整状态流转见 `06-任务状态机.md`，枚举取值见 `01-枚举字典与配置规范.md`。

### TTS 时序说明

TTS 在配音、字幕、口播、纯图文模板成片等 workflow 中很重要，可以用真实音频时长回填 `durationSeconds`，保障声画同步。

但在当前 `image_to_video` 主线里，先以分镜设定时长和视频模型输出时长为准，不要求 TTS 先于生图/图生视频执行。

## 六、Sidecar 与环境检测

FFmpeg/FFprobe、Playwright driver 与 Chromium 作为 sidecar 随包发布。`resources/bin` 是打包源目录，运行时通过 FileBucket 解析为 `sidecars/*`：

```text
resources/bin/ffmpeg.exe
resources/bin/ffprobe.exe
```

启动时检测：

```text
是否存在
是否可执行
版本是否可读取
Playwright driver / Chromium 是否可启动
```

缺失时在设置页给明确提示，不让生产线在最后一步才报错。

## 七、能力接入方式

所有能力都按同一套架构接入：

```text
Vue 页面 / 组件
→ Tauri command
→ Rust Service
→ ProviderManager / FFmpegService / TemplateRenderer / StorageService
→ SQLite + 文件系统
→ Task / TaskStep / ProgressEvent 回写状态
```

AI 视频、数字人、画布精修、商品视频等能力不单独开一套技术栈；只是在 Provider、Service、TaskStep 和功能页面上增加对应实现。

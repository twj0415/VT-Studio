# 前端工程规范

> 这篇定义 Vue 前端怎么组织、怎么调用后端、怎么使用字典、多语言和主题。功能模块只写页面需要什么能力，不重复定义前端工程规则。

## 一、技术栈

```text
Vue 3
Vite
TypeScript
Pinia
Vue Router
Naive UI
vue-i18n
```

建议从初始阶段预留 Rust → TypeScript 类型生成能力，可用 `specta / tauri-specta` 或等价方案。第一版如手写镜像类型，也必须集中维护。

## 三、分层职责

| 层 | 职责 |
|---|---|
| `app` | 应用启动、路由、全局 Provider |
| `shared` | 全局基础能力：API、字典、类型、主题、i18n、工具、基础组件 |
| `entities` | 核心业务实体：Project、Task、StoryboardItem、ImageCandidate、VideoSegment、Provider、Template、Asset |
| `features` | 业务动作：导入、编辑、确认、重生成、测试 Provider |
| `widgets` | 页面级组合块：侧边栏、任务进度、分镜列表、日志面板 |
| `pages` | 路由页面，只做页面编排 |

禁止：

```text
所有组件都丢 components
所有 API 都丢 api.ts
所有类型都丢 types.ts
页面直接写复杂业务
页面直接调 invoke
页面直接写固定 options
```

## 四、Tauri invoke 封装

页面禁止直接写：

```ts
invoke('xxx')
```

必须统一封装：

```text
src/shared/api/invoke.ts
src/shared/api/errors.ts
```

业务 API 放在实体模块：

```text
entities/task/api.ts
entities/project/api.ts
entities/provider/api.ts
entities/template/api.ts
```

页面只能调用：

```text
feature action
entity api
store action
```

示例：

```ts
export async function invokeCommand<TResponse, TPayload = void>(
  command: string,
  payload?: TPayload,
): Promise<TResponse> {
  try {
    return await tauriInvoke<TResponse>(command, payload as Record<string, unknown>)
  } catch (error) {
    throw normalizeTauriError(error)
  }
}
```

## 五、API 模块规范

业务 API 按实体划分：

```ts
// entities/task/api.ts
export function getTaskDetail(taskId: string) {
  return invokeCommand<TaskDetail>('get_task_detail', { taskId })
}

export function cancelTask(taskId: string) {
  return invokeCommand<void>('cancel_task', { taskId })
}
```

命名必须贴近后端 command，不要在前端自造另一套语义。

## 六、Pinia Store 规范

推荐 Store：

```text
useAppStore
useConfigStore
useDictStore
useProjectStore
useTaskStore
useStoryboardStore
useProviderStore
useTemplateStore
useAssetStore
useLogStore
```

规则：

```text
1. Store 只放跨页面共享状态。
2. Store 可以调用 api。
3. Store 不写复杂生产线逻辑。
4. 表单临时状态不要全塞 Store。
5. 生产线逻辑不进前端 Store。
```

禁止：

```text
在 Store 拼 Provider 参数
在 Store 拼 FFmpeg 命令
在 Store 处理文件路径安全
在 Store 维护任务状态机
```

## 八、字典 options 使用

页面禁止：

```ts
const options = [
  { label: '开启', value: true },
  { label: '关闭', value: false },
]
```

必须：

```ts
const taskStatusOptions = useDictOptions('task_status')
const workflowTypeOptions = useDictOptions('workflow_type')
const inputTypeOptions = useDictOptions('input_type')
const aspectRatioOptions = useDictOptions('aspect_ratio')
```

字典项只存 `labelKey`，展示时走：

```ts
t(item.labelKey)
```

### 10.1 组件库统一规则

第一版统一使用 `Naive UI` 作为基础组件库，页面和业务组件不得随手用原生控件重新造一套视觉。

必须：

```text
按钮      → NButton / BaseButton
输入框    → NInput / BaseInput
选择器    → NSelect / BaseDictSelect
卡片      → NCard / BaseCard
标签      → NTag / BaseStatusTag
表格      → NDataTable
弹窗      → NModal / NDialogProvider / BaseConfirmDialog
表单      → NForm / NFormItem
布局间距  → NSpace / NGrid / NLayout，或项目统一 layout class
```

允许自定义 CSS 的范围：

```text
AppShell / WorkbenchShell / StepBar / StoryboardItemPreview 这类产品级布局和业务视觉组件。
```

禁止：

```text
1. 页面直接写 <button>、<select>、<input> 再手搓一套样式。
2. 每个页面自己定义 primary / ghost / danger 按钮样式。
3. 绕过 Naive UI 和 Base 组件重复实现表单、弹窗、标签、表格。
4. 为了局部好看写一次性组件样式，导致主题切换不一致。
```

页面优先组合 `Naive UI + shared/components/base + widgets`，只有文档明确的产品级复杂组件才允许定制。

## 十一、前端禁止事项

```text
1. 页面直接调用 Tauri invoke。
2. 页面直接写业务枚举字符串。
3. 页面直接写固定 options。
4. 页面直接拼本地路径。
5. 页面直接判断状态颜色。
6. 页面硬编码用户可见文案。
7. 页面承载生产线逻辑。
8. Store 维护后端任务状态机。
9. 前端保存真实 API Key。
10. 前端把任意系统路径直接交给后端使用。
```

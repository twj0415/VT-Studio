# 模型适配与工作流注册规范

> 这篇定义多模型、ComfyUI 自托管工作流、RunningHub 云端工作流怎么统一接入。目标是让开发者按本文就能实现 ProviderAdapter、ModelRegistry、WorkflowRegistry、workflow preset、模型 / 工作流页面测试和任务执行，不在功能模块里临时写模型分支。

## 一、目标和边界

本规范解决三类问题：

```text
1. API 模型很多，不能在业务代码里按 model_name 写 if/else。
2. ComfyUI 工作流节点差异很大，不能让功能模块直接拼 workflow JSON。
3. 云端 ComfyUI / RunningHub 与本地 ComfyUI 要统一成同一套可执行能力。
```

边界：

```text
provider_models        只管理 API 模型能力矩阵
workflow_presets       只管理 ComfyUI / RunningHub 工作流预设
ProviderAdapter        只处理供应商协议差异
ModelRegistry          只解析、校验、返回 API 模型能力
WorkflowRegistry       只解析、校验、返回 workflow preset
ProviderManager        是业务层唯一调用入口
```

功能模块只提交 `ImageRequest / VideoRequest / WorkflowRequest`，不得直接判断 OpenAI、DashScope、ComfyUI、RunningHub 或具体模型名。

---

## 二、核心规则

必须：

```text
1. 所有模型调用必须走 ProviderManager。
2. API 模型能力写入 provider_models。
3. ComfyUI / RunningHub 工作流写入 workflow_presets。
4. 工作流执行前必须校验 param_schema / node_map / output_map。
5. 任务创建时必须冻结 provider_snapshot / model_snapshot / workflow_snapshot。
6. Provider 返回远程 URL 时，必须下载转存到本地相对路径后再入库。
7. 真实 API Key 只能由 key_alias 从系统 keyring 读取。
```

禁止：

```text
1. 禁止业务代码按具体 model_name 写分支。
2. 禁止前端直接拼 Provider 参数。
3. 禁止执行用户上传的 JS / TS / Python Provider 代码。
4. 禁止执行未注册 workflow。
5. 禁止绕过 param_schema / node_map / output_map 直接改 workflow JSON。
6. 禁止把 API Key 写入 SQLite / JSON / YAML / 日志 / 导出包。
7. 禁止把远程结果 URL 直接写入数据库作为最终产物。
```

## 五、ProviderAdapter 分工

ProviderAdapter 只处理供应商协议，不处理业务流程。

示例：

```text
OpenAICompatibleLlmProvider
OpenAIImageProvider
DashScopeImageProvider
SeedreamImageProvider
KlingVideoProvider
SeedanceVideoProvider
ComfyUiWorkflowProvider
RunningHubWorkflowProvider
EdgeTtsProvider
```

Adapter 必须遵守：

```text
1. 输入使用内部 DTO。
2. 输出使用内部 DTO。
3. 错误映射为 ProviderError / AppError。
4. 不写 DB。
5. 不读真实 API Key，只接收 ProviderManager 注入的 secret。
6. 远程结果必须交给 StorageService 下载转存。
```

## 六、ModelRegistry

ModelRegistry 负责 API 模型能力：

```text
1. 从 provider_models 读取模型。
2. 解析 ability_types_json / input_modalities_json / output_modalities_json / feature_flags_json / limits_json。
3. 返回可执行 API 模型候选项。
4. 校验 ability_type、输入媒体、输出媒体、时长、比例、参考图数量。
5. 为任务生成 model_snapshot。
```

示例能力：

```json
{
  "provider_kind": "video",
  "vendor": "kling",
  "model_name": "kling-v2-6",
  "ability_types": ["text_to_video", "image_to_video"],
  "input_modalities": ["text", "image"],
  "output_modalities": ["video"],
  "feature_flags": ["reference_image", "duration", "resolution", "aspect_ratio"],
  "limits": {
    "duration_seconds": { "min": 5, "max": 10, "integer": true },
    "supported_aspect_ratios": ["vertical_9_16", "horizontal_16_9"],
    "resolutions": ["720p", "1080p"],
    "max_reference_images": 1
  },
  "api_contract_verified": true
}
```

---

## 七、WorkflowRegistry

WorkflowRegistry 负责 workflow preset：

```text
1. 扫描内置 resources/workflows。
2. 解析 preset.json。
3. 写入或更新 workflow_presets。
4. 校验 workflow_key / workflow_id / param_schema / node_map / output_map。
5. 返回可执行 workflow 候选项。
6. 为任务生成 workflow_snapshot。
```

workflow preset 是业务可执行单元，不等同于底层模型。Flux、SDXL、Wan、LTX、AnimateDiff 都只存在于 workflow 内部，业务层只知道 preset 支持什么能力。

## 八、数据库表

`provider_models` 只管 API 模型；`workflow_presets` 只管 ComfyUI / RunningHub 工作流。两者不能混在一张表里。

```sql
CREATE TABLE workflow_presets (
  workflow_preset_id TEXT PRIMARY KEY,
  provider_id TEXT NOT NULL,
  vendor TEXT NOT NULL,
  workflow_key TEXT NOT NULL,
  workflow_id TEXT,
  display_name TEXT NOT NULL,
  workflow_version TEXT NOT NULL,
  ability_types_json TEXT NOT NULL,
  input_modalities_json TEXT NOT NULL,
  output_modalities_json TEXT NOT NULL,
  limits_json TEXT NOT NULL,
  param_schema_json TEXT NOT NULL,
  node_map_json TEXT NOT NULL,
  output_map_json TEXT NOT NULL,
  is_builtin INTEGER NOT NULL DEFAULT 0,
  is_enabled INTEGER NOT NULL DEFAULT 1,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  FOREIGN KEY(provider_id) REFERENCES providers(provider_id) ON DELETE CASCADE,
  UNIQUE(provider_id, workflow_key, workflow_version)
);
```

字段规则：

```text
workflow_preset_id      本项目内部 ID
provider_id             指向 providers.provider_id
vendor                  comfyui / runninghub
workflow_key            本地资源路径或逻辑 key
workflow_id             RunningHub 云端 workflow_id，本地 ComfyUI 可为空
workflow_version        preset 版本，变更 node_map 时必须升级
ability_types_json      text_to_image / image_to_image / text_to_video / image_to_video 等
param_schema_json       对外可传参数 schema
node_map_json           参数到 ComfyUI 节点字段的映射
output_map_json         输出类型到节点结果的映射
```

## 十三、node_map / output_map 规则

`node_map` 路径格式：

```text
{node_id}.inputs.{field_name}
```

示例：

```text
12.inputs.text
7.inputs.image
31.inputs.seed
```

规则：

```text
1. node_id 必须存在于 workflow_api.json。
2. inputs 字段必须存在。
3. field_name 必须存在，除非该节点允许动态输入。
4. 写入前必须做类型转换和 schema 校验。
5. output_map 必须声明 image / video / audio 的来源。
```

缺失节点返回：

```text
workflow.invalid_node_map
```

输出缺失返回：

```text
workflow.output_missing
```

## 十四、ComfyUI selfhost 执行流程

```text
1. 根据 workflow_preset_id 读取 workflow preset。
2. 校验 provider 可用、base_url 可达。
3. 校验 ability_type、输入媒体、时长、比例、参考图数量。
4. 读取 workflow_api.json。
5. 按 param_schema 校验请求参数。
6. input_image / reference_images 是本地资产时，先上传到 ComfyUI。
7. 按 node_map 修改 workflow JSON。
8. 调 ComfyUI prompt API 提交任务。
9. 轮询任务状态。
10. 按 output_map 解析输出。
11. 下载或复制输出文件到任务目录。
12. 用 ffprobe / 图片 probe 校验真实媒体信息。
13. 写入 artifacts，并回写 ImageCandidate / VideoSegment 当前产物路径。
14. 返回 WorkflowResponse。
```

---

## 十五、RunningHub 执行流程

```text
1. 根据 workflow_preset_id 读取 workflow preset。
2. 校验 workflow_id 非空。
3. ProviderManager 通过 key_alias 读取 RunningHub API Key。
4. 校验 ability_type、输入媒体、时长、比例、参考图数量。
5. 本地资产上传到 RunningHub 所需输入位。
6. 按 node_map 转换 RunningHub 参数。
7. 提交 workflow_id 任务。
8. 轮询任务状态。
9. 按 output_map 解析远程输出 URL。
10. 下载到任务目录。
11. probe 校验媒体信息。
12. 写入 artifacts，并回写 ImageCandidate / VideoSegment 当前产物路径。
13. 返回 WorkflowResponse。
```

RunningHub 不允许把远程 URL 当最终产物保存。

## 十六、模型 / 工作流页面规则

模型 / 工作流是独立全局入口，不放在系统设置里。页面必须拆成三层：

```text
1. Provider 连接配置
2. API 模型能力配置
3. Workflow Preset 配置
```

系统设置只保留主题、路径、默认值、密钥安全、日志诊断等全局运行参数。

前端选择器统一调用：

```text
list_executable_media_options
```

展示时可以混合显示：

```text
OpenAI Image API
DashScope 图片模型
ComfyUI Flux 生图工作流
RunningHub Wan 图生视频工作流
```

但后端必须保留来源：

```text
sourceType = provider_model
sourceType = workflow_preset
```

## 十七、生图 / 视频输入规划

前端不能按固定表单写死模型输入。业务页面选择模型或工作流后，必须先请求输入规划。

用户侧名称：

```text
生图输入规划
视频输入规划
生图缺项检查
视频缺项检查
模型输入要求
动态表单
```

技术 DTO 可以使用：

```text
ImageInputPlan
VideoInputPlan
```

### 生图输入规划

生图前必须合并：

```text
图片类型：分镜图 / 角色参考图 / 场景参考图 / 风格参考图 / 道具图 / 尾帧图 / 控制图 / 封面图
模型能力：text_to_image / image_to_image / multi_reference_image / inpaint / control_image / workflow
项目数据：StoryboardItem、Bible、素材库引用、已选候选图
工作流配置：paramSchema、nodeMap、outputMap、requiredAssets
```

返回内容：

```text
requiredInputs：必填输入
optionalInputs：可选输入
missingInputs：当前缺什么
uiSchema：前端如何渲染动态表单
normalizedParams：后端可执行的标准参数
```

### 视频输入规划

视频前必须合并：

```text
视频模型能力
已选分镜图
尾帧图
角色参考图
风格参考图
pose / depth / mask
workflow preset 参数
```

典型要求：

```text
普通图生视频：startFrame
首尾帧视频：startFrame + endFrame
参考图视频：startFrame + characterReference / styleReference
ComfyUI / RunningHub：workflowPresetId + paramSchema + requiredAssets
无分镜图模型：资产图 + prompt + 参数
```

约束：

```text
1. 前端只渲染输入规划返回的字段。
2. 缺项必须在生成前提示，不允许提交后才失败。
3. 业务页面不得判断具体 vendor。
4. workflow preset 必须声明 requiredAssets，不能靠前端猜。
5. 任务快照必须保存输入规划结果。
```

## 十八、任务快照

任务创建时必须冻结：

```json
{
  "provider_snapshot": {},
  "model_snapshot": {},
  "workflow_snapshot": {},
  "capability_snapshot": {}
}
```

规则：

```text
1. 断点续跑读取任务快照，不读取最新全局配置。
2. 用户修改 provider_models 不影响历史任务。
3. 用户修改 workflow_presets 不影响历史任务。
4. workflow_version 变更后，新任务使用新版本，旧任务仍可复现旧参数。
```

## 二十、验收用例

必须通过：

```text
1. 能扫描内置 workflow preset 并写入 workflow_presets。
2. 能在模型 / 工作流页看到 ComfyUI / RunningHub workflow preset。
3. 未配置 ComfyUI endpoint 时，workflow 显示不可用且说明原因。
4. 参数超出能力矩阵时，后端拒绝。
5. 缺少 node_map 对应节点时，返回 workflow.invalid_node_map。
6. 缺少 output_map 输出时，返回 workflow.output_missing。
7. 图生视频 workflow 能上传输入图、提交任务、下载 mp4。
8. API 图片模型和 ComfyUI 生图 workflow 能在同一个选择器展示。
9. 修改 Provider 配置后，旧任务使用 snapshot 不受影响。
10. 日志和诊断包不出现 API Key。
11. 普通图生视频只要求首帧图。
12. 首尾帧视频能提示缺少尾帧图。
13. ComfyUI workflow 能按 requiredAssets 动态渲染 pose / depth / mask 等输入。
```

## 二十一、第一版实现范围

第一版建议顺序：

```text
1. OpenAI-compatible LLM。
2. Edge TTS。
3. OpenAI / DashScope 图片 API。
4. ComfyUI selfhost workflow preset。
5. RunningHub workflow preset。
6. 视频 API 模型，如 Kling / Seedance / Luma / Runway。
```

第一版不做：

```text
1. 用户自定义 Provider 代码。
2. 任意 workflow 文件直接执行。
3. 自动解析任意 ComfyUI workflow 节点语义。
4. 前端直接编辑 workflow JSON。
```

# 工程规范总纲

> 这篇定义全项目最高优先级的工程纪律。功能模块只写“某个功能怎么做”，底层规则统一看这里和同目录规范文档。

## 一、这篇负责什么

本文件负责统一以下原则：

```text
前端不承载生产线逻辑
后端是强协议源头
页面禁止魔法字符串
字典统一出口
配置分层管理
任务统一调度
Provider 统一调用
文件统一管控
密钥单独保护
UI 统一主题
多语言统一 key
```

详细规范分散在：

| 文档 | 负责什么 |
|---|---|
| `01-枚举字典与配置规范.md` | 枚举、字典、配置、错误码、任务快照、Provider 能力矩阵 |
| `02-前端工程规范.md` | Vue 目录结构、invoke 封装、Store 边界、字典 options、组件拆分 |
| `03-后端工程规范.md` | Rust 分层、Command、Service、Repository、Pipeline、Provider、Storage |
| `04-UI主题与国际化规范.md` | UI 风格、主题 token、Naive UI themeOverrides、i18n、状态色 |
| `数据结构.md` | Project / Bibles / StoryboardItem / ImageCandidate / VideoSegment / Task 字段定义 |
| `Provider与安全.md` | Provider Manager、密钥、PathGuard、重试、限流、安全红线 |

## 二、最高优先级规则

后续开发必须遵守：

```text
1. 页面不直接写业务枚举字符串。
2. 页面不直接写固定 options。
3. 页面不直接调用 Tauri invoke。
4. 页面不直接拼接本地文件路径。
5. 页面不直接调用 Provider。
6. 后端 command 不写业务逻辑。
7. Provider 不写数据库。
8. Repository 不调用 Provider。
9. 文件读写必须走 StorageService + PathGuard。
10. 密钥只进系统钥匙串，不进 SQLite，不进配置文件，不进日志。
11. 所有任务状态必须走 TaskService。
12. 所有 AI 模型调用必须走 ProviderManager。
13. 所有用户可见文案必须走 i18n。
14. 所有主题颜色必须走 theme token。
15. 所有可扩展选项必须走 Dictionary Service。
```

## 四、命名总原则

```text
Rust / SQLite / JSON 存储层：snake_case
前端 TypeScript：camelCase
Tauri 边界：用 serde rename 显式映射
枚举 code：snake_case
数据库枚举值：稳定 code，不存中文，不存数字下标
```

字段命名必须语义明确：

| 类型 | 命名规则 | 示例 |
|---|---|---|
| ID | `_id` | `project_id` / `item_id` / `image_id` / `segment_id` |
| 路径 | `_path` | `image_path` / `audio_path` |
| URL | `_url` | `source_url` |
| 时长 | `_duration_seconds`，单位秒 | `audio_duration_seconds` |
| 数量 | `_count` | `target_scene_count` |
| 状态 | `_status` | `task_status` |
| 布尔 | `is_ / has_ / should_ / can_ / requires_` | `requires_user_confirmation` |

不要使用 `data / event / dur / cnt / flag / info / type / status` 这类含糊字段名作为核心业务字段。

## 六、必须避免的方向

```text
1. 把生产线逻辑写进前端页面。
2. 每个页面自己写一份状态 options。
3. 每个功能文档自己定义状态取值。
4. Provider 调用散落在业务代码里。
5. 文件路径由前端直接传给 FFmpeg 使用。
6. 配置、密钥、日志混在一起。
7. 用中文文案当数据库值。
8. 用 1/2/3 这类数字枚举做协议。
9. 用 9:16 这类展示值当协议 code。
10. 把参考项目里的命名直接当成本项目协议。
```

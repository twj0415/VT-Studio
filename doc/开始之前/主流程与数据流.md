# 主流程与数据流

> 本文定义从输入到成片的数据怎么流动。页面、后端和 TODO 都要围绕这条数据流，不能各做各的。

---

## 一、MVP 主流程

```text
我的作品
→ 开始创作
→ 内容创作
→ 分镜
→ 生图
→ 视频
→ 合成
→ 导出 / 历史
```

技术流程：

```text
Project draft
→ StoryboardItem[]
→ ImageCandidate[]
→ selectedImageId
→ VideoSegment[]
→ selectedVideoSegmentId
→ CompositionTask
→ ExportRecord
```

### 1.1 配置引用链

主流程的数据是 `Project → StoryboardItem → Candidate → Segment → Composition`，但生成时还会读取三类配置。后续实现不能自由拼：

```text
视频包
→ 引用创作规则 rule_key / rule_id
→ 推荐 providerModelId 或 workflowPresetId
→ 引用素材库 Asset / AssetReference
→ 创建作品时写入 Project 当前配置
→ 创建 Task 时写入 TaskStep.input_json / snapshot_json
```

分层职责：

| 层 | 负责什么 | 不能负责什么 | 运行时怎么用 |
|---|---|---|---|
| 视频包 | 默认策略组合：内容类型、语气、结构、画幅、时长、分镜数、默认规则引用、推荐模型 / workflow、默认素材引用 | 不保存 prompt 正文、不保存 API Key、不保存 Provider 连接配置、不当作“模板创建”第二入口 | 创建作品时拷贝“引用和默认参数”到项目当前配置 |
| 创作规则 | prompt 模板、结构化输出 schema、参数 schema、校验规则、测试样例 | 不直接写 StoryboardItem、不直接调 Provider、不执行用户代码、不保存素材文件 | 任务按 rule_key / rule_id 渲染输入并校验输出 |
| Provider 连接 | vendor、base_url、auth_type、key_alias、enabled | 不保存 modelName、不保存 workflow 参数、不保存真实 key | ProviderManager 运行时用 key_alias 取密钥 |
| API 模型能力 | modelName、abilityTypes、输入/输出模态、limits、featureFlags | 不登记 ComfyUI / RunningHub workflow、不保存密钥 | ModelRegistry 校验 ImageRequest / VideoRequest |
| Workflow Preset | workflowKey / workflowId、paramSchema、nodeMap、outputMap、limits | 不混入 provider_models、不保存 key、不执行未注册 workflow | WorkflowRegistry 校验并执行已启用 preset |

落地规则：

```text
1. 业务页面只读 list_executable_media_options，不直接拼 provider_models + workflow_presets。
2. 视频包只能保存 ruleRefs、recommendedExecutableRefs、assetRefs 和默认参数。
3. 创作规则只能输出结构化建议或 prompt，不直接写库。
4. Provider / 模型 / workflow 的真实可执行性由后端 dry_run / real_generate 测试记录证明。
5. Task 记录 pack_id、rule_key / rule_id、providerModelId 或 workflowPresetId、输入摘要和脱敏输出摘要。
6. 任何日志、诊断包、导出包不得包含 API Key、完整请求头、本地绝对路径和未脱敏 Provider 原文。
```

## 二、内容创作数据流

### topic

```text
用户输入主题
→ 保存 Project.input_type = topic
→ LLM 生成短视频文案 / 分镜草稿
→ 写入 StoryboardItem.source_text / narration_text
```

参考：

```text
Pixelle generate_narrations_from_topic
```

要求：

```text
主题是发散生成。
返回数量要接近 target_scene_count。
输出必须结构化。
```

### paste

```text
用户粘贴文案
→ split_mode 切分
→ 用户预览 / 合并 / 拆分 / 删除
→ 原文写入 StoryboardItem.source_text / narration_text
```

参考：

```text
Pixelle split_narration_script
```

要求：

```text
固定文案一字不改。
不调用 LLM 改写。
必须给用户切分预览。
```

### article

```text
用户粘贴或导入文章
→ 保存 Project.input_type = article
→ LLM 浓缩提炼
→ 写入 StoryboardItem.source_text / narration_text
```

参考：

```text
Pixelle generate_narrations_from_content
```

要求：

```text
文章是浓缩提炼，不是发散创作。
长文超过阈值转存为 source_text_path。
```

## 三、分镜数据流

```text
Project + input content
→ 生成 StoryboardItem[]
→ 用户编辑
→ 用户确认
→ 进入生图
```

StoryboardItem 必须承载：

```text
source_text
narration_text
visual_description
character_ids_json / characters_json
location_id / scene_description
image_prompt
negative_prompt
video_prompt
duration_seconds
lock_flags_json
selected_image_id
selected_video_segment_id
```

规则：

```text
1. source_text / narration_text 至少一个非空。
2. visual_description 和 image_prompt 是进入生图前置。
3. duration_seconds 必须大于 0。
4. shot_size / camera_motion / transition_type 是高级字段，不阻塞主线。
5. lock_flags_json 保护用户编辑成果。
```

参考：

```text
Toonflow：导演化字段和资产引用
Pixelle：线性 StoryboardFrame
```

## 四、生图数据流

```text
StoryboardItem[]
→ 选择生图模型 / workflow
→ ImageInputPlan
→ 生成 ImageCandidate[]
→ 用户选择
→ StoryboardItem.selected_image_id
```

ImageInputPlan 必须区分：

```text
required
optional
unused
```

ImageCandidate 必须支持：

```text
多候选
预览
选择
确认
重生成追加
旧结果保留
保存为素材
```

规则：

```text
1. 远程 URL 必须转存本地。
2. 入库只存相对路径。
3. 没有 selected_image_id 不能进入普通 image_to_video。
4. 角色图、场景图、风格图、道具图等写入对应 owner，不混进分镜候选图。
```

参考：

```text
Pixelle：API / ComfyUI / RunningHub 双轨
Toonflow：多参考图和资产图收集
```

## 五、视频数据流

```text
StoryboardItem.selected_image_id
→ 读取 ImageCandidate.image_path
→ 选择视频模型 / workflow
→ VideoInputPlan
→ 生成 VideoSegment[]
→ 用户选择
→ StoryboardItem.selected_video_segment_id
```

VideoInputPlan 必须区分：

```text
required
optional
unused
```

VideoSegment 必须支持：

```text
多候选
预览
选择
确认
失败重试
旧结果保留
```

规则：

```text
1. 普通 image_to_video 必须读取 selected_image_id。
2. 不同模型的 duration / resolution / aspect_ratio 由后端能力矩阵校验。
3. 审核失败可以触发中性化改写重试。
4. 没有 selected_video_segment_id 不能进入合成。
```

参考：

```text
Pixelle：i2v、VIDEO_MODEL_CAPABILITIES、中性化重试
Toonflow：首尾帧和参考图视频语义
```

## 六、合成数据流

```text
StoryboardItem.selected_video_segment_id
→ VideoSegment.video_path
→ 按 StoryboardItem.item_index 排序
→ ffprobe 检查
→ concat / 转码兜底
→ CompositionTask.output_path
→ ExportRecord
```

规则：

```text
1. 合成顺序按 item_index，不按文件名。
2. 路径必须走 StorageService + PathGuard。
3. 片段格式不一致时转码兜底。
4. 当前主线不要求 TTS / 字幕 / BGM / 封面。
5. 合成失败要保留已生成片段。
```

参考：

```text
Pixelle VideoService
```

## 七、任务数据流

所有生成动作都要进入任务系统：

```text
Task
→ TaskStep
→ Attempt
→ Artifact
→ ProgressEvent
```

最小要求：

```text
1. 每个批量生成任务可看到总数、成功数、失败数。
2. 每个失败能定位到步骤和分镜行。
3. 每个失败能重试。
4. 取消后保留已完成产物。
5. 任务日志不泄露密钥、绝对路径、完整请求头。
```

参考：

```text
Pixelle ProgressEvent
Toonflow 任务状态思路
本项目自研持久化队列
```

## 八、后续增强数据流

### TTS

```text
StoryboardItem.narration_text
→ TTS
→ AudioCandidate / audio_path
→ audio_duration_seconds
→ 可选校准 duration_seconds
```

### 字幕

```text
narration_text / source_text
→ subtitle_chunks
→ subtitles.json / srt
→ 可选烧录进 final.mp4
```

### 封面

```text
selected_image_id / final.mp4 首帧
→ cover template
→ cover.png
→ ExportRecord
```

### BGM

```text
BGM asset
→ volume / loop / fade
→ final_with_bgm.mp4
```

这些增强都不能阻塞当前主线。

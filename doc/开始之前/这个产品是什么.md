# 这个产品是什么

> 这篇先把产品说清楚。后面的功能文档都围绕这里，不跑偏。
> `doc/` 目录记录完整需求和规范；真正开发顺序以项目 `plan/阶段路线图.md` 的 TODO 为准。

## 一、一句话定义

这是一个 **AI 短视频生成桌面工具**。

当前主线是：用户给一个主题、一段文案、一篇文章，系统把它变成：

```text
输入文字 → 分镜 → 生图 → 图生视频 → 合成后的 MP4
```

重点不是“炫技”，而是让用户稳定产出一条可编辑、可重试、风格尽量统一、最终能合成导出的短视频。

配音 TTS、字幕、封面、BGM、模板动效、数字人口播、素材成片都要保留在 TODO 和功能文档里；它们不是不做，而是优先级和所属 workflow 不同，不能阻塞当前 `image_to_video` 主线。

---

## 二、目标用户是谁

主要服务这些人：

1. 短视频创作者
2. 自媒体运营者
3. 图文视频制作者
4. 知识科普内容创作者
5. 情绪文案视频创作者
6. 小红书 / 抖音 / 视频号内容生产者
7. 有长文、笔记、小说素材，想批量转视频的人

不作为当前主线定位：

1. 专业影视团队
2. 多人协作团队
3. 复杂短剧团队
4. 漫画工作室
5. 数字人直播团队
6. 云端 SaaS 团队

这些方向可以做，也有对应功能文档，但不能污染当前主线：**先稳定跑通文字到图生视频成片**。

---

## 三、用户拿它做什么

典型场景：

| 用户输入 | 当前主线输出 | 适合内容 |
|---|---|---|
| 一个主题 | AI 拆分成分镜，再生图、图生视频、合成 | 知识科普、情绪观点、清单盘点 |
| 已写好的文案 | 原文切分为分镜行，不改字，再生图、图生视频、合成 | 口播稿、带货稿、情绪文案 |
| 一篇文章/笔记 | AI 提炼为分镜内容，再生图、图生视频、合成 | 公众号、小红书、长笔记 |
| 一本小说 | 分章、提取事件、再改编 | 后续长内容工作流 |
| 用户已有图片/视频 | 分析素材，编排成片 | 后续素材成片工作流 |

## 四、当前主 workflow

当前主 workflow：

```text
workflowType = image_to_video
```

当前主流程：

```text
01 内容导入 / 新建项目
→ 03 分镜
→ 07 生图提示词（可并入生图页行内反推）
→ 08 AI 生图
→ 14 AI 视频：图生视频 image_to_video
→ 13 视频合成
→ 21 导出与项目备份
→ 17 任务与历史
```

对应页面主线：

```text
新建项目
→ 分镜工作台
→ 生图表格页
→ 图生视频页
→ 合成页
→ 导出 / 打开输出目录
```

生图页必须是表格 / 列表工作台：

```text
每一行 = 一个分镜
列包含：序号、原文、角色、反推/出图提示词、候选图、已选图、状态、操作
```

## 六、必须做好什么

不是功能越多越好，而是当前这些事必须稳定：

1. 输入入口清楚：主题、文案、文章、小说、素材各走各的逻辑。
2. 制作流程清楚：`workflowType` 和 `inputType` 必须分开。
3. 分镜可编辑：用户能改原文片段、画面描述、角色、场景、提示词和时长。
4. 生图表格清楚：每个分镜一行，能看原文、角色、反推/出图提示词、候选图、已选图。
5. 图片不满意能重生成或替换，且候选图不能丢。
6. 图生视频使用已选图片作为输入，不能绕过图片确认。
7. 多个视频片段能按分镜顺序合成最终 MP4。
8. 任务失败能从失败步骤继续，不要从头重跑。
9. 输出文件好找，历史可回看。
10. TTS、字幕、封面等增强能力要放在 TODO 里排期，不要误删需求。

## 七、设计取舍

### 1. 链路工程借鉴 Pixelle

Pixelle 在这些方面有参考价值：

- Edge TTS
- HTML + Playwright 字幕/封面渲染
- FFmpeg 拼接、混音、合成
- ComfyUI / RunningHub / 直连 API 双轨
- TTS 时长驱动画面时长

这些是成片能力和增强能力，当前主线先用图生视频片段合成跑通；TTS、字幕、封面按 TODO 后续接入。

### 2. AI 内容组织借鉴 Toonflow

Toonflow 在这些地方更强：

- 小说分章和事件提取
- 分层 Agent：骨架、策略、剧本、导演规划
- Skill 文件化
- 画风库
- 角色/场景/道具资产库
- 多参考图保持角色一致性

这些是内容长期独立能力，当前先服务 `image_to_video` 主线。

### 3. 安全和任务系统自研

两个参考项目都有明确问题：

- Toonflow 用 vm2 执行用户 TS 供应商代码，有 RCE 风险。
- Toonflow 和 Pixelle 都把密钥明文存配置或数据库。
- Toonflow 多处 fire-and-forget，Pixelle 任务管理也是内存态，重启会丢。

所以必须自研：

- keyring 存 API Key
- PathGuard 管文件路径
- 持久化任务队列
- 断点续跑
- 幂等重试

## 八、成功标准

最低成功标准：

1. 用户 10 分钟内能从一段文字生成一条完整短视频。
2. 用户能在生成前编辑分镜。
3. 用户能查看并编辑每行生图提示词。
4. 用户能为每个分镜生成多张候选图。
5. 用户能为每个分镜选择一张最终图。
6. 用户能用已选图片生成视频片段。
7. 用户能把多个视频片段合成 final.mp4。
8. 失败后能继续生成，不需要从头来。
9. 生成记录可回看，输出文件好找。

更高标准：

1. 同一风格下生成 5 条视频，视觉统一。
2. 固定角色模式下，多张图角色不明显跑偏。
3. 生成的视频不是静态 PPT，而是有图生视频动态。
4. 后续可打开 TTS、字幕、封面、BGM、模板动效等增强能力。
5. 用户不懂 Prompt 也能完成创作。

# 最终产品形态

> 本文定义最终要做成什么样。后续 TODO 重排、页面实现、功能取舍都以这里为准。参考项目只提供优点，不作为完整蓝图。

---

## 一、一句话定义

本项目要做成一个 **桌面端 AI 短视频 / 短剧创作工作台**。

核心体验不是“堆模型入口”，而是：

```text
用户输入想法、文案或文章
→ 系统整理成分镜
→ 用户可编辑每个镜头
→ 每个镜头生成多张候选图
→ 用户选择最终图
→ 每个镜头生成视频片段
→ 用户选择最终片段
→ 合成 final.mp4
→ 导出、回看、继续优化
```

当前主线：

```text
workflowType = image_to_video
inputType = topic | paste | article
```

后续保留：

```text
novel
material_edit
digital_human
image_slideshow
TTS
字幕
封面
BGM
模板动效
画布精修
```

## 二、最终菜单

一级菜单固定为：

```text
我的作品
AI 工具
创作资源
模型 / 工作流
系统设置
```

作品内部工作台：

```text
内容创作
分镜
生图
视频
合成
导出 / 历史
```

理解方式：

```text
我的作品 = 用户要做什么
AI 工具 = 不依赖作品的小型 AI 辅助能力
作品工作台 = 一条视频怎么生产
创作资源 = 用什么规则、风格和素材生产
模型 / 工作流 = 调哪些模型和 workflow
系统设置 = 软件如何运行、安全和诊断
```

AI 工具只做独立小工具，例如爆款开头、标题生成、角色设定、多语言改写、画面描述优化。它不做“输入主题生成完整作品”，避免和 `开始创作 / 内容创作` 的主题模式重复。

| 领域 | 优先参考 | 吸收什么 | 本项目保留 / 强化 |

| 前端主流程 | waoowaoo | 线性推进、步骤清楚、用户确认后进入下一步 | Tauri 桌面工作台、Vue3、Pinia、Naive UI、Tailwind |

| UI 视觉 | 本项目为主 | 只借鉴信息结构和交互密度 | 统一桌面工具风格，不复制任何参考项目视觉 |

## 四、必须保留的本项目优势

这些设计不能因为参考项目而丢掉：

```text
1. Vue3 + Tauri 2 + Rust + SQLite 的桌面架构。
2. 前端 camelCase、Rust/SQLite/JSON snake_case、Tauri 边界 serde 映射。
3. ProviderManager 统一模型调用，不让页面直接调模型。
4. provider_models 和 workflow_presets 分离。
5. ImageInputPlan / VideoInputPlan 动态输入规划。
6. keyring 保存真实密钥，SQLite 只保存 key_alias。
7. StorageService + PathGuard 控制文件路径。
8. Task / TaskStep / ProgressEvent 贯穿生成链路。
9. ImageCandidate[] / VideoSegment[] 多候选机制。
10. lockFlagsJson 保护用户编辑成果。
11. Mock 和真实生成明确区分。
12. TTS、字幕、封面、BGM、素材成片、数字人都保留为后续能力，不从产品中删掉。
```

## 五、最终体验标准

最终体验必须统一：

```text
生成前可控
生成中可见
生成后可选
失败可恢复
结果可导出
规则可编辑
模型可替换
素材可复用
```

不要做成：

```text
1. ComfyUI 工作流管理器。
2. Prompt 玩具。
3. 传统项目管理软件。
4. 复杂短剧 SaaS。
5. 数字人直播平台。
6. 只会调用模型、没有编辑和选择过程的自动生成器。
7. UI 像 A、流程像 B、设置像 C 的拼接产品。
```

## 六、第一版成品验收

第一版必须形成完整闭环：

```text
1. 用户打开应用看到“我的作品”。
2. 点击“开始创作”进入内容创作。
3. 输入主题 / 文案 / 文章，设置视频包、画幅、时长、分镜数。
4. 生成或确认分镜。
5. 修改分镜、提示词、时长。
6. 进入生图页，按分镜行生成多张候选图。
7. 每行选择最终图。
8. 进入视频页，用已选图生成视频片段。
9. 每行选择最终片段。
10. 进入合成页，检查片段并生成 final.mp4。
11. 打开输出目录或导出视频。
12. 失败时能看到是哪一步、哪一行、怎么恢复。
```

不要求第一版完成：

```text
完整 SkillSnapshot / hash / 规则版本复现
字幕烧录
BGM 混音
封面模板
小说长内容
数字人口播
素材成片
画布精修
```

但这些能力必须继续存在于文档和 TODO 中。

## 七、后续 TODO 重排依据

重排 TODO 时按这个顺序：

```text
1. 先保主线闭环。
2. 再保真实生成前置：Provider、模型、workflow、keyring、PathGuard。
3. 再保可靠性：任务、重试、历史、导出。
4. 再保一致性：画风、角色、场景、素材库、创作规则。
5. 再接增强：TTS、字幕、封面、BGM、模板。
6. 最后做独立 workflow：素材成片、数字人、小说长内容、画布精修。
```

任何 TODO 如果让当前主线被后续能力阻塞，都要降级或后移。

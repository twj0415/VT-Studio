# 内容导入

> 这是给 AI 开发用的功能说明书。读完这一篇，就能把"内容导入"这个模块从零写出来，不用再问。
> 配套阅读：字段定义看 `底层设计/数据结构.md`；模型调用看 `底层设计/Provider与安全.md`。

## 一、这个功能是什么

让用户把"想做成视频的素材"喂进系统，是整个流水线的第一步。用户在这里给一段内容，系统把它整理成一个标准的「输入对象」。当前 `image_to_video` 主线下一步直接进入「分镜」；「脚本生成」是后续文本预处理/高级编辑能力，不是必经独立页面。

支持 4 种入口，覆盖从"我只有一个想法"到"我有一整本小说"的所有情况：

1. **主题输入** —— 只有一句话的想法，让 AI 帮你写内容。例：输入"为什么要早睡"。
2. **文案粘贴** —— 已经写好了文案，直接用，不让 AI 改。
3. **文章/笔记导入** —— 一篇长文章/公众号/笔记，让 AI 浓缩成视频内容。
4. **小说导入** —— 一整本长篇小说，先分章、再逐章提取剧情事件（做长剧用）。

## 二、两种处理模式（这是核心，先理解再看子功能）

不管哪个入口，内容进来后只有两条路，开发时必须想清楚走哪条：

| input_process_mode | 什么意思 | 谁走这条 | 下一步 |
|------|---------|---------|--------|
| **AI 生成模式**（generate） | 把素材交给 AI，AI 重新组织成 N 个分镜输入段 | 主题、文章/笔记 | 进入分镜生成；可选经过脚本生成 |
| **固定文案模式**（fixed） | 用户的文字一字不改，只做机械切分 | 文案粘贴 | 直接切分成分镜原文段，跳过 AI 改写 |

> 参考证据：Pixelle 有「generate / fixed」双模式。源码 `web/components/content_input.py` 的 `render_content_input()`，用 `st.radio(["generate","fixed"])` 切换。固定模式按 `split_mode`（段落/行/句）切分，见 `pixelle_video/utils/content_generators.py` 的 `split_narration_script()`。
> 我们的改动：Pixelle 是把模式做成一个单选按钮；我们按"入口"自动决定模式（选主题就是 generate，选粘贴文案就是 fixed），用户不用懂这个概念，更顺手。

### 参考项目补充机制

waoowaoo 的内容导入强在“先处理输入风险，再进入生产”。它不是让长文本直接生成分镜，而是先用 `LongTextDetectionPrompt` 做长文本提示，用 `SmartImportWizard` 把长内容导入拆成来源、解析、映射、确认几步。这个机制要吸收，但不能照搬成当前主线的独立脚本编辑页。

本项目落地规则：

```text
1. 内容创作页是输入确认页，不是完整分镜编辑页。
2. topic / article 可以 AI 整理，paste 只能机械切分。
3. 长文本必须提示风险，并建议切分方式。
4. 切分预览必须可逐条编辑、合并、拆分、删除、移动和恢复原文。
5. 用户确认后才写入 StoryboardItem.sourceText / narrationText。
6. AI 整理失败时，用户仍可手动确认段落进入分镜。
7. 不新增 script-editor 必经页；高级脚本编辑只能作为后续可选增强。
```

### 页面施工规格

```text
页面区域：
  左侧输入来源 Tabs
  中间输入区 / AI 结果区 / 切分预览区
  右侧作品设置：视频包、画幅、目标时长、分镜数、内容语言、当前创作规则
  底部固定操作条：保存草稿、创建并进入分镜

必须按钮：
  AI 生成 / 整理
  切分预览
  编辑段落
  合并段落
  拆分段落
  删除段落
  移动段落
  恢复原文
  保存草稿
  创建并进入分镜

必须状态：
  empty_input
  too_long
  needs_split_preview
  splitting
  ai_generating
  ai_failed
  ready_to_confirm
  confirmed
  draft_saved
```

## 三、子功能清单

1. 主题输入（generate 模式）
2. 文案粘贴（fixed 模式）

## 子功能 1：主题输入

### 界面
- 一个多行输入框，占位提示"输入你的视频主题，例如：为什么要早睡"
- 「视频类型」下拉：知识科普 / 故事讲述 / 情感口播 / 带货种草 …（影响后面脚本的语气）
- 「目标时长」选择：15秒 / 30秒 / 60秒 / 自定义
- 「画面段数」滑块：3 ~ 30 段，默认 5（决定分成几个镜头）
- 「内容语言」下拉：中文 / 英文，默认中文；字段使用 `content_language`，取值看 `底层设计/01-枚举字典与配置规范.md`
- 一个「创建并进入分镜」按钮

### 用户操作
输入"为什么要早睡" → 选「知识科普」→ 选 60 秒 → 画面段数拉到 8 → 点「下一步」。

### 系统逻辑（一步步）
1. **校验**：主题非空、长度 2~50 字。空或超长 → 红字提示，不放行。
2. **打包**成「归一化输入对象」（字段见下方），`input_type` 固定为 `"topic"`，`input_process_mode` 固定为 `"generate"`，并写入 Project。
3. **落库**：写入 `project` 表（一个输入 = 一个项目），项目生命周期置为 `draft`，同时创建默认 Bible。
4. **草稿保存**：项目创建后即为可恢复草稿，后续表单修改自动保存到 SQLite，不依赖前端内存。
5. **跳转**：当前主线带着这个对象进入「分镜」，逻辑见 `功能模块/03-分镜.md`；如启用脚本增强，再参考 `功能模块/02-脚本生成.md`。

### 数据结构
```json
{
  "input_type": "topic",
  "input_process_mode": "generate",
  "input_options_json": {},
  "topic": "为什么要早睡",
  "content_category": "knowledge",
  "target_duration_seconds": 60,
  "target_scene_count": 8,
  "content_language": "zh-CN"
}
```
> 完整字段定义和取值范围以 `底层设计/数据结构.md` 为准，这里只是示意。

### 参考证据 / 本项目落地
- Pixelle `pixelle_video/utils/content_generators.py` → `generate_narrations_from_topic(llm_service, topic, target_scene_count, min_words, max_words)`。它就是拿主题 + 段数，调 LLM 返回 N 段解说词。
- 提示词模板：Pixelle `pixelle_video/prompts/topic_narration.py`（`build_topic_narration_prompt`）。
- 本项目只借鉴“主题 + 目标段数 → 结构化段落”的能力，提示词和 DTO 必须使用本项目 PromptSkill / schema，不复制参考项目提示词正文。

### 坑提醒
- Pixelle 对段数有硬校验：AI 返回数量 > 期望就截断、< 期望直接抛错（见 `generate_narrations_from_topic` 末尾）。我们要保留这个校验，但**不能直接报错给用户**，要做一次自动重试，重试还不够再提示用户调小段数。
- `target_scene_count` 别让用户填太大，Pixelle 上限 30。段数越多，后面生图/配音任务越多、越慢。

## 子功能 2：文案粘贴（固定文案模式）

### 界面
- 一个大的多行输入框（比主题框高，占位"把你写好的文案粘到这里，每段一个画面"）
- 「切分方式」下拉：按段落（空行分）/ 按行 / 按句号。默认按段落。
- 「内容语言」下拉
- 「下一步」按钮
- 不显示「画面段数」滑块（段数由切分结果决定，不是用户定的）

### 用户操作
粘贴一段写好的文案（中间用空行分段）→ 选「按段落」→ 点「下一步」。

### 系统逻辑（一步步）
1. **校验**：文案非空。
2. **切分**：按所选 `split_mode` 把文案切成若干段，每段对应一个画面/镜头：
   - `paragraph`：按双换行 `\n\n` 切，段内单换行保留。
   - `line`：按单换行切，每行一段。
   - `sentence`：按句末标点 `。.!?！？` 切。
3. **清洗**：去掉切分后的空段、首尾空白。
4. **打包**：`input_process_mode="fixed"`，`split_mode` 写入 `input_options_json`，把切好的段落数组作为后续脚本步骤输入，**不调 AI**。
5. **落库 + 跳转**：当前 `image_to_video` 主线不把 `script_generation` 作为 fixed 模式必经步骤。用户确认切分预览后，直接按顺序写入 `StoryboardItem.source_text / narration_text`，进入分镜页。只有显式启用“脚本增强 / 高级脚本编辑”时，才创建 `script_generation / script_review`。

### 数据结构
```json
{
  "input_type": "paste",
  "input_process_mode": "fixed",
  "input_options_json": {
    "split_mode": "paragraph"
  },
  "content_language": "zh-CN",
  "narrations": ["第一段文案", "第二段文案", "第三段文案"]
}
```

### 参考证据 / 本项目落地
- Pixelle `pixelle_video/utils/content_generators.py` → `split_narration_script(script, split_mode)`。可参考它的三种切分模式：段落、行、句。
- 本项目应自己实现切分规则，并在切分后提供可编辑预览、合并、拆分、删除和确认，不复制参考项目代码。
- waoowaoo 的 `LongTextDetectionPrompt` 是明确拦截式提示，`SmartImportWizard` 是来源、解析、映射、确认几步。落到本项目时，不能只做一个黄色提示条，必须有用户可选择的处理动作和可回退的切分预览。

### 切分预览必须落到这些细节

```text
列表行字段：
  index
  original_text
  edited_text
  char_count
  suggested_duration_seconds
  validation_issues
  locked

行内按钮：
  编辑
  合并上一段
  合并下一段
  拆分
  删除
  上移
  下移
  恢复原文
  锁定 / 解锁

确认写回：
  确认前只存在 Project 草稿和 split_preview 临时状态。
  确认后才创建或更新 StoryboardItem。
  paste 模式写入 source_text / narration_text 时不得改写用户原文。
  topic / article 模式的 AI 结果也必须先进入切分预览，再由用户确认。
```

### 坑提醒
- 固定模式下用户最在意"一字不改"。切分逻辑绝对不能顺手做"AI 优化/改写"，否则违背预期。
- 切分后给用户一个可编辑的段落预览（每段可改、可合并、可删），确认后再进下一步。这一点 Pixelle 没做（它切完直接用），是我们要补的体验亮点。

## 子功能 3：文章/笔记导入

### 界面
- 「上传文件」按钮（支持 txt / md）+ 一个粘贴框，二选一
- 「内容类别」下拉、「目标时长」、「画面段数」滑块、「内容语言」（同主题输入）
- 「创建并进入分镜」按钮

### 用户操作
粘贴一篇 2000 字的公众号文章 → 选「知识科普」→ 段数 10 → 点「下一步」。

### 系统逻辑（一步步）
1. **校验**：内容非空；超长（比如 > 1 万字）给提示，建议改用「小说导入」。
2. **打包**：`input_type="article"`，`input_process_mode="generate"`。
3. **落库**：20KB 以内原文存入 `projects.source_text`；超过 20KB 写入项目 `input/source.txt`，库里存 `source_text_path`，可在 `source_text` 存摘要或留空。
4. **跳转**：当前主线进入「分镜」，分镜模块使用“从已有内容提炼分镜”的提示词；如启用脚本增强，再调脚本生成，但用的是“从已有内容浓缩”的提示词，不是“从主题发散”。

### 数据结构
```json
{
  "input_type": "article",
  "input_process_mode": "generate",
  "input_options_json": {},
  "source_text": "20KB 以内文章原文或长文摘要",
  "source_text_path": null,
  "content_category": "knowledge",
  "target_duration_seconds": 60,
  "target_scene_count": 10,
  "content_language": "zh-CN"
}
```

### 参考证据 / 本项目落地
- Pixelle `content_generators.py` → `generate_narrations_from_content(llm_service, content, target_scene_count, ...)`。注意它和主题版（`..._from_topic`）是**两个不同函数、两套提示词**：主题是"发散补充"，内容是"浓缩提炼"。
- 提示词模板：Pixelle `pixelle_video/prompts/content_narration.py`。
- 本项目只吸收“主题生成”和“文章浓缩”必须分开建模的规则，具体提示词由本项目创作规则生成并做 schema 校验。

### 坑提醒
- 主题和文章一定要用不同提示词。用错了会出现"拿一篇完整文章却让 AI 瞎发挥"或"拿一个短主题却让 AI 硬浓缩"，效果都很差。开发时按 `input_type` 选对函数。

## 子功能 4：小说导入（长内容，单独一套流程）

> 这是 4 个入口里最复杂的，做长篇剧情类视频用。它不是"切几段"，而是「分章 → 逐章提取剧情事件 → 事件再去生成脚本」。Toonflow 的小说流程可作为能力证据，但本项目必须按自己的任务队列、结构化事件和数据表落地。

### 界面
- 上传 txt 或粘贴长文
- 「AI 智能分章」按钮
- 分章结果列表：每章一行（卷名、章名、字数），章节边界可手动拖动调整
- 每章一个「事件提取状态」小标（等待中 / 提取中 / 成功 / 失败）
- 「全部提取事件」按钮 + 单章「重新提取」按钮

### 用户操作
1. 粘贴整本小说 → 点「AI 智能分章」→ 系统切出 50 章 → 用户检查、微调几个章节边界。
2. 点「全部提取事件」→ 系统逐章跑、列表实时更新每章状态。
3. 全部成功后 → 进入小说/长内容脚本生成或分镜生成。

### 系统逻辑（一步步）

**第一步：AI 智能分章**
1. 用户点「AI 智能分章」，系统把原文前 2000 字发给 LLM，让它**返回一个正则表达式**（不是直接切，是让 AI 找出"第X章 标题"的分隔规律）。
2. 整本小说原文写入 `source_text_path` 指向的项目输入文件；再用返回的正则在原文上 `match`，切成章节数组，每章带 `chapter_index / volume_title / chapter_title / chapter_content`。
3. 正则为空（没识别出章节模式）→ 提示用户手动分章或按固定字数切。
4. 切好的章节写入 `novel_chapters` 表，每条 `event_status="pending"`。

**第二步：逐章提取事件**
1. 用户点「全部提取事件」，系统**并发**逐章调 LLM（控制并发数，默认 5），每章让 AI 输出"剧情事件摘要"。
2. 每章完成就更新该章状态：成功 `event_status=succeeded`、失败 `event_status=failed` 并存 `error_reason`。
3. 用事件流（如 EventEmitter）边跑边推进度，前端列表实时刷新。
4. 失败的章可单独「重新提取」。

**第三步：进入长内容后续处理**
全部章节事件就绪后，可以把事件序列交给 `script_generation` TaskStep 生成剧情脚本，也可以由后续长内容 workflow 直接生成分镜。小说导入属于后续扩展，不阻塞当前 image_to_video 主线。

### 数据结构（NovelChapter，以本项目字段为准）
```json
{
  "novel_chapter_id": "chapter_xxx",
  "project_id": "proj_xxx",
  "chapter_index": 1,
  "volume_title": "第一卷",
  "chapter_title": "第一章 开端",
  "chapter_content": "本章正文……",
  "structured_event": {
    "character_ids": [],
    "location_id": null,
    "actions": [],
    "conflict": ""
  },
  "event_status": "succeeded",
  "error_reason": null,
  "created_at": 1718000000000
}
```

### 参考证据 / 本项目落地
- 分章正则：Toonflow `src/routes/script/getAiRegex.ts`。它的思路是让 AI 从样本文本里识别章节标题规律，并返回可执行的章节匹配规则。
- 新增章节 + 触发事件提取：Toonflow `src/routes/novel/addNovel.ts`。我们只借鉴它的流程，落库使用本项目 `novel_chapters`、`event_status="pending"`，并纳入 Task。
- 逐章并发提取：Toonflow `src/utils/cleanNovel.ts`（`CleanNovel` 类，`concurrency=5`，`processChapter` 调 LLM、emit `item` 事件、失败存 `error_reason`）。
- 本项目不能复用 Toonflow 的提示词正文；需要重新写成本项目 PromptSkill，并要求输出结构化章节规则和结构化事件。

### 坑提醒（Toonflow 的毛病，我们必须改）
- **事件是纯文本空壳**：Toonflow 的 `processChapter` 把 AI 返回的文本原样塞进 `event` 字段（一个字符串），没有结构。后面分镜/角色提取拿不到结构化数据，全靠再解析。**我们改成结构化事件**（人物、地点、动作、冲突分字段），字段定义见 `底层设计/数据结构.md`。
- **fire-and-forget 没有续跑**：`novelClass.start()` 是发完就不管，进程一关、任务一断就丢，无法续跑/重试整批。**我们要把"全部提取事件"做成可恢复的任务**，纳入统一任务系统，见 `功能模块/17-任务与历史.md`。
- **并发无限流风险**：并发数写死 5，没有针对 LLM 限流的退避。我们要做指数退避重试，见 `底层设计/Provider与安全.md`。

## 四、四个入口汇总对照

| 入口 | input_type | 模式 | 调 AI? | 当前主线下一步 |
|------|-----------|------|--------|---------------|
| 主题输入 | topic | generate | 是（发散） | 03-分镜；脚本生成仅作为可选增强 |
| 文案粘贴 | paste | fixed | 否（只切分，不调 LLM） | 03-分镜；保留原文分段 |
| 文章/笔记 | article | generate | 是（浓缩） | 03-分镜；脚本生成仅作为可选增强 |
| 小说导入 | novel | generate | 是（分章+事件+脚本） | 长内容 workflow；不进入当前 image_to_video 主线 |

## 五、这个模块对外提供什么（给开发的接口约定）
- 输出统一是「归一化输入对象」，下游只认这个对象，不关心用户从哪个入口来。
- 落库后一定有一个 `project` 记录，`project_lifecycle=draft`；后续所有生成动作围绕这个 project 创建 Task 推进。

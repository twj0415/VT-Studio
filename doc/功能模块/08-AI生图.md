# AI 生图

> 这是给 AI 开发用的功能说明书。读完这一篇，就能把“AI 生图”模块从零写出来，不用再问。
> 配套阅读：Provider 与安全看 `底层设计/Provider与安全.md`；模型能力和 workflow preset 看 `底层设计/12-Provider接口契约与能力矩阵Schema.md`、`底层设计/23-模型适配与工作流注册规范.md`；提示词来源看 `功能模块/07-生图提示词.md`。

## 一、这个功能是什么

AI 生图模块负责读取每条 `StoryboardItem` 的 `imagePrompt / negativePrompt`，调用图片模型生成多张候选图，并让用户为每个分镜选择一张最终图。

当前主线中的位置：

```text
输入文字 → 分镜 → 生图提示词 → AI 生图 → 选择最终图 → 图生视频 → 合成
```

核心产物不是单个图片路径字段，而是：

```text
ImageCandidate[]
StoryboardItem.selectedImageId
```

核心原则：

```text
业务代码不直接调某家 API
一律走 ImageProvider
模型候选项只来自 list_executable_media_options
API 模型用 providerModelId，ComfyUI/RunningHub 用 workflowPresetId
远程 URL 必须下载转存本地
入库只存相对路径
候选图必须可回看、可选择、可重生成
```

---

## 二、生图页页面设计

生图页必须是表格 / 列表工作台，不以中间大预览为核心。

```text
每一行 = 一个分镜
```

建议列：

```text
序号
原文
角色
反推 / 出图提示词
生成图片
候选图
已选图
状态
操作
```

顶部操作：

```text
批量生成全部
只生成未生成
只生成失败项
进入视频阶段
```

行内操作：

```text
编辑提示词
生成图片
重新生成
选择候选图
确认图片
查看大图
```

大图预览可以做成弹窗 / 抽屉 / hover 预览，但不是页面主布局。

---

## 三、子功能清单

1. 按分镜行批量生成候选图
2. 单行重生成
3. 候选图选择与确认
4. 参考图生图（角色/场景一致性）
5. 云 API 生图
6. ComfyUI/RunningHub 生图（workflow preset）
7. 失败重试与状态回写

---

## 四、统一调用规则

AI 生图模块只负责收集业务输入，不负责判断具体模型或拼供应商参数。

```text
前端选择器
→ list_executable_media_options
→ 用户选择 provider_model 或 workflow_preset
→ 后端组装 ImageRequest
→ ProviderManager
→ ImageProvider 或 WorkflowRegistry
```

硬规则：

```text
1. 功能模块不得按 modelName / workflowKey 写分支。
2. 功能模块不得直接拼 ComfyUI workflow JSON 或 RunningHub 参数。
3. providerModelId 和 workflowPresetId 不能同时为空，默认不能同时有值。
4. workflow 内部要绑定 API 模型时，只能由 workflow preset 声明。
5. 单独测试 workflow 时可用 WorkflowRequest，生产线生图默认走 ImageRequest。
```

## 子功能 1：按分镜行批量生成候选图

### 界面

每一行显示：

```text
StoryboardItem.index
sourceText / narrationText
characters
imagePrompt / negativePrompt
生成按钮
候选图列表
已选图
状态
```

### 系统逻辑

1. 读取 `StoryboardItem[]`，筛选需要生成图片的行。
2. 校验每行 `imagePrompt` 非空。
3. 从 `list_executable_media_options` 读取可用生图候选项。
4. 根据用户选择写入 `providerModelId` 或 `workflowPresetId`。
5. 对每条 StoryboardItem 组装 `ImageRequest`。
6. 每行默认生成 4 张候选图，数量可配置。
7. Provider 返回本地文件或远程 URL。
8. 如果是远程 URL，下载到受控任务目录。
9. 为每张图创建 `ImageCandidate`。
10. 行状态变为 `waiting_select` 或 `image_generated`。
11. 失败则标记失败，记录 `lastErrorJson` 和 `retryCount`。

### 数据结构

```json
{
  "imageId": "img_001",
  "itemId": "item_001",
  "imagePath": "images/item_001/img_001.png",
  "prompt": "young office worker waking up refreshed...",
  "negativePrompt": "low quality, distorted face",
  "model": "provider_model_xxx",
  "status": "succeeded",
  "selected": false,
  "createdAt": 1710000000000
}
```

### 参考项目取舍

- 借鉴 Pixelle `pixelle_video/services/media.py::MediaService` 的 ComfyUI 工作流媒体生成。
- 借鉴 Pixelle `pixelle_video/services/api_media.py::APIProviderMediaService` 的 API 生图封装。
- 借鉴 Toonflow `src/routes/production/storyboard/batchGenerateImage.ts` 的分镜批量生成图和资产参考图收集。
- 不照搬 Pixelle 的明文 key 配置和硬编码模型选择。
- 不照搬 Toonflow 的前端直连模型细节。

### 坑提醒

- 不要把远程 URL 直接存库。
- 不要无限并发打图片 API，要走 Provider 限流。
- 不要在内存里长期保存大 base64。
- 不要在批量生图逻辑里判断具体模型名。

## 子功能 2：单行重生成

### 界面

- 每行有「重新生成」按钮。
- 可选：使用原 prompt / 先改 prompt 再生成。
- 旧候选图保留，新的候选图追加或形成新版本。

### 系统逻辑

1. 用户选择某行 StoryboardItem。
2. 保留旧候选图，不立即删除。
3. 用当前 `imagePrompt / negativePrompt` 重新调用 ImageProvider。
4. 成功后新增 ImageCandidate。
5. 如果该行已有 selectedImageId，重生成后不自动覆盖，需要用户重新选择。

### 坑提醒

- 不要重生成失败后把旧图覆盖没了。
- 不要静默替换用户已经选中的图。

## 子功能 3：候选图选择与确认

### 界面

每行候选图区域显示：

```text
图 1 / 图 2 / 图 3 / 图 4
```

用户点击某张图：

```text
该图标记 selected
该行写 selectedImageId
该行状态变为 selected / confirmed
```

### 系统逻辑

1. 用户点击候选图。
2. 将同一 item 下其他 ImageCandidate 的 selected 置为 false。
3. 当前 ImageCandidate 的 selected 置为 true。
4. 写入 `StoryboardItem.selectedImageId`。
5. 所有分镜都存在 selectedImageId 后，允许进入图生视频阶段。

### 进入下一阶段条件

```text
所有 StoryboardItem 都必须有 selectedImageId
```

如果没有全部选择，提示：

```text
还有 N 个分镜未选择图片，请先完成选择。
```

### 坑提醒

- 不要允许没有已选图时进入图生视频。
- 用户替换本地图片也要复制进项目目录，不引用外部绝对路径。

## 子功能 4：参考图生图

### 系统逻辑

1. 读取 StoryboardItem.characters 或 characterIds。
2. 找到 Character Bible 的 `referenceImagePath`。
3. 读取 StoryboardItem.sceneDescription 或 locationId。
4. 找到 Environment Bible 的 `referenceImagePath`。
5. 组合为 `ImageRequest.referenceImages`，由 ProviderAdapter 转成具体供应商的参数。
6. 模型不支持参考图时，降级为纯文本 prompt。

### 参考项目取舍

- 借鉴 Toonflow `src/utils/ai.ts` 的 `referenceList` 思路。
- 借鉴 Toonflow `src/routes/production/storyboard/batchGenerateImage.ts::getAssetsImageBase64` 的资产图收集方式。
- 借鉴 Toonflow `src/routes/assetsGenerate/generateAssets.ts` 的资产生图参考图输入。
- 不照搬 Toonflow 的供应商私有字段到业务层。

### 坑提醒

- 多参考图要控制数量，太多模型会混乱。
- 角色参考图和场景参考图要在 prompt 中明确用途。

## 子功能 5：云 API 生图

### 系统逻辑

1. ProviderManager 根据 `providerModelId` 从 ModelRegistry 解析 DashScope/OpenAI/其他兼容图片服务。
2. 传入 prompt、negativePrompt、尺寸、seed、参考图、候选数量。
3. Provider 内部处理不同 API 参数差异。
4. 结果统一成 `{ imageUrl 或 imagePath }[]`。
5. 后端统一转存为 ImageCandidate。

### 参考项目取舍

- 借鉴 Pixelle `pixelle_video/services/api_media.py::APIProviderMediaService.IMAGE_MODELS` 的模型能力声明思路。
- 借鉴 Pixelle `pixelle_video/services/api_services/image_dashscope.py` / `image_gpt.py` / `image_seedream.py` 的供应商适配器拆分方式。
- 不照搬硬编码模型列表，模型能力必须进入 `provider_models`。

### 坑提醒

- 不要在业务层写 DashScope/OpenAI 参数分支，要封装在 Provider。
- 不要把 `model_name` 当业务判断条件。

---

## 子功能 6：ComfyUI/RunningHub 生图

### 系统逻辑

1. 从 `list_executable_media_options` 选择 `sourceType=workflow_preset` 的候选项。
2. 后端根据 `workflowPresetId` 从 WorkflowRegistry 读取 workflow preset。
3. 按 preset 的 `param_schema` 校验 prompt、尺寸、参考图、输出路径。
4. 按 preset 的 `node_map` 写入 ComfyUI workflow 或 RunningHub 任务参数。
5. 按 preset 的 `output_map` 下载或复制结果到任务目录。
6. 为输出创建 ImageCandidate。

### 参考项目取舍

- 借鉴 Pixelle `pixelle_video/services/comfy_base_service.py::ComfyBaseService._scan_workflows` 的 workflow 扫描。
- 借鉴 Pixelle `pixelle_video/services/comfy_base_service.py::ComfyBaseService._resolve_workflow` 的 workflow 解析。
- 借鉴 Pixelle `pixelle_video/services/media.py::MediaService` 的执行流程。
- 不照搬“业务层直接读取 workflow key”的方式。

### 坑提醒

- ComfyUI 走同一 ImageProvider/工作流接口，未配置时不出现在可执行任务里，避免半成品混进主流程。
- 未注册到 `workflow_presets` 的 workflow 一律不能执行。

---

## 子功能 7：失败重试与状态回写

### 系统逻辑

1. 每行生成状态独立维护。
2. 单行失败不影响其他行完成。
3. 失败行可重试。
4. 批量重试只处理失败行或未生成行。
5. 错误写入 TaskStep / ImageCandidate / StoryboardItem 的错误字段。

### 状态建议

```text
not_generated
running
failed
waiting_select
selected
confirmed
```

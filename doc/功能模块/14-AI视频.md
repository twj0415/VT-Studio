# AI 视频

> 这是给 AI 开发用的功能说明书。读完这一篇，就能把“AI 视频”模块从零写出来，不用再问。
> 配套阅读：Provider 与安全看 `底层设计/Provider与安全.md`；视频能力和 workflow preset 看 `底层设计/12-Provider接口契约与能力矩阵Schema.md`、`底层设计/23-模型适配与工作流注册规范.md`。

## 一、这个功能是什么

AI 视频模块负责把图片、文本、首尾帧、参考素材生成视频片段。

当前主线只做：

```text
image_to_video：用生图阶段已选图片生成视频片段
```

其他能力是后续扩展：

```text
text_to_video
start_end_frame_i2v
reference_to_video
video_continuation
```

当前主线位置：

```text
输入文字 → 分镜 → 生图 → 选择最终图 → 图生视频 → 合成
```

核心原则：

```text
业务代码不直接调某家视频 API
模型候选项只来自 list_executable_media_options
API 视频模型用 providerModelId，ComfyUI/RunningHub 用 workflowPresetId
统一提交 VideoRequest，不按具体 modelName 写分支
远程 URL 必须下载转存本地
生成视频前必须先做 VideoInputPlan
不同视频模型 / workflow 需要的资源不同，不能固定写死
```

### 参考项目补充机制

Pixelle 的价值在于 `image_to_video` 链路和视频能力矩阵：不同模型的时长、分辨率、比例和输入图片要求不同，必须在后端校验。

Toonflow-web 的价值在于视频测试和结果选择思路：

```text
VideoModelTest 把输入模式拆成 text、singleImage、startEndRequired、endFrameOptional、startFrameOptional、multiRef。
video store 用 selectedResultId 表达用户最终选择的生成结果。
workbench / storyboard 生产区强调先有图片或参考素材，再进入视频生成。
```

本项目吸收为：

```text
1. 所有视频模型差异收敛到 VideoInputPlan。
2. 当前主线普通 image_to_video 只需要 selectedImageId + videoPrompt + durationSeconds。
3. 首尾帧、多参考、文生视频都是后续模式，不能污染当前必经主线。
4. 每条分镜保留 VideoSegment[]，用户确认后写 selectedVideoSegmentId。
5. 重生成只追加新片段，不自动覆盖 selectedVideoSegmentId。
6. 清理未选片段只能清理未选、未引用、可再生的片段，必须二次确认。
```

## 三、统一调用规则

AI 视频模块只负责选择输入素材和生成参数，不负责拼供应商私有参数。

```text
前端选择器
→ list_executable_media_options
→ 用户选择 provider_model 或 workflow_preset
→ 后端校验 abilityTypes / inputModalities / limits
→ 后端组装 VideoRequest
→ ProviderManager
→ VideoProvider 或 WorkflowRegistry
```

硬规则：

```text
1. 功能模块不得直接合并 provider_models 和 workflow_presets。
2. 功能模块不得按 modelName / workflowKey 写分支。
3. durationSeconds / resolution / fps / inputImages 必须由后端按能力矩阵或 preset limits 校验。
4. providerModelId 和 workflowPresetId 不能同时为空，默认不能同时有值。
5. ComfyUI / RunningHub 视频生成只能执行已注册 workflow preset。
```

### 3.1 视频资源规划

生成视频前必须先判断当前视频模型或 workflow 需要什么资源。

后端返回 VideoInputPlan：

```text
required：必须补齐，缺少就不能生成。
optional：可选增强，用户决定是否补。
unused：当前模型不需要，不展示为缺项，也不自动生成。
```

典型差异：

```text
普通图生视频：
required = selectedImage / videoPrompt / durationSeconds
optional = seed / motionStrength
unused = endFrame / characterReference / pose / depth / mask

首尾帧视频：
required = startFrame / endFrame / videoPrompt
optional = durationSeconds / fps
unused = character_expression_sheet / scene_detail_view

参考图视频：
required = startFrame / videoPrompt
optional = characterReference / styleReference
unused = mask，除非 workflow preset 声明需要

ComfyUI / RunningHub 视频 workflow：
required / optional / unused 由 workflow preset.param_schema 和 requiredAssets 决定
```

规则：

```text
1. 普通图生视频只需要已选分镜图时，不生成尾帧、pose、depth、mask。
2. 首尾帧模型需要尾帧时，再提示生成或选择尾帧图。
3. 参考图视频需要角色或风格参考时，从 Character Bible / Style Bible 读取或提示补齐。
4. ComfyUI / RunningHub 需要什么，必须由 workflow preset 声明，不能靠前端猜。
5. 后端必须再次校验 VideoInputPlan，不能只靠前端禁用。
```

## 子功能 1：图生视频 image_to_video

### 界面

图生视频页建议也采用表格 / 列表工作台：

```text
每一行 = 一个分镜
```

建议列：

```text
序号
已选图片
视频动作提示词
时长
模型输入要求
生成视频
视频候选
已选视频
状态
操作
```

顶部操作：

```text
批量生成全部
只生成未生成
只生成失败项
批量重试
进入合成
```

行内操作：

```text
编辑视频提示词
单行生成
重新生成
预览视频
选择片段
确认片段
查看输入要求
查看审核失败改写
清理未选片段
```

必须状态：

```text
not_generated
input_missing
running
failed
waiting_select
selected
confirmed
cancelled
```

### 系统逻辑

1. 读取 `StoryboardItem[]`。
2. 校验每条 item 都有 `selectedImageId`。
3. 通过 selectedImageId 找到 ImageCandidate 的 `imagePath`。
4. 读取 `videoPrompt`。
5. 从 `list_executable_media_options` 选择支持 `image_to_video` 的候选项。
6. 后端根据 `providerModelId` 或 `workflowPresetId` 检查能力和 limits。
7. 组装 `VideoRequest` 并交给 ProviderManager。
8. 下载/转存结果到受控工作区，例如 `videos/item_001/segment_001.mp4`。
9. 创建 `VideoSegment`。
10. 用户确认片段后写入 `StoryboardItem.selectedVideoSegmentId`。
11. 如果该行已有 selectedVideoSegmentId，新片段生成成功后保持旧选择，直到用户显式改选。
12. 所有分镜都有 selectedVideoSegmentId 后，才允许进入合成。

### 数据结构

```json
{
  "segmentId": "seg_001",
  "itemId": "item_001",
  "inputImageId": "img_002",
  "videoPath": "videos/item_001/seg_001.mp4",
  "videoPrompt": "soft morning light moves across the room, character slowly opens eyes and sits up",
  "durationSeconds": 4,
  "model": "provider_model_xxx",
  "status": "succeeded",
  "selected": true,
  "createdAt": 1710000000000
}
```

### 参考项目取舍

- 借鉴 Pixelle `web/pipelines/i2v.py::ImageToVideoPipelineUI` 的图生视频交互。
- 借鉴 Pixelle `pixelle_video/services/api_media.py::APIProviderMediaService.VIDEO_MODEL_CAPABILITIES` 的能力矩阵。
- 借鉴 Pixelle `pixelle_video/services/api_media.py::APIProviderMediaService.__call__` 的统一执行入口。
- 不照搬硬编码模型列表和业务层供应商分支。

### 坑提醒

- 不同模型允许的 durationSeconds 不一样，不能让前端随便传。
- 不同 workflow preset 的输入节点不同，必须走 `param_schema / node_map / output_map`。
- 没有 selectedImageId 的分镜不能进入图生视频。
- 单个分镜失败不能影响其他分镜结果保存。
- 不要把“最新生成成功的视频”自动当成最终片段。
- 不要为了重生成清空旧片段；历史片段和已确认片段必须可回看。

## 子功能 2：视频提示词生成

### 系统逻辑

1. 从 StoryboardItem.sourceText / narrationText / visualDescription 生成 `videoPrompt`。
2. 批量 10 条。
3. 重试 3 次。
4. 校验返回数量。
5. 用户编辑后可锁定，批量重生成不得覆盖。

### 参考项目取舍

- 借鉴 Pixelle `pixelle_video/utils/content_generators.py::generate_video_prompts`。
- 借鉴 Pixelle `pixelle_video/prompts/video_generation.py::build_video_prompt_prompt`。

### 坑提醒

- videoPrompt 描述“画面怎么动”，不要把它和 imagePrompt 混用。
- 景别/运镜等概念如果要用，优先放到 videoPrompt 或高级字段里。

## 子功能 3：视频模型能力矩阵

### 系统逻辑

模型能力矩阵以 `底层设计/12-Provider接口契约与能力矩阵Schema.md` 和 `底层设计/23-模型适配与工作流注册规范.md` 为准。

API 视频模型写入 `provider_models`：

```json
{
  "ability_types": ["text_to_video", "image_to_video"],
  "input_modalities": ["text", "image"],
  "duration_seconds": { "min": 2, "max": 15, "integer": true },
  "resolutions": ["720P", "1080P"],
  "supported_aspect_ratios": ["horizontal_16_9", "vertical_9_16", "square_1_1"]
}
```

ComfyUI / RunningHub 视频工作流写入 `workflow_presets`，能力由 preset 声明：

```json
{
  "source_type": "workflow_preset",
  "ability_types": ["image_to_video", "start_end_frame_i2v"],
  "input_modalities": ["text", "image"],
  "output_modalities": ["video"],
  "limits": {
    "duration_seconds": { "min": 2, "max": 6, "integer": true },
    "resolutions": ["720p"]
  }
}
```

### 参考项目取舍

- 借鉴 Pixelle `pixelle_video/services/api_media.py::APIProviderMediaService.VIDEO_MODEL_CAPABILITIES`。
- 借鉴 Pixelle `pixelle_video/services/api_media.py::_video_duration`。
- 不照搬 Pixelle 的硬编码矩阵，落库并做 schema 校验。

### 坑提醒

- 能力矩阵必须由后端校验，不能只靠前端禁用。
- 前端只展示 `list_executable_media_options` 的返回结果，不自行拼候选项。

## 子功能 4：失败重试和审核中性化

### 系统逻辑

1. Provider 返回审核失败。
2. 调 LLM 中性化 prompt。
3. 重试。
4. 记录原始 prompt 和改写 prompt。
5. 失败行可单独重试。

### 参考项目取舍

- 借鉴 Pixelle `pixelle_video/services/api_media.py::_neutralize_video_prompt`。

### 坑提醒

- 中性化只处理审核风险，不要改掉核心动作。
- 原 prompt 要保留，方便排查。

---

## 子功能 5：文生视频 text_to_video（后续扩展）

### 系统逻辑

1. 读取 visualDescription 或 videoPrompt。
2. 从 `list_executable_media_options` 选择支持 `text_to_video` 的候选项。
3. 后端根据能力矩阵检查 limits。
4. 组装 VideoRequest。
5. 输出视频片段。

### 坑提醒

- 文生视频成本高、失败率高，不作为当前主线依赖。
- 视频候选项不可用时，不应出现在默认主线里。

---

## 子功能 6：首尾帧 / 尾帧续接（后续扩展）

### 系统逻辑

1. 当前镜头生成后，提取尾帧。
2. 下一镜头把上一镜头尾帧作为参考首帧。
3. 如果模型支持 start/end frame，则同时传首帧和尾帧。

### 参考项目取舍

- 借鉴 Pixelle `pixelle_video/pipelines/asset_based.py` 中素材视频尾帧续接思路。
- 借鉴 Toonflow `src/utils/ai.ts` 的 VideoMode / referenceList 设计。
- 不照搬 Toonflow 的供应商私有参数到业务层。

### 坑提醒

- 尾帧续接会增加失败率，要允许用户关闭。

## 四、这个模块对外提供什么

写回：

```json
{
  "itemId": "item_001",
  "videoSegments": [],
  "selectedVideoSegmentId": "seg_001",
  "videoStatus": "confirmed"
}
```

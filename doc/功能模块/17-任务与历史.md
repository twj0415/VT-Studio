# 任务与历史

> 这是给 AI 开发用的功能说明书。这个模块是全项目可靠性的核心。
> 配套阅读：字段看 `底层设计/数据结构.md`；状态流转看 `底层设计/06-任务状态机.md`；枚举取值看 `底层设计/01-枚举字典与配置规范.md`；事件错误看 `底层设计/09-错误日志与事件规范.md`。

## 一、这个功能是什么

任务与历史模块负责记录每次生成动作的状态、进度、错误、输出文件，并支持失败后继续生成。

两个参考项目都有问题：

- Toonflow 多处 fire-and-forget，进程重启任务丢。
- Pixelle `TaskManager` 是内存态，重启任务丢。

所以我们必须自研持久化任务系统。

### 参考项目补充机制

waoowaoo 的体验强在任务态可见：运行中 badge、阶段进度、流式反馈和可最小化的过程提示，让用户知道系统正在处理哪一步。Pixelle 的 `ProgressEvent` 提供了媒体处理过程的事件模型。

本项目吸收体验，但不吸收任务可靠性缺点：

```text
1. 页面可以展示类似流式控制台 / 阶段日志，但最终状态必须以 SQLite Task / TaskStep 为准。
2. 生图、视频、合成页必须显示总数、成功数、失败数、当前行、当前步骤和已用时。
3. 不显示“预计剩余时间 / ETA”，AI 生成波动大，这个值不可靠。
4. 取消后保留已完成 ImageCandidate / VideoSegment / CompositionTask / Artifact。
5. 失败必须能定位到 task_id、step_kind、item_id / image_id / segment_id 和脱敏 error_json。
6. 页面刷新后应从后端状态恢复，不依赖前端内存 loading。
```

### 页面落点

任务不是一个单独一级菜单，而是贯穿作品内部：

```text
我的作品卡片：显示当前阶段、失败摘要、可恢复按钮、资源消耗摘要。
工作台 Header：显示当前任务运行中 / 失败 / 可继续，以及图片 / 视频 / LLM 调用次数。
生图页底部：TaskProgressPanel，显示 5/8、当前行、当前步骤、已用时、失败数、取消和重试。
视频页底部：同生图页，但当前对象是 VideoSegment。
合成页右侧或底部：FFmpeg 检查、转码、拼接、输出写入的阶段状态。
历史详情：TaskStep 列表、脱敏日志、Artifact、恢复动作。
```

流式控制台只用于体验：

```text
可以显示 Provider / LLM / FFmpeg 的阶段日志、逐步消息、最小化面板。
不能把流式消息当最终状态源。
不能靠前端数组保存历史。
页面刷新后必须重新从 Task / TaskStep / Artifact 恢复。
```

### 资源消耗统计

资源消耗统计归任务体系，不单独做一套账。

展示口径：

```text
图片 N 次
视频 N 次
LLM N 次
TTS N 次（后续）
Workflow N 次（可并入图片 / 视频，也可单独展示）
```

数据来源优先级：

```text
1. TaskStep / Attempt 的 provider_call_count、ability_type、provider_model_id、workflow_preset_id。
2. Artifact 的产物类型和生成来源。
3. generation_context_snapshot 里的脱敏模型来源。
4. 如果没有真实统计字段，只显示“未接入统计”或隐藏，不允许写假次数。
```

禁止展示：

```text
API Key
Bearer Token
完整请求头
本地绝对路径
完整未脱敏 prompt 原文
不可靠费用估算
```

## 子功能 1：创建任务

### 系统逻辑

1. 用户点击生成动作。
2. 创建 Task，`task_status=pending`。
3. 按 `task_kind` 创建 TaskStep 列表。
4. 冻结任务运行时快照。
5. 创建任务目录。
6. 启动 Pipeline Engine。
7. Task 进入 `running`。

### 数据结构

```json
{
  "task_id": "task_xxx",
  "project_id": "proj_xxx",
  "task_kind": "image_to_video_pipeline",
  "task_status": "running",
  "current_step_kind": "tts_generation",
  "progress": 0.3,
  "snapshot_json": {},
  "steps": []
}
```

完整字段以 `底层设计/数据结构.md` 为准。

### 参考证据 / 本项目落地

- Pixelle `api/tasks/manager.py::TaskManager.create_task`。
- Pixelle `pixelle_video/utils/os_util.py::create_task_output_dir`。

### 我们的改动

Task 必须写 SQLite，不放纯内存。

## 子功能 2：TaskStep 状态机

### 步骤顺序

TaskStepKind 以 `底层设计/01-枚举字典与配置规范.md` 为准。当前 `image_to_video` 主线推荐顺序是：

```text
project_init
storyboard_generation
storyboard_review
image_prompt_generation
image_generation
image_review
video_prompt_generation
video_generation
video_review
final_composition
export
cleanup
```

后续增强步骤按 workflow 配置插入，不是取消：

```text
script_generation
script_review
tts_generation
subtitle_generation
cover_generation
cover_review
template_rendering
segment_composition
```

用户确认类步骤统一用 `review`，不用 `confirm`。

### 状态

TaskStatus 和 TaskStepStatus 以 `底层设计/06-任务状态机.md` 为准。

Task 整体状态：

```text
pending / running / succeeded / failed / cancelled
```

TaskStep 细状态：

```text
pending / running / retrying / succeeded / failed / skipped / cancelled / waiting_user
```

### 人工确认节点

以下步骤进入人工确认时使用 `waiting_user`：

```text
storyboard_review
image_review
video_review
script_review（脚本增强时）
cover_review（封面增强时）
```

用户确认后对应步骤变成 `succeeded`。

### 坑提醒

- 不要只给整个 Task 一个状态。必须细到 Step，才能断点续跑。
- 不要把 `retrying / skipped / waiting_user` 放进 TaskStatus，它们是步骤状态。

## 子功能 3：进度事件

### 系统逻辑

1. 后端每个步骤发 ProgressEvent。
2. Tauri event 推给前端。
3. 前端显示总进度、当前步骤、当前 StoryboardItem / ImageCandidate / VideoSegment。
4. 前端事件只做展示，最终状态以 SQLite 为准。

### 数据结构

```json
{
  "task_id": "task_xxx",
  "step_kind": "image_generation",
  "step_status": "running",
  "progress": 0.45,
  "item_id": "item_003",
  "message_key": "progress.image.generating",
  "created_at": 1718000000000
}
```

前端展示建议：

```text
████████░░░░ 5/8
当前：第 3 条 / image_generation
已用时：2 分 15 秒
失败：1 条，可重试
```

禁止展示：

```text
预计剩余 2 分 30 秒
```

### 参考证据 / 本项目落地

- Pixelle `pixelle_video/models/progress.py::ProgressEvent`。
- Pixelle `pixelle_video/services/frame_processor.py` 每帧 4 步进度。
- Toonflow Socket.io 流式 thinking 体验，迁移到 Tauri event。

## 子功能 4：失败重试

### 系统逻辑

1. Provider / FFmpeg / Chromium / Storage 调用失败后按错误分类。
2. 可重试错误进入 `retrying`，指数退避后再执行。
3. 不可重试错误进入 `failed`，停止并提示用户。
4. 重试次数写入 TaskStep。
5. 最终失败写入 `error_json` 和 Task 的 `last_error_json`。

### 坑提醒

- 鉴权失败不要反复重试。
- 审核失败可以走 prompt 中性化重试。
- 重试策略来自任务运行时快照，不受用户后续改配置影响。

---

## 子功能 5：取消任务

### 系统逻辑

1. 用户点击取消。
2. Task 状态置 `cancelled`。
3. 当前运行中的 TaskStep 置 `cancelled`。
4. 给正在执行的 Provider / FFmpeg / Chromium 发送取消信号。
5. 已完成产物保留。
6. 历史记录中标明任务已取消。

### 参考证据 / 本项目落地

- Pixelle `api/tasks/manager.py::TaskManager.cancel_task`。
- Toonflow `src/routes/assetsGenerate/cancelGenerate.ts`。

---

## 子功能 6：断点续跑

### 系统逻辑

1. 读取 TaskStep 状态。
2. 找到第一个 `failed / pending / retrying` 步骤。
3. 如果遇到 `waiting_user`，必须等用户确认。
4. 校验该步骤需要的输入文件是否存在。
5. 从该步骤继续。
6. 已 `succeeded` 的步骤不重复跑，除非用户强制重跑。
7. 重跑某一步时，按影响范围重置后续步骤状态。

### 坑提醒

- 断点续跑必须基于 DB 状态 + 文件存在性 + 任务快照三重校验。
- 不要因为用户改了全局 Provider 配置，就改变历史任务续跑使用的模型。

## 子功能 7：任务运行时快照

任务创建时必须冻结：

```json
{
  "provider_snapshot": {},
  "model_snapshot": {},
  "pipeline_config_snapshot": {},
  "template_snapshot": {},
  "retry_policy_snapshot": {}
}
```

原因：

```text
1. 用户在任务运行中改 Provider，不能影响正在运行的任务。
2. 断点续跑必须基于当时配置。
3. 历史任务要可追溯。
```

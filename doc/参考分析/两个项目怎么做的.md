# 两个项目怎么做的

> 这篇不是功能说明书，而是源码依据。后面的功能文档写「抄谁」时，优先引用这里。
> 结论先说清：**链路工程抄 Pixelle，AI 内容组织抄 Toonflow，安全和任务可靠性必须自研补强。**

## 一、Toonflow 怎么做

Toonflow 是 Electron + Express + TypeScript + better-sqlite3 的「小说 → AI 短剧」工具。它强在内容组织、Agent 编排、资产一致性，弱在 TTS、成片合成、安全沙箱和任务可靠性。

### 1. 长内容入口：小说分章 → 事件摘要

链路：

```text
小说导入
→ AI 识别章节正则
→ 按章节切分入库 o_novel
→ CleanNovel 并发逐章提取事件
→ event / event_status 回写章节
```

证据：

- `src/routes/script/getAiRegex.ts`：让 AI 根据原文片段返回章节标题正则。
- `src/routes/novel/addNovel.ts`：插入 `o_novel`，字段包括 `project_id / chapter_index / reel / chapter / chapter_content / event / event_status / error_reason`。
- `src/utils/cleanNovel.ts`：`CleanNovel` 类逐章调 LLM，默认并发 5，完成后 emit 单章结果。

可抄：

- 小说分章思路。
- 逐章事件提取。
- 章节级状态显示。

必须改：

- Toonflow 的 `event` 基本是 AI 文本原样入库，我们要改成结构化对象（人物、地点、动作、冲突等字段）。
- Toonflow 是 fire-and-forget，进程重启会丢；我们要纳入持久化任务系统。

---

### 2. AI 内容组织：ScriptAgent / ProductionAgent / Skill

Toonflow 的 Agent 不是多个常驻进程，而是「主 Agent + 工具化子 Agent」：主 Agent 决策，调用 `run_sub_agent_xxx` 工具，子 Agent 独立完成某一步，再把结果返回。

典型链路：

```text
ScriptAgent：故事骨架 → 改编策略 → 逐集剧本
ProductionAgent：导演规划 → 资产提取 → 分镜表 → 分镜面板 → 分镜图
```

证据：

- `src/agents/scriptAgent/index.ts`：脚本 Agent 主入口。
- `src/agents/productionAgent/tools.ts`：生产 Agent 工具，包含分镜、资产、工作区读写等操作。
- `src/agent/skillsTools.ts`：Skill 文件按需加载，含路径防护。
- `skills/art_skills/*`、`skills/*`：画风/叙事/导演类技能文件。

可抄：

- 长内容用分层 Agent：骨架 → 策略 → 剧本。
- Skill 文件化：把提示词能力做成可维护的 `.md`，不要全部写死在代码里。
- 只在需要时加载技能，避免上下文膨胀。

必须改：

- 短视频主题/文章生成不要完整照搬复杂 Agent 主循环；短视频主题/文章生成用 Pixelle 的一次性结构化输出更稳。
- Skill 要有版本和引用关系，不能散落不可追踪。

---

### 3. 资产和一致性：角色 / 场景 / 道具 / 衍生资产

Toonflow 把资产作为一等公民，类型包含角色、道具、场景、片段等。分镜生成图片时，会把相关资产图作为参考图喂给图像模型。

证据：

- `src/routes/assetsGenerate/*`：资产生成相关接口。
- `src/routes/production/batchGenerateImage.ts`：批量生成分镜图。
- `src/utils/ai.ts`：`Ai.Image` 支持多参考图等参数。
- `src/routes/artStyle/*`：画风库和画风相关接口。
- `src/routes/artStyle/extractStylePrompt.ts`：用视觉模型从参考图反推画风提示词。

可抄：

- 资产库模型。
- 角色/场景/道具引用到分镜。
- 多参考图保持角色一致性。
- 从参考图反推画风。

必须改：

- 我们的数据字段要统一到 `底层设计/数据结构.md`，功能文档只引用，不重复造字段。
- 生成图片后统一转存本地相对路径，不让远程 URL 直接进库。

---

### 4. 数据库和本地文件

Toonflow 用 SQLite（better-sqlite3 + knex），结构化实体存表，工作区长文本存 JSON。

证据：

- `src/lib/initDB.ts`：初始化数据库。
- `src/lib/fixDB.ts`：修表/迁移。
- `src/types/database.d.ts`：数据库类型定义。
- `src/utils/oss.ts`：本地文件读写、相对路径、路径防穿越。

可抄：

- SQLite 做本地持久化。
- 数据导入/导出能力。
- 文件访问走统一工具，避免到处直接读写。

必须改：

- 我们用 Tauri + Rust + sqlx，不抄 Node 运行时。
- PathGuard 要用 canonicalize/resolve 判断白名单根目录，不用字符串 startsWith。

---

### 5. Provider 与模型配置

Toonflow 的 Provider 很灵活：用户能写 TypeScript 供应商代码，运行时在 vm2 沙箱里执行；同时它支持简易/高级模型绑定，按不同 Agent 角色选模型。

证据：

- `src/utils/ai.ts`：统一 AI 调用入口。
- `src/utils/vm.ts`：运行用户供应商脚本。
- `src/routes/setting/vendorConfig/*`：供应商增删改、模型配置、连通性测试。
- `src/routes/setting/vendorConfig/modelTest/*`：文本/图片/视频模型测试。

可抄：

- `vendorId:modelName` 模型寻址。
- 简易/高级模型绑定。
- 设置页连通性测试。

必须避开：

- **不要抄 vm2。**vm2 已停维护且有逃逸/RCE 风险。默认固定 Provider，若做可编程供应商，用 isolated-vm + 出站白名单。
- 不要把 API Key 明文放数据库。

---

### 6. Toonflow 的硬伤

| 硬伤 | 表现 | 我们怎么处理 |
|---|---|---|
| TTS 空壳 | `AiAudio.run` 类型混乱，`catch(e){}` 静默吞错，供应商 TTS 多数未实现 | 直接做 Edge TTS 闭环 |
| 无真正成片引擎 | 后端没有完整 FFmpeg 拼接、字幕、混音链路 | 抄 Pixelle 的 FFmpeg + HTML 模板流水线 |
| 任务不可靠 | 多处 fire-and-forget，进程重启任务丢失 | 自研 DB 持久化队列和断点续跑 |
| vm2 风险 | 执行用户 TS 供应商代码 | 禁用任意代码执行 |
| 密钥明文 | vendor 配置里存 API Key | 用系统钥匙串 |
| RAG 无阈值 | 向量召回只排序 slice，可能召回无关内容 | 加最低相似度阈值 |

## 二、Pixelle-Video 怎么做

Pixelle 是 Python + Streamlit + FastAPI 的「主题/文案/素材 → 短视频」工具。它强在工程生产链路：TTS、图片/视频生成、HTML 字幕模板、FFmpeg 合成、ComfyUI/API 接入；弱在项目概念、角色一致性、任务持久化和密钥安全。

### 1. 输入入口：generate / fixed / 素材模式

标准短视频入口在 `StandardPipeline`：

```text
setup_environment
→ generate_content
→ determine_title
→ plan_visuals
→ initialize_storyboard
→ produce_assets
→ post_production
→ finalize
```

证据：

- `pixelle_video/pipelines/standard.py::StandardPipeline`
- `pixelle_video/pipelines/standard.py::generate_content`
- `pixelle_video/utils/content_generators.py::generate_narrations_from_topic`
- `pixelle_video/utils/content_generators.py::split_narration_script`

两种模式：

| 模式 | Pixelle 做法 | 我们怎么抄 |
|---|---|---|
| generate | `generate_narrations_from_topic`：主题 + `target_scene_count` → LLM 返回 `narrations` | 主题输入/文章导入走 AI 重组 |
| fixed | `split_narration_script`：按 paragraph / line / sentence 切分 | 文案粘贴一字不改，只机械切分 |

素材模式入口：

- `pixelle_video/pipelines/asset_based.py::AssetBasedPipeline`
- `AssetBasedPipeline.setup_environment`：逐个分析用户上传的图/视频。
- `AssetBasedPipeline.generate_content`：用 LLM 结构化输出 `VideoScript`，直接把素材分配给场景。

可抄：

- generate/fixed 双模式。
- 固定文案绝不改写。
- 素材先 VLM 分析，再让 LLM 派给场景。

必须改：

- Pixelle 没有强项目概念；我们要保留 Project / StoryboardItem / ImageCandidate / VideoSegment / Task。
- fixed 切分后要给用户预览编辑，不能切完直接生成。

---

### 2. 旁白生成和提示词工程

Pixelle 的旁白生成在 `content_generators.py`，提示词在 `prompts/`。

证据：

- `pixelle_video/utils/content_generators.py::generate_narrations_from_topic`
- `pixelle_video/utils/content_generators.py::generate_narrations_from_content`
- `pixelle_video/prompts/topic_narration.py::build_topic_narration_prompt`
- `pixelle_video/prompts/content_narration.py::build_content_narration_prompt`

关键做法：

- 要求 LLM 返回 JSON，字段是 `narrations`。
- 返回数量大于目标就截断。
- 返回数量小于目标就抛错。
- topic 和 content 是两套提示词：主题是发散生成，文章是浓缩提炼。

可抄：

- 数量校验。
- topic/content 分开提示词。
- 输出 JSON，不让下游解析长文本。

必须改：

- Pixelle 少量地方错误直接抛给上层；我们要做统一重试和用户可读错误。

---

### 3. 分镜结构

Pixelle 的分镜比较扁平，适合执行流水线：

证据：

- `pixelle_video/models/storyboard.py::StoryboardConfig`
- `pixelle_video/models/storyboard.py::StoryboardFrame`
- `pixelle_video/models/storyboard.py::Storyboard`

核心字段：

```text
StoryboardConfig：media_width / media_height / n_storyboard / video_fps / tts_inference_mode / voice_id / tts_workflow / media_workflow / frame_template / template_params
StoryboardFrame：index / narration / image_prompt / audio_path / media_type / image_path / video_path / composed_image_path / video_segment_path / duration
Storyboard：title / config / frames / final_video_path / total_duration_seconds
```

可抄：

- 执行层按线性 frame 跑。
- `duration_seconds` 由音频/视频真实时长回填。
- 每帧记录音频、图片、合成图、视频片段路径。

必须改：

- Pixelle 的分镜字段不够导演化；我们要加 `visual_goal / characters / location / shot_size / camera_motion / emotion / transition_type`，见 `底层设计/数据结构.md`。

---

### 4. 图片/视频提示词生成

Pixelle 对提示词生成做了批处理和重试。

证据：

- `pixelle_video/utils/content_generators.py::generate_image_prompts`
- `pixelle_video/utils/content_generators.py::generate_video_prompts`
- `pixelle_video/utils/prompt_helper.py::build_image_prompt`

关键参数：

```text
batch_size = 10
max_retries = 3
min_image_prompt_words = 30
max_image_prompt_words = 60
```

关键校验：

- 每批返回数量必须等于本批 narrations 数量。
- JSON 解析失败或数量不一致会重试。
- 最后用 `build_image_prompt` 拼全局 `prompt_prefix`。

可抄：

- 批量 10 条。
- 最多重试 3 次。
- 数量强校验。
- 基础提示词 + 风格前缀组合。

必须改：

- 我们的风格不只是一串 `prompt_prefix`，要接入 Style Bible / Character Bible / Environment Bible。

---

### 5. LLM / TTS / 图片 / 视频 Provider

LLM：

- `pixelle_video/services/llm_service.py::LLMService`
- 直接用 OpenAI SDK 的 `AsyncOpenAI`。
- 支持 OpenAI-compatible：OpenAI、Qwen、DeepSeek、Ollama、中转等。
- `response_type` 支持 Pydantic 结构化输出。

TTS：

- `pixelle_video/services/tts_service.py::TTSService`
- `TTSService.__call__` 支持 `inference_mode="local"` 和 `"comfyui"`。
- local 模式调用 `pixelle_video/utils/tts_util.py::edge_tts`。
- 语速用 `pixelle_video/tts_voices.py::speed_to_rate` 转 Edge TTS rate。

图片/视频：

- `pixelle_video/services/media.py`：ComfyUI 工作流模式。
- `pixelle_video/services/api_media.py::APIProviderMediaService`：直连 API 模式。
- `APIProviderMediaService.VIDEO_MODEL_CAPABILITIES`：模型能力矩阵，记录 text_to_video / image_to_video / digital_human / action_transfer / duration / resolutions / ratios 等。

可抄：

- OpenAI-compatible LLM。
- Edge TTS 本地兜底。
- ComfyUI + 直连 API 双轨。
- 视频模型能力矩阵，防止传非法时长/分辨率。

必须改：

- Pixelle 的 config.yaml 存 api_key，是明文；我们用 keyring。
- 业务代码不能直接散调 Provider，要走 Provider Manager。

---

### 6. ComfyUI / RunningHub 接入

Pixelle 用统一基类扫描工作流。

证据：

- `pixelle_video/services/comfy_base_service.py::ComfyBaseService`
- `ComfyBaseService._scan_workflows`
- `ComfyBaseService._resolve_workflow`
- `ComfyBaseService._prepare_comfykit_config`

关键做法：

- 扫描 `workflows/selfhost/*.json` 和 `workflows/runninghub/*.json`。
- workflow key 形如 `selfhost/image_xxx.json`、`runninghub/tts_xxx.json`。
- RunningHub wrapper 里可读 `workflow_id`。
- 默认 workflow 从 config 读取，没有默认值就抛错。

可抄：

- 工作流目录约定。
- selfhost / runninghub 双来源。
- 配置热重载。

必须改：

- ComfyUI 能力要走清晰接口，未接通时不要半实现。

---

### 7. 帧流水线：TTS 驱动时长

Pixelle 最值得抄的工程链路是单帧处理。

证据：

- `pixelle_video/services/frame_processor.py::FrameProcessor`
- `FrameProcessor.__call__`
- `FrameProcessor._step_generate_audio`
- `FrameProcessor._step_generate_media`
- `FrameProcessor._step_compose_frame`
- `FrameProcessor._step_create_video_segment`

流程：

```text
1. TTS 生成 audio_path
2. ffprobe 读真实 audio duration
3. 如果是视频工作流，把 duration 作为目标视频时长传给 media
4. HTML 模板合成字幕帧
5. 图片/视频 + 音频合成 video_segment_path
```

可抄：

- **音频驱动时长**。TTS 先跑，用真实音频时长决定画面长度。
- 每个 frame 独立产物路径。
- 已有素材模式跳过媒体生成。

必须改：

- 我们要把每一步状态落库：当前主线使用 `image_status / video_status`，启用增强时再使用 `audio_status / subtitle_status`；模板渲染状态必须等字段在数据结构中定义后再接入，便于单步重试。

---

### 8. HTML 模板、字幕和封面

Pixelle 用 HTML + Playwright 渲染帧，而不是直接用 FFmpeg 字幕滤镜硬画。

证据：

- `pixelle_video/services/frame_html.py::HTMLFrameGenerator`
- `HTMLFrameGenerator.generate_frame`
- `HTMLFrameGenerator.parse_template_parameters`
- `templates/1080x1920/*.html`
- `templates/1920x1080/*.html`
- `templates/1080x1080/*.html`

关键做法：

- 模板路径包含分辨率目录，如 `1080x1920/image_default.html`。
- 模板里可用 meta 写媒体尺寸：`template:media-width` / `template:media-height`。
- 模板参数 DSL：`{{param:type=default}}`，支持 text / number / color / bool。

可抄：

- HTML 模板 + Headless Chromium 截图。
- 分辨率目录。
- 模板参数 DSL。

必须加强：

- 字幕是我们的亮点，要做逐字/卡拉OK、安全区、自动换行、描边、不同平台预设。
- 模板资源路径必须过 PathGuard。

---

### 9. FFmpeg 合成

Pixelle 的 `VideoService` 是成片核心。

证据：

- `pixelle_video/services/video.py::VideoService`
- `VideoService.concat_videos`
- `VideoService.merge_audio_video`
- `VideoService.create_video_from_image`
- `VideoService.overlay_image_on_video`
- `VideoService.add_bgm` / 内部 `_add_bgm_to_video`

关键做法：

- 拼接支持 demuxer（快，不重编码）和 filter（慢，兼容不同格式）。
- `merge_audio_video` 自动处理音画时长不一致：视频短则 freeze/black 补齐，视频长则按容差裁切。
- BGM 支持 volume 和 loop/once。

可抄：

- FFmpeg 服务层集中封装。
- 片段拼接 + BGM + 音画对齐。

必须改：

- Pixelle 依赖系统 FFmpeg；我们桌面端要 sidecar 打包并启动自检。
- fade_out 等细节要自己补，不要只停留在 BGM 音量。

### 10. 任务、历史、配置

任务：

- `api/tasks/manager.py::TaskManager`
- `_tasks` 和 `_task_futures` 都在内存。
- 支持 `create_task / execute_task / update_progress / cancel_task / _cleanup_old_tasks`。

历史：

- `pixelle_video/services/persistence.py`
- `pixelle_video/services/history_manager.py`
- `web/utils/history_persistence.py`

配置：

- `pixelle_video/config/manager.py::ConfigManager`
- `ConfigManager.reload / save / update / set_llm_config / set_comfyui_config / set_api_provider_config`
- `pixelle_video/config/schema.py` 定义配置结构。

可抄：

- ProgressEvent 粒度。
- 历史可回看 metadata + storyboard。
- 配置单例和热重载思路。

必须改：

- TaskManager 纯内存，重启即丢；我们必须 DB 持久化。
- API Key 明文配置，必须改 keyring。

## 三、最终取舍

| 领域 | 抄 Toonflow | 抄 Pixelle | 我们自研/补强 |
|---|---|---|---|
| 输入 | 小说分章、章节事件 | topic/fixed/素材三入口 | 统一 InputSource |
| 脚本 | 长内容分层 Agent | 短视频 JSON 直出 | 结构化校验+重试 |
| 分镜 | 导演化字段、角色场景镜头 | 线性 frame 执行 | StoryboardItem + ImageCandidate + VideoSegment schema |
| 角色一致性 | 资产库、多参考图、衍生资产 | 素材 VLM 分析 | Bible + lock_flags_json |
| TTS | 不抄 | Edge TTS + ComfyUI | 真实时长回填 |
| 字幕模板 | 不抄 | HTML + Playwright | 逐字字幕/安全区/样式系统 |
| 合成 | 不抄 | FFmpeg 全链路 | sidecar + PathGuard |
| Provider | 模型绑定、连通性测试 | OpenAI-compatible、能力矩阵 | keyring、安全 Provider Manager |
| 任务 | 部分状态思路 | ProgressEvent、取消 | 持久化队列、断点续跑、幂等重试 |
| 安全 | 路径隔离思路 | 少量资源目录思路 | keyring、PathGuard、禁 vm2 |

一句话：**Toonflow 给我们“怎么组织内容和一致性”，Pixelle 给我们“怎么真正生成并合成视频”。两个项目的坑都不能照搬。**

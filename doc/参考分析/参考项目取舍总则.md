# 参考项目取舍总则

> 本文是参考项目使用边界。开发时不能把任何参考项目当成完整蓝图，只能把它们当成能力来源。最终产品形态以本项目 `doc/` 设计、`plan/TODO` 顺序和现有 `Vue3 + Tauri + Rust + SQLite` 架构为准。

---

## 一、总原则

```text
waoowaoo          学前端流程清晰度和短视频制作节奏
Toonflow-app      学桌面端创作资源、Skill、画风/角色/场景一致性
Toonflow-web      学 Web 管理页的信息组织、模型配置和规则管理交互
Pixelle-Video     学真实生成链路、TTS、字幕模板、FFmpeg、ComfyUI/RunningHub/API 双轨
huobao-drama      只作为短剧业务形态补充参考，不作为核心技术蓝图
本项目自研        任务可靠性、密钥安全、PathGuard、桌面端体验、统一数据模型
```

必须坚持：

```text
1. 不复制参考项目代码、品牌、视觉 UI、素材、文案和接口命名。
2. 不把参考项目的技术栈搬进来。本项目仍是 Vue3 + Tauri 2 + Rust + SQLite。
3. 不因为某项目有某功能，就把它混入当前主线。当前主线仍是 image_to_video。
4. 不因为 MVP 简化，就删除后续功能。TTS、字幕、封面、BGM、素材成片、数字人都保留在功能账本和 TODO。
5. 操作按钮、状态、失败处理、重试、确认、回退、导出入口必须按功能账本验收，不能只做“能跑通”的空壳。
```

## 二、各参考项目定位

| 参考项目 | 重点学习 | 不学习 / 不照搬 |
|---|---|---|
| waoowaoo-main | 线性生产流程、页面推进节奏、短视频制作主路径、前端状态组织 | 不复制 UI 视觉、代码结构、接口命名；不把它当完整桌面应用蓝图 |
| Toonflow-app | 桌面端创作资源、Skill 文件、画风库、角色/场景/道具资产、多参考图一致性、长内容 Agent 思路 | 不照搬 Electron/Express 架构；不照搬 vm2；不复制品牌、技能内容和 UI |
| Toonflow-web | Prompt/Skill 管理、模型配置页面、管理端信息结构、规则绑定思路 | 后端不开源，不按完整系统复刻；不假设其服务端能力可二开 |
| Pixelle-Video | TTS、HTML 模板、字幕/封面渲染、FFmpeg 合成、ComfyUI/RunningHub/API 双轨、能力矩阵、素材分析流水线 | 不照搬 Streamlit/Python 架构；不明文保存密钥；不把 YAML 配置当主数据源 |
| huobao-drama | 短剧业务菜单、内容管理、前后端分层可做补充参考 | 不作为核心实现来源；不因它扩张成复杂短剧 SaaS |

## 三、最终产品形态

本项目不是参考项目拼盘，目标是一个统一的桌面工作台：

```text
我的作品
→ 开始创作
→ 内容创作
→ 分镜
→ 生图
→ 视频
→ 合成
→ 导出 / 历史回看
```

创作资源作为辅助能力：

```text
视频包
创作规则
素材库
画风 / 角色 / 场景 / 道具
```

模型能力作为独立管理：

```text
Provider 连接配置
API 模型能力
ComfyUI / RunningHub Workflow Preset
连通性测试
```

系统设置只放运行偏好和安全配置：

```text
主题
语言
工作区路径
sidecar 检测
密钥安全
日志诊断
导出默认值
```

## 四、不能做成四不像的硬规则

1. 当前主线页面必须统一成工作台式流程，不做多个项目风格混杂的独立工具页。
2. 按钮命名必须服务当前任务，不直接搬参考项目按钮文案。
3. 每个页面必须有明确主操作、次操作、危险操作和恢复操作。
4. 每个生成步骤必须有 `未生成 / 运行中 / 失败 / 等待选择 / 已选择 / 已确认` 之类的状态闭环。
5. 每个 AI 结果都必须能编辑、重生成、保留旧结果、选择最终结果。
6. 每个真实生成都必须能失败重试，不能一个失败导致整个作品状态混乱。
7. 每个需要用户选择的地方都必须有缺项提示，不能靠按钮静默禁用。
8. 每个远程产物必须转存本地相对路径，不让远程 URL 或用户绝对路径成为主数据。
9. 每个模型调用必须走 ProviderManager，不在页面或业务模块按供应商写分支。
10. 每个后续增强必须写清“不阻塞当前主线”，但不能从文档中删除。

## 五、开发时怎么用参考项目

开发某个功能前，按顺序读：

```text
1. doc/参考分析/功能借鉴矩阵.md
2. doc/功能模块/对应模块.md
3. doc/底层设计/对应底层规范.md
4. doc/参考分析/两个项目怎么做的.md
5. 必要时再打开参考项目源码确认
6. 最后回到 plan/TODO 当前阶段施工
```

禁止：

```text
1. 直接打开参考项目照着改本项目代码。
2. 只看一个参考项目，不看功能借鉴矩阵。
3. 为了复刻某个参考项目的交互而改变本项目主线。
4. 因为某个参考项目没做某能力，就把本项目功能删掉。
```

# 功能对比总表

> 对比对象：项目一 **Toonflow**（Electron + Express + TypeScript + better-sqlite3，小说→AI短剧工厂，多Agent）；项目二 **Pixelle-Video**（Python + Streamlit + FastAPI，主题/文案/素材→短视频，基于 ComfyUI/RunningHub/直连API）。
> 「推荐我们怎么做」原则：链路工程借鉴 Pixelle 的成熟实现，AI 内容组织借鉴 Toonflow 的强项；字幕/封面/任务可靠性自研做亮点。密钥、沙箱、workflow、模型能力只借鉴思路，不照搬实现。
> 这张表是"决定做哪些功能"的依据；每个功能"具体怎么做"在 `功能模块/` 对应文档里展开。

## 一、输入与项目

| 功能 | 项目一(Toonflow)怎么做 | 项目二(Pixelle)怎么做 | 推荐我们怎么做 |
|---|---|---|---|
| 输入源类型 | 小说原文整本/分章导入，按"卷/章/章节内容"结构化入库 `routes/novel/addNovel.ts`，`o_novel` 表 | 三种入口：主题(topic→LLM扩写)、固定文案(fixed→切分)、用户素材(图/视频)；`pipelines/standard.py`、`asset_based.py` | 两条入口都做：长文本(小说/文章)走 Toonflow 式结构化，短主题/文案走 Pixelle 式 topic/fixed/素材；入口层抽象成统一 `InputSource` |
| 项目/工程概念 | 强项目概念：`o_project` 存画风、画幅、图像/视频模型、导演手册、视觉手册 `routes/project/*` | 无显式"项目"，以 task_id 为单位(`create_task_output_dir`)，配置全局 config.yaml | 采用 Toonflow 的项目概念(便于复用画风/角色/模型绑定)，但任务执行用 Pixelle 的 task_id 隔离目录 |
| 章节/分集管理 | 章节列表、章节事件状态机 `getNovelIndex.ts`/`getNovelEventState.ts`，支持批量删除 | 无 | 借鉴 Toonflow 章节状态机，做"集/段"级别进度可视化 |
| 小说→事件摘要(预处理) | `cleanNovel.ts`：按章节并发(默认5)调 LLM 抽取"事件摘要"，`generateEvents.ts`，emitter 回写 `event/event_status` | 无（直接 topic→narration） | 借鉴 Toonflow 的事件抽取做长文本压缩层；但要加失败重试与阈值（见任务类） |
| 项目统计/总览 | `general/generalStatistics.ts` 项目级统计 | 无 | 自研做项目仪表盘（任务数、成功率、耗时） |
| 导演手册/视觉手册 | 项目可挂"导演手册""视觉手册"文本 `addDirectorManual.ts`/`addVisualManual.ts`，注入 Agent 上下文 | 无 | 借鉴 Toonflow，作为可复用的风格/叙事约束注入提示词 |
| 多语言输入 | 中文为主 | prompt 内强制"输出语言跟随输入语言" `prompts/topic_narration.py` | 借鉴 Pixelle 的语言一致性约束写进所有提示词 |

## 二、脚本/分镜/提示词

| 功能 | 项目一(Toonflow)怎么做 | 项目二(Pixelle)怎么做 | 推荐我们怎么做 |
|---|---|---|---|
| 脚本/剧本生成 | 多Agent：决策Agent→子Agent(故事骨架/改编策略/剧本)，`agents/scriptAgent/index.ts`，技能(skill .md)驱动，XML标签写回工作区 | LLM 直接生成 narrations 列表(JSON)，`content_generators.generate_narrations_from_topic` | 内容组织借鉴 Toonflow 的分层Agent(骨架→策略→剧本)做长剧；短视频借鉴 Pixelle 的单次 JSON 直出。统一产物为分镜数组 |
| 旁白/解说文案 | 由剧本派生，台词写入分镜 videoDesc | 核心能力：topic/content→narration，含字数约束、开头多样性约束、引用规范 `prompts/topic_narration.py` | 借鉴 Pixelle 的 narration 提示词工程（开头去重、字数自检、引用不杜撰），这是其最成熟资产 |
| 固定文案切分 | 无 | `split_narration_script`：按段落/行/句切分 | 借鉴 Pixelle，保留三种切分模式 |
| 分镜(storyboard)结构 | 完整分镜面板：duration/prompt/track/关联资产/景别/运镜/情绪 `agents/productionAgent/tools.ts` `add_flowData_storyboard` | 扁平 frame：narration + image_prompt + duration，`models/storyboard.py` | 借鉴 Toonflow 的丰富分镜字段(运镜/景别/情绪)，但执行模型用 Pixelle 的线性 frame 列表 |
| 分镜表/拍摄计划 | 专门子Agent生成"拍摄计划""分镜表" `run_sub_agent_director_plan`/`storyboard_table` | 无独立分镜表 | 借鉴 Toonflow，做导演视角的拍摄计划中间产物 |
| 图像提示词生成 | 由资产/分镜的 prompt 字段承载，Agent 产出 | `generate_image_prompts`：分批(batch10)+重试(3次)+数量校验+前缀拼接 `prompt_helper.build_image_prompt` | 借鉴 Pixelle 的分批+重试+数量校验（健壮），加 Toonflow 的画风前缀注入 |
| 视频提示词生成 | `generateVideoPrompt.ts`/`batchGeneratePrompt.ts` | `generate_video_prompts`(同上分批重试) | 同上，统一一套"批量+重试+校验"提示词管线 |
| 提示词润色 | `assetsGenerate/polishAssetsPrompt.ts`/`batchPolishAssetsPrompt.ts` | 无显式润色，但有"内容审核失败→中性化改写"`api_media._neutralize_video_prompt` | 两者都要：常规润色(Toonflow) + 审核失败自动中性化重试(Pixelle，亮点) |
| 提示词模板管理 | `setting/promptManage/*`、`modelMap/bindingPrompt`：提示词可编辑、绑定模型、useData覆盖 | prompts 为代码内常量(`prompts/*.py`) | 借鉴 Toonflow：提示词入库可编辑、可绑定模型、可版本化，不写死在代码 |
| AI正则/资产抽取 | `script/getAiRegex.ts`、`extractAssets.ts` 从剧本抽角色/场景/道具 | LLM 结构化输出(Pydantic VideoScript) 直接产 scene 列表 | 借鉴 Pixelle 的 Pydantic 结构化输出(可靠解析)，借鉴 Toonflow 的从剧本自动抽资产 |

## 三、视觉资产与一致性（画风/角色/场景）

| 功能 | 项目一(Toonflow)怎么做 | 项目二(Pixelle)怎么做 | 推荐我们怎么做 |
|---|---|---|---|
| 画风(art style)体系 | 一等公民：`artStyle/*`，画风库、从参考图反推画风提示词 `extractStylePrompt.ts`(VLM)，画风技能目录 `skills/art_skills/<style>` | 仅一个全局 `prompt_prefix` 字符串(火柴人风格默认) `config.schema.ImageSubConfig` | 借鉴 Toonflow 的画风体系(画风库+从参考图反推+技能目录)，远胜 Pixelle 的单字符串前缀 |
| 角色一致性 | 资产(role/tool/scene/clip)+衍生资产，生成分镜时把关联资产图作为 referenceList 多图参考喂入 `batchGenerateImage.ts` `getAssetsImageBase64` | 弱：素材模式靠 VLM 描述+LLM 派场景；无角色锚定 | 借鉴 Toonflow 的资产+衍生资产+多图参考机制做角色一致性，这是 Toonflow 核心优势 |
| 场景/道具资产 | 同上，统一 `o_assets`，类型 role/tool/scene/clip，可批量生成图 `assetsGenerate/*` | 无资产库概念 | 借鉴 Toonflow 的统一资产模型 |
| 资产图生成 | `Ai.Image().run()`，按项目 imageModel + 画质 + 画幅，多参考图 `utils/ai.ts` AiImage | `media()` 走 ComfyUI 工作流或直连API(`api_media`)，模型矩阵丰富 | 生成链路借鉴 Pixelle 的 ComfyUI/API 适配；资产组织与多参考借鉴 Toonflow；具体执行必须走 provider_models / workflow_presets |
| 用户素材分析 | `uploadClip.ts` 上传素材 | 强项：`image_analysis`/`video_analysis`/`api_asset_analysis` 用 VLM 分析图/视频生成描述，再交 LLM 派镜 `asset_based.py` | 借鉴 Pixelle 的素材 VLM 分析→LLM 派镜，做"已有素材成片"能力 |
| 衍生资产 | 衍生资产(同一角色不同姿态/表情)`add_deriveAsset`/`generate_deriveAsset` | 无 | 借鉴 Toonflow |
| 图像编辑/流程图 | `production/editImage/*`：图像流程(flow)、上传、默认模型、生成流程图 | 无 | 视需要借鉴 Toonflow（无限画布精修，见 16-画布精修.md） |
| 首尾帧/尾帧续接 | 视频模式支持 startEnd/首尾帧 `utils/ai.ts` VideoMode | 素材模式提取上一镜尾帧作为下一镜首帧参考 `asset_based._extract_video_tail_frame` | 借鉴 Pixelle 的尾帧续接(镜头连贯)，结合 Toonflow 的首尾帧模式枚举 |

## 四、生成接入（LLM/TTS/图片/视频/ComfyUI）

| 功能 | 项目一(Toonflow)怎么做 | 项目二(Pixelle)怎么做 | 推荐我们怎么做 |
|---|---|---|---|
| LLM 接入 | 多 provider via AI SDK，供应商 TS 脚本在 **vm2 沙箱**里 eval `utils/ai.ts`+`utils/vm.ts`；按 Agent 角色映射模型(简易/高级配置) | 直连 OpenAI 兼容 SDK `services/llm_service.py`，支持结构化输出(Pydantic)，热重载 config | LLM 借鉴 Pixelle 的直连 OpenAI 兼容(简单稳)，**绝不照搬 vm2 eval 第三方代码**(RCE 风险)；保留 Toonflow 的按角色映射模型 |
| 供应商扩展机制 | 用户提交 TS 代码定义 textRequest/imageRequest/videoRequest，`addVendor.ts`→写盘→运行时 `transform`+`vm()` 执行 **(vm2 已废弃，存在 RCE)** | 内置 provider 适配器(dashscope/openai/seedream/kling/seedance)，能力矩阵硬编码 `api_media.VIDEO_MODEL_CAPABILITIES` | 用 Pixelle 的"内置适配器+能力矩阵"模式（声明式、可审计）替代 Toonflow 的"跑用户代码"，安全得多 |
| TTS | **空壳**：`AiAudio.run` 用 ttsRequest，但 `catch(e){}` 静默吞错且类型用的是 VideoConfig，实际无可用 TTS 产物链路 `utils/ai.ts:324-351` | 真实可用：本地 Edge TTS(`tts_util.edge_tts`，速度→rate) + ComfyUI TTS 工作流双模式 `services/tts_service.py`，含 voice 库 `tts_voices.py` | 借鉴 Pixelle 的 TTS(本地 Edge TTS 兜底 + 工作流/API 可选)；**避开 Toonflow 的 TTS 空壳和静默吞错** |
| 图片生成 | `Ai.Image` 单一路径，base64 落 OSS | ComfyUI 工作流(self-host/RunningHub) + 直连API(dashscope wan/gpt-image/seedream) 双轨 `media`/`api_media` | 借鉴 Pixelle 双轨(自托管 ComfyUI + 云API)，给不同成本/质量档位；API 模型进 provider_models，ComfyUI/RunningHub 进 workflow_presets |
| 视频生成 | `Ai.Video`，支持多参考/首尾帧/文生视频多 mode，`generateVideo.ts` 落 `o_video` | 丰富模型矩阵 + 能力声明(t2v/i2v/r2v/数字人/动作迁移/原生音频)，时长/分辨率按 provider 归一化 `api_media._video_duration` | 借鉴 Pixelle 的能力矩阵+参数归一化(避免传非法时长/分辨率)，借鉴 Toonflow 的 mode 枚举语义；不照搬硬编码模型矩阵 |
| ComfyUI/RunningHub | 无 | 核心：`comfy_base_service` 扫描 workflows/{selfhost,runninghub}，ComfyKit 懒加载+配置hash热重载 `service._get_or_create_comfykit` | 借鉴 Pixelle 的 ComfyUI/RunningHub workflow 扫描和热加载思路；不照搬业务层直接执行 workflow key，必须注册 workflow_presets 并校验 param_schema / node_map / output_map |
| 图像/视频理解(VLM) | `extractStylePrompt` 用图像模型反推画风 | `vlm_client`/`image_analysis`/`video_analysis` 多 provider VLM | 借鉴 Pixelle 的 VLM 抽象层(素材理解)，加 Toonflow 的画风反推用途 |
| 内容审核重试 | 无 | 视频生成触发审核失败→LLM 中性化改写 prompt 重试 `_neutralize_video_prompt`+关键词兜底 | 借鉴 Pixelle(亮点)，做敏感词中性化自动重试 |
| URL→base64/图片处理 | `urlToBase64`(3次重试)、sharp 压缩/拼接/缩放 `utils/vm.ts` | httpx 下载、ffmpeg 处理 | 借鉴 Toonflow 的图片预处理工具(压缩到目标大小、横向拼接多参考图) |

## 五、渲染/合成/成片

| 功能 | 项目一(Toonflow)怎么做 | 项目二(Pixelle)怎么做 | 推荐我们怎么做 |
|---|---|---|---|
| 成片合成引擎 | 无真正成片：分镜图/视频片段落库，靠前端"工作台/轨道"概念，后端无 ffmpeg 拼接 | **核心**：ffmpeg-python 全链路 `services/video.py`(拼接/混音/图生视频/叠加/补帧裁切) | 借鉴 Pixelle 的 ffmpeg VideoService(成熟完整)，这是 Toonflow 最大缺口 |
| 字幕/帧合成 | 无(分镜图即画面，字幕靠前端?) | HTML 模板 + Playwright 截图渲染字幕/封面 `frame_html.py`，模板含尺寸/媒体尺寸 meta、DSL 参数 | 借鉴 Pixelle 的 HTML+Playwright 渲染字幕；**字幕做自研亮点**(逐字/卡拉OK、安全区、自动换行、描边) |
| 帧处理流水线 | 无统一 frame 流水线 | `frame_processor`：TTS→图/视频→字幕合成→片段，TTS 时长驱动视频时长(零补帧) | 借鉴 Pixelle 的单帧流水线 + TTS 时长驱动(音画同步)，工程优雅 |
| 音画时长对齐 | 无 | `merge_audio_video` 智能补帧/裁切(freeze/black)、容差控制 | 借鉴 Pixelle |
| 视频片段拼接 | 无 | `concat_videos`(demuxer 快/filter 兼容) | 借鉴 Pixelle |
| 背景音乐(BGM) | 无 | `add_bgm`：音量/循环/淡入，BGM 资源库(`bgm/`+`data/bgm/`) `_resolve_bgm_path` | 借鉴 Pixelle，BGM 库 + 淡入淡出自研补全(其 fade_out 未实现) |
| 视频叠加(overlay) | 无 | `overlay_image_on_video`：透明 HTML 叠到视频(视频模板) | 借鉴 Pixelle |
| 封面/标题生成 | 海报概念 posterItem，工作台 cover 字段 | `generate_title` + HTML 模板渲染标题 | 借鉴 Pixelle 的 LLM 生成标题；**封面做自研亮点**(多套封面模板+智能选图) |
| 模板系统 | 无成片模板 | 按分辨率分目录 `templates/{1080x1920,1080x1080,1920x1080}`，HTML+DSL 参数(`{{param:type=default}}`) `frame_html.parse_template_parameters` | 借鉴 Pixelle 的分辨率模板体系 + DSL 参数化 |
| 多分辨率/画幅 | 项目级 videoRatio(16:9/9:16) | 模板路径含尺寸，media 尺寸 meta 解析 | 借鉴 Pixelle 的模板驱动尺寸 + Toonflow 的项目级画幅 |

## 六、任务/基础设施（任务系统/重试/记忆/数据库/密钥）

| 功能 | 项目一(Toonflow)怎么做 | 项目二(Pixelle)怎么做 | 推荐我们怎么做 |
|---|---|---|---|
| 任务系统 | 轻量：`o_tasks` 记录(进行中/完成/失败)，`utils/taskRecord.ts` 返回 done() 闭包；但生成是 **fire-and-forget**(路由 res 后台 .then，无队列/无恢复) `generateVideo.ts` | 内存 TaskManager `api/tasks/manager.py`：pending/running/完成/失败/取消、进度、自动清理；但**重启即丢**、纯内存 | 自研可靠任务系统做亮点：**持久化队列(DB)+断点续跑+取消+进度**，避开两者的"fire-and-forget"和"内存易失" |
| 任务进度 | 任务状态字段，socket thinking 流式 | `ProgressEvent` 细粒度(帧/步骤/百分比)+回调 `models/progress.py` | 借鉴 Pixelle 的细粒度进度事件 + Toonflow 的流式 socket 推送 |
| 重试机制 | 图像/视频 catch 落"生成失败"，**无自动重试**；urlToBase64 有3次重试 | 提示词分批重试3次、审核失败中性化重试、浏览器断连重启一次 `frame_html._reset_browser` | 借鉴 Pixelle 的分层重试；自研做"幂等重试+指数退避+死信" |
| 并发控制 | 简单批次 `Promise.all` 按 concurrentCount 分批；socket 操作串行队列(800ms) `productionAgent/tools.createSocketQueue` | RunningHub 并发 semaphore(可配 1-10) `standard.produce_assets` | 借鉴 Pixelle 的 semaphore 并发；自研做全局限流/配额 |
| 取消/中止 | `assetsGenerate/cancelGenerate.ts`、AbortSignal 透传 Agent | `cancel_task` future.cancel() | 两者结合：AbortSignal + 任务级取消 + 已生成资产保留 |
| 记忆/RAG | **亮点**：本地 onnx 嵌入(all-MiniLM)`embedding.ts`，记忆=短期+摘要+向量召回，自动滚动摘要 `utils/agent/memory.ts`，deepRetrieve 二段检索 | 无记忆系统 | 借鉴 Toonflow 的本地嵌入记忆做 Agent 长程记忆；**但避开"向量召回无相似度阈值"坑**(cosineSimilarity 排序后直接 slice，需加阈值过滤无关项) |
| 历史/持久化 | `o_*` 全表入 SQLite，导入导出 `dbConfig/exportData`/`importData` | task metadata + storyboard 落盘 JSON `persistence.py`，`HistoryManager` + History 页面 | 借鉴 Pixelle 的历史可回看(metadata+storyboard)，落 DB 而非散 JSON |
| 数据库 | better-sqlite3 + knex，建表/修表 `lib/initDB.ts`/`fixDB.ts`，类型 `types/database.d.ts` | 无 DB，全 config.yaml + 文件系统 | 用 SQLite(借鉴 Toonflow 的 schema 化)，比 Pixelle 的纯文件可查询/可统计 |
| 密钥管理 | **明文**：vendor apiKey 存 `o_vendorConfig.inputValues` JSON 明文 `agentSetKey.ts`/`addVendor.ts` | **明文**：api_key 存 config.yaml 明文 `config/schema.py` | 两者都踩明文坑→自研做系统钥匙串存储+脱敏日志(按 key 名引用不打印值) |
| 文件存储 | 本地"OSS"抽象 `utils/oss.ts`：根目录隔离、路径穿越校验(isPathInside)、base64读写、缩略图 | task 目录隔离 `os_util.create_task_output_dir`，resource 多目录合并(默认+data覆盖) | 借鉴 Toonflow 的路径穿越防护(PathGuard)+ Pixelle 的 task 目录隔离 |
| 沙箱/RCE 风险 | **vm2 执行用户 TS**(vm2 已停维护、有已知逃逸/RCE) `utils/vm.ts` | 无(适配器硬编码) | **坚决避开 vm2**；用声明式适配器或 worker 进程 + 严格白名单 |

## 七、配置/交互/系统

| 功能 | 项目一(Toonflow)怎么做 | 项目二(Pixelle)怎么做 | 推荐我们怎么做 |
|---|---|---|---|
| 前端形态 | Electron 桌面应用 + Express 后端 + Socket.io 实时 `socket/index.ts` | Streamlit Web(`web/`) + FastAPI(`api/`)双前端 | 我们用 Tauri 2 桌面端：重交互(分镜编辑/资产管理)走桌面体验，链路工程借鉴 Pixelle |

## 关键结论与避坑清单

- **链路工程借鉴 Pixelle**：ffmpeg 成片(`video.py`)、HTML+Playwright 字幕(`frame_html.py`)、帧流水线(`frame_processor.py`)、ComfyUI/API 双轨接入与能力矩阵(`api_media.py`)、TTS 双模式(`tts_service.py`)。这些是 Toonflow 完全缺失或做空的部分。
- **AI 内容组织借鉴 Toonflow**：分层 Agent(骨架/策略/剧本)、技能(skill .md)系统、画风库+从参考图反推画风、资产+衍生资产+多图参考的角色一致性、本地嵌入记忆。这些是 Pixelle 薄弱的部分。
- **模型与 workflow 不照搬参考项目实现**：Pixelle 的 API/ComfyUI 双轨、workflow 扫描、能力矩阵、参数归一化值得借鉴；但不照搬明文 key、硬编码矩阵、弱 schema、业务层 workflow key 直连。Toonflow 的模型测试、Agent 模型绑定、referenceList、内容组织可借鉴；但不照搬 vm2、用户 TS Provider、明文密钥、TTS 静默吞错。
- **自研做亮点**：① 字幕(逐字/卡拉OK、安全区、描边)；② 封面(多模板+智能选图)；③ 可靠任务系统(持久化队列+断点续跑+幂等重试+取消+细粒度进度)。
- **必须避开的坑**：
  1. **Toonflow TTS 空壳**：`AiAudio.run` 用错类型且静默吞错，无可用产物——别照搬，用 Pixelle 的 Edge TTS。
  2. **fire-and-forget 任务**：Toonflow 路由 `res` 后台 `.then()` 跑生成，进程重启即丢；Pixelle TaskManager 纯内存同样易失——必须持久化。
  3. **vm2 RCE**：Toonflow 用已停维护的 vm2 执行用户提交的 TS 供应商代码，存在沙箱逃逸/RCE——改用声明式适配器。
  4. **明文密钥**：两者都明文存——必须用系统钥匙串 + 日志脱敏。
  5. **向量召回无阈值**：Toonflow RAG 仅按相似度排序后 `slice`，不设阈值会召回无关记忆——加最低相似度门槛。

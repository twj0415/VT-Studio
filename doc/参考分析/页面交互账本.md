> 本文回答“最终有哪些页面、每个页面有什么区域、按钮、弹窗、状态和参考来源”。后续优化 UI 时先看本文，避免只凭感觉优化导致功能缺失。
>
> 页面视觉以本项目桌面工作台为准；参考项目只借鉴交互结构和流程，不复制 UI。

## 三、全局交互规则

1. 每个页面只保留一个最醒目的主操作。
2. 危险操作必须二次确认，例如删除作品、清理素材、删除 Provider。
3. 生成类按钮必须有 loading、禁用原因、失败重试入口。
4. Icon-only 按钮必须有 tooltip。
5. 表格行内操作不要超过 5 个常驻按钮，更多操作放入菜单。
6. 批量操作必须说明影响范围，例如“只生成失败项 3 条”。
7. 页面离开前如果有未保存编辑，必须自动保存或提示。
8. 所有抽屉 / 弹窗支持 Esc 关闭；有未保存内容时关闭要确认。
9. 所有列表和表格必须有空态、加载态、错误态。
10. 所有可点击区域至少 44px 高，桌面端也按这个标准保证可用性。
11. 批量生成进度不显示预计剩余时间或 ETA，只显示完成数、当前项、当前步骤、已用时、失败数和取消 / 重试入口。

## 五、内容创作页

### 页面区域

```text
左侧：输入来源 Tabs
中间：输入编辑区 / AI 结果预览
右侧：视频包、画幅、目标时长、分镜数、内容语言
底部固定操作条：保存草稿、创建并进入分镜
```

### 参考项目具体机制

```text
waoowaoo：LongTextDetectionPrompt 在长文本输入后先提示风险，不让用户直接进入不可控生成；SmartImportWizard 把长内容拆成来源、解析、映射、确认几步，导入成功后再进入后续生产。
Pixelle：generate / fixed 分成两种处理模式；fixed 只做段落 / 行 / 句机械切分。
Toonflow-web：小说和脚本入口有批量添加、编辑脚本和解析工具，但长内容能力不应变成当前短视频主线必经页。
```

深度吸收细节：

```text
1. 长文本提示必须是一次明确选择：智能切分 / 继续当前输入，不是只显示一行 warning。
2. 切分预览不只是展示数组，必须能编辑段落内容、调整顺序、删除、恢复原文。
3. 确认动作必须发生在写入 StoryboardItem 之前；未确认段落不能进入分镜。
4. 手动创建兜底必须存在：AI 整理失败时仍能用用户编辑后的段落继续。
5. 内容创作只确认 sourceText / narrationText，不编辑 visualDescription / imagePrompt / videoPrompt；这些属于分镜阶段。
```

### 输入来源

```text
素材说明（后续）
```

## 六、分镜页

### 页面区域

```text
顶部：项目摘要、当前规则、生成分镜、确认分镜、进入生图
左侧：分镜列表 / 缩略概览
中间：当前分镜编辑表单
右侧：AI 辅助、上下文、校验问题、影响范围提示
底部：保存状态、错误提示
```

也可以切换为表格视图：

```text
序号
原文
画面描述
角色
场景
imagePrompt
videoPrompt
时长
状态
操作
```

当前执行口径优先采用表格密集编辑。Inspector 不重复编辑主字段，只提供校验、AI 助手、候选预览和高级字段。

### 参考项目具体机制

```text
waoowaoo：分镜卡片不是空镜头，包含 description、characters、location、shot_type、camera_move、source_text、video_prompt、photographyRules、actingNotes、candidateImages、imageUrl；AIDataModal 可查看 / 调整摄影规则和表演指导，ai-modify-shot-prompt / ai-modify-location 围绕当前镜头修改。
Toonflow-web：script → scriptPlan → storyboardTable → storyboard → workbench 把文本规划、总纲表格、逐镜头卡片和生成结果分层；storyboardTable 提供总览，storyboard 卡片提供插入、批量、预览和生成图。
本项目：MVP 不拆两套主编辑页，用一个表格承接总览和主编辑，用 Inspector 承接辅助能力。
```

深度吸收细节：

```text
1. 表格主编辑必须承接 Toonflow storyboardTable 的总览价值：行号、快速跳转、状态筛选、无效行筛选、批量选择。
2. Inspector 承接 waoowaoo / Toonflow 的单镜头辅助价值：校验、AI 助手、候选预览、高级字段，不做主字段第二编辑器。
3. 高级字段固定为本项目字段 `shotSize / cameraMotion / composition / pace / transitionType`；waoowaoo 的 `photographyRules / actingNotes` 只能作为 Inspector 建议或后续 `advancedDirectingJson` 扩展，不能临时改名落库。
4. AI 助手可生成修改建议，应用时必须回写表格字段，并遵守 lockFlagsJson。
5. 删除、合并、拆分、插入都必须提示对 ImageCandidate[]、VideoSegment[]、selectedImageId、selectedVideoSegmentId 和合成状态的影响。
```

```text
右侧抽屉：当前行输入规划、参考图、缺项、模型参数

```

### 表格列

```text
序号

角色 / 场景
imagePrompt
最终 prompt 预览
参考图
候选图
已选图
状态
操作
```

### 参考项目具体机制

```text
waoowaoo：candidateImages 保留多候选，imageUrl 表示当前确认图；有生成、选择候选、清理未选、下载图片、撤回重生成等围绕候选集合的操作。
waoowaoo：ai-modify-shot-prompt 和 AIDataModal 让 prompt、摄影规则、演技指导贴着当前镜头走。
Toonflow-web：生成区支持全选、按状态选择、批量预览、历史图替换当前图、AI 润色 prompt、按资产类型筛选。
本项目：候选图进 ImageCandidate[]，最终选择写 selectedImageId；最终 prompt 弹窗展示系统实际发送的组装结果。
```

深度吸收细节：

```text
1. 候选区不是一个“选择图片”按钮，必须有候选缩略图、当前已选图、预览大图、选中态、确认、取消。
2. 重生成进入 waiting_select 时，旧 selectedImageId 仍保留；用户确认新候选前不能自动覆盖。
3. 撤回本次生成只撤回本次新增候选或恢复上一版选择，不删除历史可追踪记录。
4. 清理未选候选只能清理未选、未引用、可再生的候选文件；必须二次确认。
5. 保存为角色 / 场景 / 风格参考是显式资产化动作，完成后才登记 Asset / AssetReference。
6. 最终 prompt 预览必须标注来源：基础 imagePrompt、negativePrompt、角色、场景、画风、创作规则、模型 / workflow 约束。
```

### 参考项目具体机制

```text
Pixelle：图生视频围绕 i2v 能力矩阵执行，不同模型有不同 duration / resolution / aspectRatio 限制；审核失败可做中性化改写后重试。
Toonflow-web：VideoModelTest 把视频模型输入拆成 text、singleImage、startEndRequired、endFrameOptional、startFrameOptional、multiRef 等模式。
本项目：所有模式收敛成 VideoInputPlan，普通 image_to_video 只要求 selectedImageId + videoPrompt + durationSeconds；首尾帧和多参考作为后续增强。
```

深度吸收细节：

```text
1. 模型切换或输入模式切换如果会清空参考图、prompt 或候选视频，必须二次确认。
2. 普通 image_to_video 的必需输入固定为 selectedImageId、videoPrompt、durationSeconds。
3. 首尾帧、多参考、text_to_video 只能通过 VideoInputPlan 暴露，不能在页面按 modelName 写分支。
4. 生成视频后写 VideoSegment[]，选择后写 selectedVideoSegmentId；重生成不覆盖旧片段。
5. 审核失败中性化改写必须让用户确认，不能静默改写用户 prompt。
```

分层职责：

```text

视频包不负责：
  不保存 API Key
  不复制创作规则正文
  不保存 Provider 连接配置
  不直接保存素材文件内容
  不作为“从模板创建”的第二套入口
```

操作：

```text

绑定推荐模型能力
```

```text
video_packs.rule_refs_json 只保存 rule_key / rule_id。
video_packs.recommended_executable_refs_json 只保存 providerModelId 或 workflowPresetId。
video_packs.asset_refs_json 只保存 Asset / AssetReference 引用。
```

## 十二、模型 / 工作流页面

### 页面结构

必须拆成四个 Tab：


参考项目具体机制：

```text
Toonflow-web：vendorConfig 集中管理供应商和模型，modelMap 绑定模型与 prompt，promptManage / skillManagement 管理提示词和 Skill，vendorTest 拆出 Text / Image / Video 测试表单。
Pixelle：用 API/ComfyUI/RunningHub 双轨和能力矩阵判断媒体生成能不能执行。
本项目：拆成 Provider 连接、API 模型能力、Workflow Preset、测试记录四个施工层；其中前三个是配置，测试记录是审计和诊断。
```

## 十五、后续 UI 优化检查表

优化任何页面前先检查：

```text
1. 有没有把主操作藏起来？
2. 有没有删掉失败重试？
3. 有没有删掉候选选择？
4. 有没有删掉缺项提示？
5. 有没有把后续增强误做成当前主线阻塞？
6. 有没有把多个页面风格混成不同产品？
7. 有没有让按钮只靠 hover 才能发现？
8. 有没有让表格文字溢出或按钮挤压？
9. 有没有让批量操作影响范围不清楚？
10. 有没有让用户不知道下一步为什么不能点？
```

如果任一项是“是”，不能算优化完成。

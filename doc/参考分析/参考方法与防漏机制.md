# 参考方法与防漏机制

> 本文定义“怎么参考别的项目”。目标不是做一个拼盘，而是用系统化方法吸收优点，并保证功能、按钮、状态、异常、数据和后续扩展都不丢。

---

## 一、参考项目不是蓝图，是证据库

开发时不能说：

```text
这个页面像 Toonflow
这个流程像 waoowaoo
这个合成像 Pixelle
```

这种说法太粗，会漏功能。

必须改成：

```text
这个模块参考哪个项目的哪一类能力
具体学哪些页面区域
具体学哪些按钮
具体学哪些状态
具体学哪些后端逻辑
具体不学哪些坑
本项目保留哪些已有设计
最终比参考项目多补什么
```

参考项目只提供证据和优点，本项目的最终蓝图由以下文档决定：

```text
1. 产品蓝图 / 菜单页面设计
2. 功能借鉴矩阵
3. 页面交互账本
4. MVP 简化口径
5. 功能模块文档
6. 底层设计
7. plan/TODO
```

---

## 二、每个功能必须做“七层拆解”

参考任何功能时，都要拆成七层，缺一层就容易做少。

| 层级 | 要看什么 | 不看会漏什么 |
|---|---|---|
| 1. 用户目标 | 用户为什么进这个页面 | 页面做成工具堆叠，没有主线 |
| 2. 页面结构 | 有哪些区域、列表、表格、抽屉、弹窗 | UI 看起来像但不好用 |
| 3. 操作按钮 | 主按钮、批量按钮、行内按钮、危险按钮 | 后续用户要求补按钮 |
| 4. 状态流转 | 未开始、运行中、失败、等待选择、已确认 | 页面能点但不能追踪 |
| 5. 数据写回 | 最终写哪个字段、哪个表、哪个产物 | 结果丢失或下一步读不到 |
| 6. 异常恢复 | 失败、取消、重试、缺项、权限、费用 | 真实生成一失败就卡死 |
| 7. 本项目增强 | 参考项目没做好的地方如何补强 | 只复刻缺点，没有更优秀 |

每写一个页面或功能，都必须留下这七层结论。

---

## 三、参考时的标准表格

每个功能最终必须形成这样的对照：

| 项 | 结论 |
|---|---|
| 本项目功能 | 例如 AI 生图 |
| 用户目标 | 每个分镜生成可选图片，并选择最终图 |
| 主要参考 | Pixelle 生成链路 + Toonflow 多参考图 |
| 学页面结构 | 表格工作台、候选图区域、输入要求面板 |
| 学按钮 | 批量生成、只生成失败项、单行重试、选择候选、查看大图 |
| 学状态 | running、failed、waiting_select、confirmed |
| 学后端逻辑 | API/ComfyUI 双轨、能力矩阵、远程产物下载 |
| 参考项目缺点 | Pixelle 明文 key；Toonflow 任务不可靠 |
| 本项目增强 | keyring、TaskStep、ImageInputPlan、PathGuard、多候选保留 |
| 验收标准 | 没有 selectedImageId 不允许进入视频 |

如果某个功能没有这张表，就说明参考还不够细。

## 四、按钮防漏规则

页面按钮按六类检查：

```text
1. 主操作：当前页面最重要的一步。
2. 批量操作：批量生成、批量重试、只处理失败项。
3. 行内操作：单条编辑、单条生成、单条预览、单条确认。
4. 恢复操作：重试、继续、取消、回滚、重新选择。
5. 危险操作：删除、清理、覆盖、导入替换。
6. 辅助操作：筛选、搜索、排序、查看详情、打开目录。
```

开发页面前必须列出这六类。

例如生图页：

```text
主操作：进入视频
批量操作：批量生成全部 / 只生成未生成 / 只生成失败项
行内操作：编辑提示词 / 生成 / 重生成 / 选择候选 / 查看大图
恢复操作：失败重试 / 替换本地图 / 保留旧候选
危险操作：删除候选图 / 覆盖已选图，需要确认
辅助操作：筛选失败项 / 查看输入规划 / 保存为素材
```

如果一个页面只有主按钮，没有批量、行内、恢复和辅助操作，基本就是半成品。

## 五、状态防漏规则

所有生成类页面至少要覆盖：

```text
not_started / not_generated
input_missing
ready
running
failed
waiting_select
selected
confirmed
cancelled
```

状态必须回答：

```text
1. 用户现在能不能操作？
2. 下一步为什么不能点？
3. 失败原因是什么？
4. 失败后能不能重试？
5. 成功后是否还需要用户选择？
6. 用户选择后写回哪个字段？
```

参考项目里没有的状态，本项目要补。

典型增强：

```text
Pixelle 有生成进度，但任务偏内存态。
Toonflow 有一些任务状态，但 fire-and-forget 不可靠。
本项目必须补：持久化 TaskStep、失败重试、断点恢复、行级状态。
```

## 六、数据防漏规则

每个功能必须明确四类数据：

| 数据类型 | 问题 |
|---|---|
| 输入数据 | 从哪里读？用户输入、项目、分镜、素材、配置？ |
| 中间数据 | 是否需要保存草稿、候选、任务产物？ |
| 最终数据 | 最终写哪个字段？ |
| 派生数据 | 是否影响下游状态、是否需要重新生成？ |

例如 AI 视频：

```text
输入数据：StoryboardItem.selectedImageId、ImageCandidate.imagePath、videoPrompt
中间数据：VideoInputPlan、TaskStep、ProviderResponse
最终数据：VideoSegment[]、StoryboardItem.selectedVideoSegmentId
派生数据：合成页 readiness、作品卡片视频进度
```

任何只描述“调用模型生成视频”的功能文档，都不合格。

## 七、异常防漏规则

每个真实功能至少写清：

```text
1. 输入缺失怎么办？
2. Provider 不可用怎么办？
3. 密钥不存在怎么办？
4. 余额/费用/审核失败怎么办？
5. 远程 URL 下载失败怎么办？
6. 文件写入失败怎么办？
7. 用户取消怎么办？
8. 部分成功怎么办？
9. 重试是否覆盖旧结果？
10. 错误是否会泄露密钥或绝对路径？
```

参考项目常见缺点：

```text
Toonflow：后台任务 fire-and-forget，进程重启丢。
Toonflow：部分 catch 静默吞错。
Toonflow：明文密钥。
Pixelle：任务管理偏内存态。
Pixelle：config.yaml 明文 key。
Pixelle：部分错误直接向上抛，用户恢复路径不完整。
```

本项目必须补：

```text
TaskStep
AppError
ProgressEvent
SecretGuard
PathGuard
重试策略
行级失败恢复
```

## 八、页面防漏规则

每个页面必须写清：

```text
1. 页面目标
2. 页面区域
3. 顶部按钮
4. 行内按钮
5. 抽屉 / 弹窗
6. 空态
7. 加载态
8. 错误态
9. 禁用原因
10. 保存策略
11. 离开页面策略
12. 参考项目
13. 本项目增强
```

页面优化不能只看视觉。任何 UI 优化都要检查是否删掉了：

```text
缺项提示
失败重试
候选选择
锁定
预览
批量操作
危险确认
引用检查
```

## 九、参考项目优点吸收方式

### waoowaoo

参考方式：

```text
看流程是否清楚
看用户每一步下一步是什么
看页面是否围绕短视频生成主线
```

吸收：

```text
线性流程
步骤推进
前端体验节奏
```

本项目增强：

```text
桌面端工作台
持久化项目
多候选选择
模型 / workflow 动态输入规划
```

### Toonflow-app / Toonflow-web

参考方式：

```text
看创作规则怎么组织
看视频包和 Skill 怎么管理
看画风 / 角色 / 场景 / 道具如何作为资产
看模型配置和测试入口
```

吸收：

```text
Skill 文件化
创作规则可编辑
内置规则 / 用户规则
画风库
角色和场景一致性
多参考图
长内容分章和事件
```

本项目增强：

```text
MVP 降低 version/hash/snapshot 复杂度
keyring
禁用 vm2 / 用户 TS Provider
任务持久化
PathGuard
```

### Pixelle

参考方式：

```text
看真实媒体链路怎么跑通
看 TTS、字幕、FFmpeg、模板如何衔接
看 API / ComfyUI / RunningHub 如何分流
看能力矩阵如何限制非法参数
```

吸收：

```text
Edge TTS
HTML 模板渲染
FFmpeg 拼接和转码
ComfyUI / RunningHub / API 双轨
模型能力矩阵
素材分析
审核失败中性化
```

本项目增强：

```text
Rust ProviderManager
provider_models / workflow_presets 入库
不明文 key
不硬编码模型矩阵
桌面 sidecar 自检
持久化任务
```

## 十、“更优秀”的判断标准

不是功能更多就更优秀，而是每个参考优点都被本项目工程化：

| 参考优点 | 只照搬会怎样 | 本项目更优秀的做法 |
|---|---|---|
| waoowaoo 流程清楚 | 只是页面像 | 加任务状态、候选、失败恢复、桌面持久化 |
| Toonflow Skill | 规则复杂、版本负担重 | MVP 先 builtin/user + schema，后续再版本快照 |
| Toonflow 多参考图 | 容易堆资源、模型混乱 | 用 ImageInputPlan 决定 required/optional/unused |
| Pixelle FFmpeg | Python 服务依赖系统环境 | Tauri sidecar + PathGuard + ffprobe 自检 |
| Pixelle 能力矩阵 | 硬编码模型表 | provider_models / workflow_presets 入库校验 |
| Pixelle TTS | 可用但不一定适合主线 | 后续可选增强，不阻塞 image_to_video |
| 参考项目任务系统 | 容易丢任务 | 持久化 Task/Step/Attempt/Artifact |
| 参考项目密钥 | 明文风险 | keyring + SecretGuard |

## 十一、开发前强制检查

开始写任何 TODO 前，先检查：

```text
1. 功能借鉴矩阵里有没有这个模块？
2. 页面交互账本里有没有这个页面？
3. 本文七层拆解是否能回答？
4. 按钮六类是否覆盖？
5. 状态是否覆盖 not_generated / running / failed / waiting_select / confirmed？
6. 数据写回字段是否明确？
7. 异常恢复是否明确？
8. 参考项目缺点是否已避开？
9. 本项目已有优势是否保留？
10. 是否对应当前 TODO，而不是临时加戏？
```

如果有一项答不上来，先补文档，再写代码。

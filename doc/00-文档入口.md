# 文档入口

> 新窗口、新对话、新开发者只需要先读这一篇。读完以后应该清楚：产品最终做成什么样、先看哪些文档、从哪个 TODO 开始、参考项目怎么用、哪些旧文档不要优先看。

> 当前目标已经不是复刻某一个项目，而是：保留本项目已有工程长处，吸收参考项目的优秀能力，做一个自己的短视频 / 短剧创作桌面工作台。

> 参考项目只作为能力证据库。任何实现都必须回到本项目的菜单、数据结构、Provider、任务队列、存储和 TODO 顺序，不能照着参考项目页面或源码直接拼。

## 一、当前唯一主线

当前第一版只优先跑通：

```text
workflowType = image_to_video
inputType = topic | paste | article
```

主流程：

```text
我的作品
→ 开始创作
→ 内容创作
→ 分镜
→ 生图
→ 视频
→ 合成
→ 导出 / 历史
```

技术数据流：

```text
Project draft
→ StoryboardItem[]
→ ImageCandidate[]
→ StoryboardItem.selectedImageId
→ VideoSegment[]
→ StoryboardItem.selectedVideoSegmentId
→ CompositionTask.outputPath
→ ExportRecord
```

第一版必须做到：

```text
1. 用户能从主题 / 文案 / 文章开始创作。
2. 分镜可编辑、可排序、可锁定、可重生成。
3. 每行分镜能生成多张候选图。
4. 每行必须选择最终图，才能进入视频。
5. 每行能生成视频片段。
6. 每行必须选择最终视频片段，才能合成。
7. 合成输出 final.mp4。
8. 失败能定位到步骤和分镜行，并能重试。
```

## 四、参考项目怎么用

先读：

```text
doc/参考分析/参考项目取舍总则.md
doc/参考分析/参考方法与防漏机制.md
doc/参考分析/参考项目深度机制与吸收口径.md

doc/参考分析/MVP简化口径.md
doc/参考分析/版权与二开边界.md
```

参考项目定位：

```text
waoowaoo-main     学前端线性流程、短视频制作节奏、步骤推进感
Toonflow-app      学桌面端创作资源、Skill、画风/角色/场景一致性
Toonflow-web      学管理页信息结构、模型配置、规则管理交互；后端不可二开
Pixelle-Video     学真实生成、TTS、字幕模板、FFmpeg、ComfyUI/RunningHub/API 双轨
huobao-drama      只补充短剧业务菜单和内容管理思路，不作为核心技术蓝图
```

必须遵守：

```text
1. 不能复制参考项目代码、UI、品牌、素材和 Skill 正文。
2. 参考项目不是蓝图，只是证据库。
3. 先看 `参考项目深度机制与吸收口径.md`，确认参考项目“怎么做、为什么有效、本项目怎么筛选吸收”。
4. 每个功能按七层拆解：用户目标、页面、按钮、状态、数据、异常、本项目增强。
5. 任何页面都要检查主按钮、批量按钮、行内按钮、恢复按钮、危险按钮、辅助按钮。
6. 任何生成类功能都要检查 not_generated / input_missing / running / failed / waiting_select / selected / confirmed。
7. 任何参考实现都要写清“学什么、避开什么、本项目怎么落地”。
8. 不能因为参考项目没有某个功能，就删掉本文档和 TODO 里已经确定的能力。
9. 不能因为某个参考项目功能丰富，就把它的技术栈、页面结构、接口命名搬进来。
```

参考吸收细度门禁：

```text
1. 不能只写“学 waoowaoo / 学 Toonflow / 有分镜 / 有视频包”。
2. 每个被吸收的参考点必须写清：参考项目怎么做、页面放哪里、有哪些字段、有哪些按钮、有哪些状态、数据写回哪里、失败怎么恢复、本项目不吸收什么。
3. 如果本项目文档已有同类概念，只补“怎么分层、怎么布局、怎么验收”，不新增第二套名词。
4. 如果参考点当前不进 MVP，也要写清它进入哪个后续 TODO、预留哪些字段或入口、为什么现在不做。
5. 如果只能说出优点但说不出字段 / 布局 / 状态 / 数据流，就不能开始写代码，必须先回到 `参考项目深度机制与吸收口径.md`、`页面交互账本.md` 和源码证据索引补齐。
6. `参考项目深度机制与吸收口径.md` 的“参考强项写入文档的审计表”是开发前必查表；命中内容导入、分镜、提示词、候选、任务态、资产、视频包、模型 / 工作流、导出时，必须按表里的“必读落点”继续读对应模块文档和 TODO。
7. 任何文档或代码如果只写“参考项目有这个功能”但没有说明“怎么实现 / 怎么布局 / 字段 / 按钮 / 状态 / 数据流 / 当前实现差异”，视为未满足文档门禁，必须先补文档。
8. `参考项目深度机制与吸收口径.md` 的“深度施工证据索引与吸收细则”是更细的执行门禁。后续做内容导入、流程推进、分镜、候选、生产画布、模型 / 工作流、视频包、资产贯穿时，不能只读第九节审计表，必须继续读第十一节对应条目。
9. `参考项目深度机制与吸收口径.md` 的“筛选后吸收执行总账”是后续写代码前的总账。参考项目优点必须先判断属于“当前主线必须吸收 / TODO 阶段必须吸收 / 后续预留 / 明确不吸收”，再进入实现；不能把参考项目所有功能无筛选塞进当前主线。
10. `参考项目深度机制与吸收口径.md` 第十二节“参考强项施工卡总账”是新窗口恢复上下文的硬入口。命中内容导入、流程推进、分镜、提示词、候选、生产画布、模型 / 工作流、资产、视频包、导出时，必须读对应施工卡后再写代码。
11. 如果文档里出现“视频包 / 创作规则 / 模型工作流分层”，必须能回答：VideoPack 保存哪些引用，CreativeRule 保存哪些正文和 schema，Provider 保存什么，ProviderModel 保存什么，WorkflowPreset 保存什么，ProjectRuntimeConfig 保存什么。
12. 如果文档里出现“学 waoowaoo / Toonflow 的分镜”，必须能回答：主编辑在哪里，高级字段在哪里，候选图和视频如何保留，AI 助手改哪些字段，锁定如何生效，删除 / 合并 / 拆分如何提示下游影响。
13. 如果文档里出现“任务态 / 流式控制台 / 进度”，必须能回答：页面显示什么，DB 记录什么，ProgressEvent 只负责什么，为什么不显示 ETA。
14. 如果文档里出现“内容导入成熟”，必须能回答：长文本如何提示，切分预览有哪些行内操作，确认前后分别写哪些字段，AI 失败时如何手动兜底。
15. 如果文档里出现“候选图 / 候选视频机制”，必须能回答：候选缩略图区怎么布局，生成 revision 怎么追加，确认 / 取消 / 撤回怎么处理，清理未选如何二次确认，哪些候选能保存为素材。
16. 如果文档里出现“资产贯穿流程”，必须能回答：作品产物、候选产物、可复用 Asset、AssetReference 的边界，哪些动作会登记素材库，哪些动作只留在作品目录。
17. 如果文档里出现“设置 / 模型 / 工作流”，必须能回答：用户在哪个 Tab 配 Provider，在哪个 Tab 录 API 模型能力，在哪个 Tab 管 Workflow Preset，业务页为什么只能读可执行选项聚合接口。
18. 写代码前如果对应页面文档没有写清“当前已有实现够不够、差距是什么、验收是什么”，不能直接靠自由发挥补 UI 或接口，必须先补页面文档和当前 TODO 的验收口径。
```

## 五、开发顺序看 plan

真正开发顺序不由 doc 决定，而由：

```text
plan/阶段路线图.md
plan/TODO-01.md ~ plan/TODO-12.md
```

当前规则：

```text
1. 按 TODO 文件编号顺序做。
2. 一个阶段没完成，不跳后续阶段。
3. 每次只推进当前阶段第一条未完成 TODO。
4. 做完必须验证、打勾、写完成记录。
5. TODO 后续会基于本文档重新排，但在重排前仍按现有 plan 执行。
```

TODO 重排时按这个优先级：

```text
1. 主线闭环：内容创作 → 分镜 → 生图 → 视频 → 合成 → 导出。
2. 真实生成前置：Provider、模型、workflow、keyring、PathGuard。
3. 可靠性：任务、重试、历史、诊断、导出。
4. 一致性：画风、角色、场景、素材库、创作规则。
5. 增强：TTS、字幕、封面、BGM、模板。
6. 独立 workflow：素材成片、数字人、小说长内容、画布精修。
```

## 七、当前要保留的项目优势

这些不能因为参考项目而丢：

```text
Vue3 + Tauri 2 + Rust + SQLite
ProviderManager 统一模型调用
provider_models / workflow_presets 分离
ImageInputPlan / VideoInputPlan
keyring 保存真实密钥
StorageService + PathGuard
Task / TaskStep / ProgressEvent
ImageCandidate[] / VideoSegment[]
lockFlagsJson
Mock 和真实生成明确区分
```

这些优势优先级高于参考项目：

```text
1. 真实生成统一走 ProviderManager，不在 Vue 页面里按供应商写分支。
2. 图片和视频产物保留候选数组，不用单字段覆盖历史。
3. 任务状态以 SQLite 持久化为准，前端只展示状态。
4. 文件路径统一走 StorageService / PathGuard。
5. API Key 只进系统钥匙串，不进数据库、日志、导出包。
6. workflowType 和 inputType 必须分离。
7. Mock 能力只能证明流程，不能混同真实生成。
```

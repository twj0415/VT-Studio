# plan 目录说明

> 本目录是项目执行计划的唯一入口。
> `doc/` 是需求库和规范库，`plan/` 是开发顺序、阶段验收和 TODO 执行口径。

---

## 文件职责

| 文件 | 职责 |
|---|---|
| `阶段路线图.md` | 总控计划：当前主线、冲突优先级、执行顺序、阶段门禁。只写索引和规则，不重复每条 TODO 细节。 |
| `当前实现审计.md` | 继续开发前的真实状态表：哪些已实现、哪些只是 Mock、哪些是占位、哪些需要先纠偏。 |
| `TODO-01.md` ~ `TODO-12.md` | 分阶段施工清单，必须按编号顺序执行；每个阶段写清问题、位置、改法、验收、验证、风险和下一步进入条件。 |

TODO 文件名固定使用两位数编号，禁止新增 `TODO-1.md` 这类非补零文件，避免目录按文件名排序时 `TODO-10` 跑到 `TODO-02` 前面。

结论：原来根目录的 `阶段路线图.md` 和 `TODO-*.md` 确实有重复。现在分工调整为：

```text
阶段路线图.md = 总纲 / 顺序 / 门禁
TODO-*.md     = 按编号顺序执行的阶段施工规范 / 验收清单
```

后续不要在 `阶段路线图.md` 里复制 TODO 详情，只维护阶段索引和优先级。

---

## 执行纪律

1. 按文件编号顺序做：`TODO-01.md` 完成后才做 `TODO-02.md`，不能跳号。
2. 每次只推进当前阶段文件里的第一条未完成 TODO。
3. 一条 TODO 完成后，必须立刻自动把 `【】` 改成 `【X】`，并写完成记录，不需要用户提醒。
4. 当前阶段文件全部子项 `【X】`，且阶段完成标准、验证命令、必要 smoke test 全部通过后，才进入下一个文件。
5. 如果被中断，恢复后先读取当前阶段文件，继续第一条未完成 TODO。
6. 每次进入新的未完成 TODO 前，必须先核对 `当前实现审计.md`，确认本条依赖的页面、API、Tauri command、Repository 当前到底是 Mock、占位还是真实实现。
7. 如果当前第一条未完成 TODO 缺少“问题 / 位置 / 改法 / 验收 / 验证 / 风险 / 下一步进入条件”，先补 TODO，再写代码。
8. 如果需求不清、验收不清、文档冲突、安全高风险或需要真实外部影响操作，必须停下来问清楚。
9. 每条 TODO 打勾前必须复核当前阶段文件剩余 `【】`、本条完成范围和实际验证结果；阶段完成前必须再次复核阶段完成标准。

## 当前主线

当前主线固定为：

```text
输入文字 → 分镜 → 生图 → 图生视频 → 合成
workflowType = image_to_video
```

当前只优先跑通 `image_to_video`。TTS、字幕、封面、BGM、数字人口播、素材剪辑、纯图文成片、画布精修都不是删除，而是后续增强或独立 workflow。

## TODO 细化规范

每条 TODO 必须能回答这几个问题：

| 字段 | 必须写清什么 |
|---|---|
| 问题 | 为什么要做；不做会造成什么用户问题、工程问题或后续返工。 |
| 位置 | 改哪些页面、组件、Store、API、Tauri command、Rust Service、数据库表、迁移、测试或文档。路径不确定时写模块边界，不能写“相关文件”。 |
| 改法 | 怎么做、先后顺序、数据流、状态流、边界条件、明确不做什么。 |
| 验收 | 做到什么客观状态才算完成；必须是可判断的结果，不写“基本可用”“差不多”。 |
| 验证 | 用什么命令、页面流程、数据库检查、文件检查、日志检查或手动 smoke test 证明验收成立。 |
| 风险 | 安全、返工、状态污染、密钥泄露、路径越权、用户数据丢失、真实费用等风险。 |
| 下一步进入条件 | 本条能打 `【X】` 并进入下一条的硬门禁；必须包含完成记录、验证结果、未解决问题处置。 |

`验收` 和 `验证` 不能混用：

```text
验收 = 产品和工程结果应该长什么样。
验证 = 用哪些动作证明它真的达到了验收。
下一步进入条件 = 验收满足 + 验证通过 + TODO 已打勾 + 完成记录已写。
```

完成判定必须覆盖：

```text
需求范围
UI 布局和交互
业务逻辑
组件封装
状态管理
API / DTO 协议
样式、Tailwind 和主题 token
多语言和字典
错误 / 空态 / 加载 / 禁用 / 成功状态
安全和路径 / 密钥 / 日志规则
测试、构建、手动 smoke test
TODO 标记和完成记录
```

样式完成时必须额外检查：

```text
页面 / 组件常规布局、间距、尺寸、文本、响应式是否优先使用 Tailwind
SCSS/CSS 是否只用于主题 token、Tailwind @theme、Naive UI 覆盖、伪元素、动画、复杂渐变、滚动条、浏览器全局规则
inline style 是否仅用于必须运行时计算且 Tailwind 不适合表达的动态样式
颜色、状态色、主题色是否来自 token 或 dictionary colorToken
```

缺任意一项，都不能打 `【X】`。

现有 TODO 中的字段映射为：

```text
问题 = 为什么要做
位置 = 改哪里
改法 = 怎么做 + 做什么
验收 = 做到怎么样
验证 = 怎么证明
风险 = 不能踩什么坑
阶段完成标准 = 本阶段最终门禁
```

旧 TODO 如果只有“验证”没有“验收”，执行该条前必须把“验证”拆成“验收 + 验证”；如果没有“下一步进入条件”，必须先补齐再开始编码。

## 修改规则

1. 完成任意 TODO 后，把 `【】` 改成 `【X】`。
2. 打勾前必须确认“验证”和“阶段完成标准”已经满足。
3. 新增 TODO 必须写清“问题 / 位置 / 改法 / 验收 / 验证 / 风险 / 下一步进入条件”。
4. 如果某条 TODO 涉及真实 Provider、密钥、导出、删除文件、外部路径或真实费用，执行前必须单独确认。
5. 不允许用页面本地 mock 代替数据层 mock。
6. 不允许把 Mock 结果伪装成真实生成结果。
7. 安全、密钥、路径、日志、导出规则优先级高于所有功能进度。
8. 前端页面和组件新增 / 修改样式默认用 Tailwind；只有主题 token、Tailwind `@theme`、Naive UI 覆盖、伪元素、动画、复杂渐变、滚动条、浏览器全局规则、必须运行时计算的 `:style` 等必要场景才写 SCSS/CSS 或 inline style。
9. 完成记录必须写明已执行的验证命令和复核结果，不能只写“已完成”。
10. 没有明确验收和下一步进入条件的 TODO，不允许进入实现阶段。

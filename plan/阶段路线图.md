# 阶段路线图

> 本文件是总控计划，只负责当前主线、冲突优先级、执行顺序、完成判定和阶段门禁。
> 细化 TODO 放在同目录 `TODO-01.md` ~ `TODO-12.md`，必须按文件编号顺序执行。

## 当前统一口径

当前产品主线固定为：

```text
输入文字 → 分镜 → 生图 → 图生视频 → 合成
workflowType = image_to_video
```

两个概念必须分开：

```text
inputType    = 用户输入什么
workflowType = 最后走什么制作流程
```

当前只开放并优先跑通：

```text
workflowType = image_to_video
inputType = topic | paste | article
```

完整产品口径是：

```text
我的作品 → 开始创作 → 内容创作 → 分镜 → 生图 → 视频
→ 合成 → 导出 / 历史
```

执行时不要误解：

```text
1. 内容创作中的 AI 生成 / 整理不是独立主线；它只是内容创作里的能力，用来把想法、文章或材料整理成当前作品可用的文本。一级菜单“AI 工具”只放独立小工具，不创建作品。
2. 开始创作负责创建作品草稿并进入作品工作台 / 内容创作，不是让用户停在空项目表单。
3. StepBar 当前仍只显示 分镜 → 生图 → 视频 → 合成；设定集、输入规划、字幕、导出历史是页面内能力或后续增强。
4. 文字设定由可编辑创作规则约束；MVP 先做 builtin/user、可编辑、schema 校验，完整版本/hash/snapshot 后续增强。
5. 字幕、TTS、封面、BGM 都要保留，但不得阻塞当前 image_to_video 最小主线。
```

后续保留但不阻塞当前主线：

```text
digital_human
material_edit
image_slideshow
TTS
字幕
封面
BGM
模板动效
画布精修
小说长内容
```

## 冲突优先级

如果文档、代码、原型之间冲突，按下面顺序判断：

```text
安全 / 密钥 / 日志 / 路径 / 导出规则
> 本路线图和 plan/TODO 执行顺序
> 底层设计协议、状态机、数据结构
> 功能模块细节
> UI 原型视觉参考
```

安全规则不能因为赶功能而降级。

## 固定红线

1. 页面不得直接调用 Tauri command，必须走统一 API / entity / store。
2. 页面不得散落 mock 数组，mock 只能在统一数据层或 adapter 内。
3. 页面不得硬编码状态色、主题色、枚举显示文案、固定 options。
4. Tauri command 名统一 `snake_case`。
5. Rust / SQLite / JSON 存储层使用 `snake_case`。
6. 前端 TypeScript 使用 `camelCase`。
7. Tauri 边界必须用 serde 显式映射。
8. `appLocale` 和 `contentLanguage` 必须分开。
9. 真实 API Key 不进 SQLite、不进 yaml/json、不进导出文件，只能进系统钥匙串。
10. 日志不得打印 API Key、Bearer Token、完整密钥、完整 Provider 请求头。
11. 导出、备份、诊断包不得包含真实密钥、keyring secret、本地绝对路径和未授权外部文件引用。
12. 视觉和交互必须对照 `doc/UI设计/全页面布局设计.md`、`doc/开始之前/菜单与页面详细说明.md`、`doc/参考分析/参考项目深度机制与吸收口径.md`、`doc/参考分析/页面交互账本.md` 和 `doc/底层设计/22-UI详细设计与页面布局.md`；页面布局、导航和交互口径冲突时以 `doc/UI设计/全页面布局设计.md` 为准。
13. 页面和组件新增 / 修改样式默认使用 Tailwind；只有主题 token、Tailwind `@theme`、Naive UI 覆盖、伪元素、动画、复杂渐变、滚动条、浏览器全局规则、必须运行时计算的 `:style` 等 Tailwind 不适合承载的场景，才允许写 SCSS/CSS 或 inline style。

## 执行规则

1. 必须按文件编号顺序执行：`TODO-01.md` 完成后才能进入 `TODO-02.md`，依次类推。
2. 一个阶段文件未完成前，不得跳到后续阶段文件；除非用户明确指定临时插队。
3. 每完成一条 TODO 子项，必须立即把该条 `【】` 改成 `【X】`，并补充简短完成记录。
4. 不需要用户提醒打勾；实现、验证、打勾、进入下一条 TODO 是连续动作。
5. 一个阶段文件内所有子项都 `【X】`，且“阶段完成标准”和全局验证通过后，才允许进入下一个阶段文件。
6. 如果执行中被打断，恢复时先读取当前阶段文件，继续未完成的第一条 TODO。
7. 如果当前第一条未完成 TODO 缺少“问题 / 位置 / 改法 / 验收 / 验证 / 风险 / 下一步进入条件”，先补 TODO，再写代码。
8. 进入任何未完成 TODO 前，必须先读 `plan/当前实现审计.md`，并核对本条依赖的页面、API、Tauri command、Repository 是 Mock、占位还是真实实现。
9. 涉及参考项目吸收、页面布局、分镜、提示词、候选、资产、视频包、模型 / 工作流或任务反馈时，必须先读 `doc/参考分析/参考项目深度机制与吸收口径.md`，明确“参考项目怎么做、本项目吸收什么、不吸收什么、落到哪里”。
10. TODO 完成记录必须明确本条结果属于 Mock 闭环、Tauri/SQLite 闭环，还是真实 Provider / FFmpeg 闭环；三者不能混写。
11. 如果发现当前 TODO 描述不清、验收标准不清、文档互相冲突、风险高到不能合理决策，必须停下来说明问题，不得靠猜测继续。
12. 每条 TODO 打勾前必须复核当前阶段文件剩余 `【】`、本条完成范围、验收、验证和下一步进入条件；阶段文件全部完成后必须再次复核阶段完成标准。

## 完成判定

一条 TODO 只有同时满足以下条件，才算完成并允许打 `【X】`：

1. **范围完成**：该条“改法”里要求的文件、接口、页面、DTO、Store、Service、文档、测试都已落地；没有只做一半。
2. **UI 完成**：涉及界面时，布局、样式、响应式、密度、主题 token、状态色、空态、加载态、错误态、禁用态、成功态都合理；文字不溢出、不重叠。
3. **逻辑完成**：数据流、路由、校验、状态流转、前置条件、失败处理、重试或阻断逻辑正确；不能只做到按钮能点。
4. **组件完成**：该抽组件时已抽，不该抽时不过度封装；props、emits、类型、命名清楚；复用现有 shared/base 组件和项目风格。
5. **多语言与字典完成**：用户可见文案优先走 i18n；枚举显示和 options 优先走 dictionary / `useDictOptions`；不在页面复制业务枚举。
6. **样式完成**：页面不硬编码状态色和主题色；颜色来自 theme token 或字典 colorToken；常规布局、间距、尺寸、文本、响应式优先 Tailwind；SCSS/CSS 只保留在主题 token、Tailwind `@theme`、Naive UI 覆盖、伪元素、动画、复杂渐变、滚动条、浏览器全局规则和必须运行时计算的 `:style`；固定尺寸不导致内容错位。
7. **类型与协议完成**：前端 camelCase，Rust/SQLite/JSON snake_case，Tauri 边界 serde 显式映射；没有用 `any` 掩盖问题。
8. **安全完成**：不泄露 API Key、Bearer Token、绝对路径、Provider 请求头；涉及文件路径、导出、诊断、密钥时遵守最高安全规则。
9. **验证完成**：该条相关的 typecheck、build、cargo check/test、单测、页面 smoke test 或手动验收已执行；失败项必须修完或明确记录阻塞。
10. **计划同步**：完成后立即更新对应 TODO 标记和完成记录；如果发现新问题，追加到当前或后续合适 TODO。
11. **进入条件完成**：本条写明的“下一步进入条件”全部满足，包含完成记录、验证结果、未解决问题处置。
12. **复核完成**：打勾前已重新检查当前阶段文件的未完成 TODO、本条验收范围、验证输出和下一步进入条件；阶段完成前已重新检查阶段完成标准。

不能满足任一条，就不能打 `【X】`。

## 停下来条件

遇到以下情况必须暂停当前实现，向用户说明，不得继续猜：

1. 需求、文档或 TODO 的验收口径互相冲突，且无法按优先级确定。
2. 需要真实 Provider、真实费用、密钥、删除文件、导出外部目录、执行外部程序或高风险系统权限。
3. UI/交互目标不明确，继续实现会造成明显返工。
4. 数据结构缺字段或字段含义不清，继续会污染数据库、DTO 或 API 契约。
5. 安全规则和功能诉求冲突。
6. 验证命令持续失败，且错误不是当前可独立修复的问题。

## 当前最小闭环验收

MVP 必须能跑通：

```text
打开桌面应用
在“我的作品”点击“开始创作”，自动创建作品草稿
输入一段文字
生成 / 编辑分镜
进入生图表格
每行生成候选图
每行选择最终图
进入视频阶段
生成 / 确认视频片段
进入合成阶段
生成最终视频
打开输出目录
```

Mock 闭环和真实闭环必须明确区分，不能把 Mock 结果当真实成片。

## TODO 维护规则

1. 完成任意条目后，把 `【】` 改成 `【X】`。
2. 新增 TODO 必须写清：问题、位置、改法、验收、验证、风险、下一步进入条件。
3. 旧 TODO 如果缺少“验收”或“下一步进入条件”，执行前必须先补齐，再进入实现。
4. 如果发现新问题，追加到对应 TODO 文件，不要塞回总路线图。
5. 如果计划顺序变化，只改本文件“执行顺序”表。
6. 不要把已完成历史长期堆在本文件里，历史通过文件状态和后续 git 记录追踪。
7. 完成记录必须写明已执行的验证命令和复核结果，不能只写“已完成”。
